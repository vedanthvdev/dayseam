## 1. Why the v0.2 release-workflow failure did not recur (second confirmation)

v0.3 shipped as evidence that the "only the capstone PR carries `semver:minor`" discipline works over a six-PR arc. v0.4 extends the sample to nine PRs: DAY-92 (plan draft) through DAY-100 (orchestrator wiring + BDD + `SerdeDefaultAudit` sweep) each carried `semver:none`, the release workflow correctly held fire on every merge, and `semver:minor` lives only on this capstone PR. Fifteen consecutive `semver:none` merges across v0.3 + v0.4 with zero drift confirm the discipline holds without mechanism backing — no release-workflow guard has been added at the label-enforcement layer, and v0.4 does not propose one.

The one adjacent pattern worth noting: DAY-92's plan-draft PR ([#96](https://github.com/vedanthvdev/dayseam/pull/96)) carried `semver:none` despite being structurally a "docs-only" change that would have been `semver:patch` under the post-v0.1 rule. That is the correct call — the plan lands on the first PR of an unreleased phase arc and is effectively rolled up into the capstone's release notes — but it illustrates that the policy is "`semver:none` on every non-capstone v0.N PR" rather than "`semver:none` on every code PR and `semver:patch` on every docs PR". Worth reaffirming in the v0.5 plan doc when it's drafted.

### 3.1 CORR-v0.4-01 (HIGH) — zero-event GitHub sources: detail + fix

The silent-failure chain surfaced by the correctness lens is the most dangerous class v0.3 went hunting for, and it shipped unnoticed because the orchestrator integration test in DAY-100 hand-seeded *both* identity rows directly, bypassing the actual IPC path that was dropping `login` at the Tauri boundary.

**Four consecutive links, every one shipped in v0.4:**

1. `connector_github::list_identities` (`crates/connectors/connector-github/src/auth.rs`) emitted exactly one row (`GitHubUserId`). The Vec-shaped return type anticipated a future `GitHubLogin` row but the implementation never materialised it.
2. `github_sources_add_impl` (`apps/desktop/src-tauri/src/ipc/github.rs`) constructed `GithubUserInfo { login: String::new(), … }` because its signature accepted only `user_id: i64`. The dialog's validated `login` was thrown away at the IPC boundary.
3. `walk::self_identity` (`crates/connectors/connector-github/src/walk.rs`) required *both* `GitHubUserId` (for the events-stream self-filter) *and* `GitHubLogin` (to compose `/users/{login}/events`); missing the login returned `None` with a `Warn` log.
4. `walk_day` converted that `None` directly into `Ok(WalkOutcome::default())` — `Ok`, not `Err`. The orchestrator treated the sync as successful-with-zero-events. No `SourceHealth::Error`, no red chip, no reconnect affordance.

**Why the test suite missed it.** The orchestrator integration test at `crates/dayseam-orchestrator/tests/github_integration.rs` manually hand-crafted both identity rows and wrote them directly via `SourceIdentityRepo::insert`, validating the walker for the "both identities present" case but never exercising the production path that persists them. The unit test `list_identities_returns_exactly_one_row_on_happy_path` even pinned the one-row behaviour as *correct*, so the bug was encoded in both directions. `validate_auth` + `sources_healthcheck` reported "Connected as @handle" because both probes only hit `GET /user` — neither endpoint touches `/users/:login/events` where the downstream failure materialises.

**User-facing consequence.** Every user who exercised the new DAY-99 "Add GitHub Source" dialog — i.e., the headline v0.4 UX — would have received a source that looked healthy, carried a populated keychain row, and contributed zero events to every daily report until they manually added a `GitHubLogin` row through the power-user `IdentityManagerDialog`. v0.3's CHANGELOG said of the walker *"missing either early-bails with an empty outcome and a `Warn` log, never a silent zero-event day"* — the "never" was load-bearing only if the IPC layer seeded both rows, which it did not.

**Fix landed in DAY-101.** Five-piece change:

1. `github_sources_add_impl` (and its Tauri command wrapper) now accept `login: String` alongside `user_id: i64`, with an `invalid_config_public(GITHUB_UPSTREAM_SHAPE_CHANGED)` guard on empty-or-whitespace values that runs before any keychain / DB write.
2. `list_identities` now returns **two** rows on happy-path (`GitHubUserId` + `GitHubLogin`) and adds a symmetric empty-login shape-change guard — the `NonPositiveId` and `EmptyLogin` rejections both route through `GithubUpstreamError::ShapeChanged` so the failure mode surfaces through the same error card the auth probes use.
3. `AddGithubSourceDialog.tsx` threads `validation.result.login` into its `github_sources_add` invoke call alongside `userId`.
4. `github_sources_reconnect_impl` now self-heals the `GitHubLogin` row on every successful reconnect via `identity_repo.ensure(…)` — idempotent, and the chief upgrade-path recovery mechanism for any v0.4-era source that was created through the pre-fix dialog.
5. Tests now pin the new two-row behaviour at three layers: the `auth.rs` unit test was renamed and rewritten; a fresh `add_happy_path_writes_source_and_keychain_and_identity` assertion covers both rows landing in the DB; a new `add_rejects_empty_login_before_any_write` guards the IPC boundary; the RTL test's capture-shape expectation adds `login: "vedanth"`; the Playwright mock captures `login` in `CapturedGithubAddCall`. The orchestrator integration test's hand-seeded login row continues to work unchanged (it predicted the fixed shape).

**Regression surface remaining.** The dialog and IPC now both reject empty login *structurally* before any keychain or DB write, and the reconnect heal path covers the upgrade case — so there is no reachable path in this workspace that would recreate the silent-zero-event source. The one residual risk class is a future connector (v0.5+) that adopts the same "two identity rows per source" shape and forgets to thread the second field through its own IPC; that pattern is now documented in `auth.rs:list_identities` and the DAY-101 CHANGELOG entry so the next reviewer sees it immediately.

## 4. Graphify re-evaluation

The v0.1 Phase 3 review filed [`docs/decisions/2026-04-20-graphify-deferred.md`](../decisions/2026-04-20-graphify-deferred.md) with a three-axis rubric and **five re-evaluation triggers**. The v0.2 and v0.3 capstones explicitly deferred re-scoring ([#61](https://github.com/vedanthvdev/dayseam/issues/61) as the tracking issue); the v0.4 plan reserved this capstone as the formal re-evaluation point. The scoring below is against `master` at this PR's tip.

### 4.1 Signal on evaluation day

Repo shape (comparable against the v0.1 baseline):

| Signal | v0.1 baseline (2026-04-20) | v0.4.0 baseline (2026-04-20, this PR) | Δ |
|---|---:|---:|---:|
| Workspace crates | 12 | 17 | +42% |
| Tracked Rust + TS/TSX source files | 294 | 391 | +33% |
| Rough total LOC (Rust + TS/TSX) | ~37 800 | ~73 100 | +93% |
| Canonical architecture docs (design + plan + review + ARCHITECTURE.md) | ~6 000 lines | ~10 600 lines | +77% |
| Active committers | 2 (1 human + bot) | 2 (1 human + bot) | 0 |
| Long-lived branches | 1 (`master`) | 1 (`master`) | 0 |
| E2E coverage of the click path | Green (1 happy-path scenario) | Green (8 scenarios: happy-path + 3 Atlassian + 3 GitHub + 1 save-IPC-contract) | +7 scenarios |

### 4.2 Trigger check against the decision doc's five re-open conditions

1. **"Workspace size roughly doubles — > ~25 crates or > ~600 source files."** Current: 17 crates, 391 files. Not met, but the trajectory (+42% crates / +33% files over four phase-arcs) projects to hitting the file-count threshold around v0.6 if each v0.N adds ~one connector's worth of scope. Close enough to re-score, not close enough to flip.
2. **"Contributor count crosses ~3 concurrent."** Unchanged — still 1 human + a bot. Not met.
3. **"Architecture docs start drifting from code."** v0.4's review explicitly verified the five seams it touches (`EntityKind::Workspace`, `ArtifactPayload::MergeRequest`, `ReportSection::Unlinked` rename, `SerdeDefaultAudit` coverage, PAT-auth symmetry) against the design doc — each one cross-references a specific §. The `CHANGELOG.md` + plan-doc pattern continues to act as the running reconciliation. No drift caught. Not met.
4. **"LLM-agent orchestration becomes a first-class development surface."** Dayseam ships no in-app AI-agent tooling that would consume a repo graph at inference time. Contributors continue to use Cursor / Claude / Codex as personal productivity aids, but nothing in the app itself reads or writes a knowledge graph. Not met.
5. **"A comparable zero-LLM-cost alternative ships."** No new tool of that shape has landed on Dayseam's radar since v0.1. `rust-analyzer`'s symbol index, `cargo doc --workspace`, and `rg` continue to answer the "where does X live" questions the graph would answer. Not met.

### 4.3 Re-scoring the three axes

**Axis (a) — does graphify surface things the existing toolchain does not?** The LOC growth from 37 800 → 73 100 is the signal that would flip this axis if anywhere would, because the cross-reference density thins out as the codebase grows. In practice it has not: the v0.3 and v0.4 reviews each cited specific design-doc sections (§6.2, §10.3, §13, §17 plus the new GitHub §18 written during DAY-93) when reasoning about cross-source consistency, and the reviewer subagents above all navigated by `rg` + `Grep` + known-filename lookups without any graph. The hand-curated cross-references continue to outperform an inferred graph at this scale. **Verdict: marginal, unchanged from v0.1.**

**Axis (b) — is the staleness signal trustworthy?** Docs have grown +77% (to 10 600 lines), including 4 new plan docs, 3 new review docs, and several decision docs under `docs/decisions/`. The AST-vs-prose split the v0.1 decision flagged is now more asymmetric, not less: the prose side is where almost all of v0.3/v0.4's architecture work lives. A graphify index freshened only on code commits would go stale on every review merge, and nothing in the toolchain refreshes the doc side automatically. **Verdict: negative, unchanged.**

**Axis (c) — acceptable trust surface when artefacts are checked in?** The secrets-adjacent file list from the v0.1 decision now has five additional entries: the Atlassian wiremock fixtures (v0.2), the GitHub wiremock fixtures (v0.4), the expanded `apps/desktop/src-tauri/capabilities/*.json` allowlist (v0.3 + v0.4 added 9 commands), the `dayseam-macros` trybuild fixtures (which an LLM paraphrasing would actively damage — the trybuild golden `.stderr` snapshots are load-bearing), and the new GitHub `errors::map_status` error-code catalogue (same risk as the Atlassian one from v0.2). Per-project exclusion tuning grows with each phase. **Verdict: negative, marginally worse than v0.1.**

### 4.4 Decision: defer again to v0.5's capstone

0 of 3 axes come up clearly positive. The rubric ("adopt if 2 of 3 are positive") resolves to **defer**. The re-evaluation trigger under which this review would have been required to re-open the question — the first of the five conditions — has not fired; v0.5 will re-score when it runs the same evaluation. The tracking issue [#61](https://github.com/vedanthvdev/dayseam/issues/61) is **not closed**; it is updated with a pointer to this §4 subsection as the latest negative re-evaluation, preserving the audit trail.

No new files land for graphify adoption this phase either. The `docs/decisions/2026-04-20-graphify-deferred.md` decision record stands; this review entry is the second piece of written evidence that its triggers have not fired.

## 6. Process note carried from v0.3 — verified clean

v0.3's review §6 noted that label discipline was a six-PR pattern and deferred the "mechanism backing" question. v0.4's execution extended that to nine more PRs with zero drift.
