## 1. Why the v0.2 release-workflow failure did not recur

The [v0.2 review §7 process note](./v0.2-review.md#7-process-note-for-v03) called out the structural bug that turned the v0.2.0 tag into a pre-review publish: the `semver:minor` label had been applied to the penultimate feature PR (DAY-83), and the release workflow correctly honoured it before the capstone review had a chance to run. [The v0.3 plan](../plan/2026-04-20-v0.3-report-polish-hardening.md) picked up that lesson: every feature / polish PR in v0.3 (DAY-85..DAY-90) carried `semver:none`, and the `semver:minor` label lives *only* on this capstone PR. That means the release workflow runs against the same commit that ships this review doc, the updated `CHANGELOG.md`, and the `VERSION` bump — atomically.

The label-discipline fix is the only process change v0.3 needed to make on top of v0.2's audit. No additional release-workflow guard was added — the six-PR run through the new discipline is the evidence that it works. If v0.3 execution had revealed a silent drift (e.g. someone relabelling a polish PR to `semver:patch` under the assumption it was a "hotfix"), we would have amended the plan doc on the affected PR; it did not. Noted as a clean exit in §6 below.

## 4. Cross-cutting observations (for v0.4's plan)

Two patterns showed up across multiple lenses and warrant attention before v0.4's first feature PR lands.

### 4.1 The per-kind section enum is the seam the MR artifact wants to land on

`ReportSection` (in [`crates/dayseam-report/src/sections.rs`](../../crates/dayseam-report/src/sections.rs)) now has three real variants (`Commits`, `JiraIssues`, `ConfluencePages`) plus one catch-all (`Other`). The plan text ["events grouped under `Commits` / `Merge requests` / `Jira issues` / `Confluence pages`"](../plan/2026-04-20-v0.3-report-polish-hardening.md) imagined a fourth *first-class* variant (`MergeRequests`), but DAY-86 scoped it down because there is no `ArtifactPayload::MergeRequest` variant yet — MRs currently attach to a `CommitSet` via the rollup_mr step, which renders under `Commits` with a `(rolled into !42)` suffix. The enum is the right seam for promoting MRs to their own section the moment the payload exists, and the exhaustive-match invariant at `ReportSection::from_payload` means the first PR to add the payload also has to pick a section for it. This is a load-bearing piece of compile-time scaffolding, not a layering accident — noted so the v0.4 planner doesn't "clean it up" by collapsing the enum into a string.

### 4.2 `SerdeDefaultAudit` wants one more coverage sweep before the next persisted type lands

The derive currently lives on `SourceConfig` (enum, wire-stable to the DB's `sources.config_json` column) and `EntityRef` (struct, wire-stable to `activity_events.entities`). Every other type in `dayseam-core/src/types/` that ever deserialises from the DB is one of: (a) no `#[serde(default)]` fields today (e.g. `SinkConfig`, `ArtifactPayload`), (b) `#[derive(sqlx::FromRow)]` so serde-default isn't the deserialisation path (e.g. the DB row structs), or (c) a wire-format HTTP DTO in a connector crate (GitLab events, Jira search responses, Confluence CQL responses) that serde-default is the *right* answer for because the upstream API rev-level is independent of the client binary.

The v0.4 design question is whether case (a) should preemptively carry the derive so that *adding* a `#[serde(default)]` field to `SinkConfig` fails compilation rather than passing silently. The plan leaves this decision to the first v0.4 PR that touches `SinkConfig`; the option is cheap (the derive emits zero code when no `#[serde(default)]` fields are present, so applying it is a future-proofing nudge not a code-gen cost) and consistent with the lesson DAY-90's derive was designed to encode.
