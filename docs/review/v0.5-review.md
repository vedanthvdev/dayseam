**Process finding (PROC-v0.5-01, Medium).** The v0.3 and v0.4 reviews both celebrated the "capstone-only release" discipline as a demonstrated pattern. v0.5 ships as evidence that the discipline is **not self-enforcing**: DAY-117 happened because DAY-115 (this review) wasn't written yet, and the `v0.6.1` release couldn't wait for it. Once DAY-117 cut the release without the review, the plan's phase-ending discipline was broken, and the next two user-reported bugs (v0.6.2 UI issues, v0.6.3 label-save bug) shipped as unplanned patches rather than surfacing through a capstone review's cross-cutting pass. **Disposition:** carry as an explicit v0.7 process finding — the v0.7 plan header should require the capstone review's written artefact as a merge-gate for the phase-ending release, with no ad-hoc release-cut path. See §7 for the filed tracking issue.

**Process finding (PROC-v0.5-02, Medium).** **DAY-114 was skipped.** The plan spec for DAY-114 was specific: for every bug in `DOGFOOD-v0.4-01..07`, `F-1..F-11`, `CORR-v0.4-01`, ensure a regression test fails on fix-revert; plus six class-level meta-tests / lints (IPC identity-seed completeness; reconcile data-loss guard proptest; filesystem-predicate symlink-safety grep gate; RTL `parentElement`-walk ESLint rule; `DayseamError`-to-DOM-leak check; `SourceKind` registry round-trip). The absence of this audit is load-bearing: v0.6.2's three issues and v0.6.3's two issues include patterns that a class-level lint pass would likely have caught (the `""`-vs-`null` PAT-arg mismatch in GitLab's edit dialog is structurally the same shape as the v0.4 CORR-v0.4-01 `login` drop at the IPC boundary — both are "dialog-side forwards the wrong sentinel value across the IPC boundary"). **Disposition:** carry forward as `DAY-114-followup` into v0.7; see §7. Pragmatically, the v0.6.2 / v0.6.3 patches have since landed regression tests for the specific bugs they fixed, so the cardinal concern (those bugs going unchecked) is now mitigated — but the class-level meta-tests that would have caught the *next* instance are still unwritten.

The rest of this review proceeds with the same multi-lens discipline the v0.3 / v0.4 capstones established, but honest about the fact that the phase's last two months' shipment is a mixture of plan execution (DAY-108..DAY-113) and plan-deviation response (DAY-116..DAY-121).

### 3.1 Correctness lens findings

**C-1 (Medium) — `supervised_spawn` breaks `JoinHandle::abort()` semantics.** [`crates/dayseam-core/src/runtime/supervised_spawn.rs`](../../crates/dayseam-core/src/runtime/supervised_spawn.rs) spawns an outer task whose body is `tokio::spawn(future); inner.await`. `JoinHandle::abort()` on the *outer* handle only cancels the outer task's `.await`; the *inner* task — which owns the user-supplied future — survives as a detached leak. The companion docstring in [`apps/desktop/src-tauri/src/ipc/broadcast_forwarder.rs`](../../apps/desktop/src-tauri/src/ipc/broadcast_forwarder.rs) still implies abort propagates cleanly. **Why not inlined:** the fix requires either restructuring `supervised_spawn` to use `AbortHandle::abort()` on the inner task (and propagating it through the outer type) or accepting the leak and narrowing the docs to say so. Neither shape has a one-line variant, and no v0.5 site actually depends on abort propagation through `supervised_spawn` (the reaper cancels the orchestrator run via `CancellationToken`, not via handle abort). Kept as a Medium defer rather than a Critical inline.

**C-4 (Medium) — `RenameSourceDialog` breaks GitLab orphan rows.** DAY-121's new dialog sends `{ label, config: null }` with `pat: null`. For a GitLab row whose `secret_ref` is `None` (a legacy orphan — DAY-120's startup audit logs these), `validate_pat_arg` rejects `pat: None` with `ipc.gitlab.pat.missing`, surfacing as a dialog error and blocking the rename. **Why not inlined:** the orphan population is small (audit runs at startup; v0.6.3 ships with a reconnect UX for them) and the Medium severity reflects that — the regression is "rename fails with a clear error for a rare state" rather than "silent data loss". The fix is one-line (detect `secret_ref.is_none()` and short-circuit to a config-only update) and belongs with the orphan reconnect polish.

**C-2 (Low) — `MAX_PAGES` silent truncation.** [`crates/connectors/connector-github/src/walk.rs`](../../crates/connectors/connector-github/src/walk.rs) breaks out of the pagination loop on `pages >= MAX_PAGES` and returns the partial result. The original plan called for `DayseamError::Internal { code: "connector.github.pagination.cycle_guard_tripped" }` so the UI can surface "we couldn't finish the scan, please retry" instead of silently reporting a partial event list. Low severity because `MAX_PAGES` is set high enough that no real user hits it in a 24h range; still a drift from plan that belongs on the followup list.

**C-5 (Low) — `useUpdater` resource leak.** [`apps/desktop/src/features/updater/useUpdater.ts`](../../apps/desktop/src/features/updater/useUpdater.ts) captures the Tauri `Resource` handle from `check()` into a ref but only closes it on the happy path. On every successive `check()` that resolves to `null`, the ref is overwritten without first calling `.close()`. Low severity: the handle is small and a relaunch clears it; still leaks one per poll interval.

### 3.2 Test-quality lens findings

**T-1 (Medium) — DAY-108 updater tests mock past signature verification.** The vitest suite in [`useUpdater.test.tsx`](../../apps/desktop/src/features/updater/__tests__/useUpdater.test.tsx) and the rust-side updater tests both mock the `check` / `download_and_install` Tauri commands directly, which means Ed25519 signature verification — the single most security-critical piece of DAY-108 — is skipped entirely. The CHANGELOG entry claims "pipeline verified"; **the pipeline tested is the UI state machine**, not the crypto. The proper regression shape is a `tauri-plugin-updater` integration test that points at a fixture manifest signed with a known-bad key and asserts `download_and_install` errors with a signature-mismatch variant. Defer: requires either vendoring fixtures or extending `tauri-plugin-updater`'s test surface.

**T-3 (Medium) — `entitlements.plist` validity gate is release-only.** DAY-120's `plutil -lint` + XML-comment `grep` runs inside [`scripts/release/build-dmg.sh`](../../scripts/release/build-dmg.sh), which only executes on the release workflow. A PR that re-introduces an XML comment into `entitlements.plist` passes `ci.yml` clean; the failure surfaces at release time as a broken DMG. Move the two checks into their own standalone script and wire them into `ci.yml`'s default pre-test matrix.

**T-5 (Low) — Label-input discovery heuristic.** The label-only subtest in [`AddGitlabSourceDialog.test.tsx`](../../apps/desktop/src/features/sources/__tests__/AddGitlabSourceDialog.test.tsx) locates the label field with `getAllByRole("textbox").find(el => !el.readOnly)`. If a future change adds a `readOnly` prop to any input in that dialog (e.g. a frozen "base URL" field on edit), the subtest finds the wrong element. Prefer a stable `data-testid` on the label input.

### 3.3 Silent-failure lens findings

**SF-1 (Critical) — ✅ inlined in v0.6.4.** Root cause, fix, and regression test fully documented in [`CHANGELOG.md`](../../CHANGELOG.md#064). Summary: [`apps/desktop/src-tauri/src/main.rs`](../../apps/desktop/src-tauri/src/main.rs) never called into `tracing_subscriber`. Every `tracing::error!`, `warn!`, `info!`, `debug!` in the workspace — including `supervised_spawn`'s panic-capture from DAY-113 — wrote to the null global dispatcher. F-10 was effectively reinstated the moment DAY-113's helper shipped. Fix: new [`tracing_init::init`](../../apps/desktop/src-tauri/src/tracing_init.rs) installs an `EnvFilter` stderr subscriber as the first statement in `main`; two unit tests pin both the "global dispatcher is set after init" and "double-init does not panic" guarantees.

**SF-2 (High) — ✅ inlined in v0.6.4.** Root cause, fix, and regression tests fully documented in [`CHANGELOG.md`](../../CHANGELOG.md#064). Summary: the `Err(JoinError)` branch of the supervised completion task in [`apps/desktop/src-tauri/src/ipc/commands.rs`](../../apps/desktop/src-tauri/src/ipc/commands.rs) only logged and returned. The orchestrator panicking or being aborted produced zero user-visible output, leaving the frontend wedged on "generating…" with no way to recover short of relaunch. Fix: extract pure `completion_payload` helper; `Err` branch synthesises a `SyncRunStatus::Failed` payload; supervisor always forwards to `app_handle.emit`. Four regression tests (`success`, `user_cancel`, `panic`, `abort`) all pass.

**SF-3 (Medium) — `no-bare-spawn.sh` grep is string-literal strict.** The CI gate in [`scripts/ci/no-bare-spawn.sh`](../../scripts/ci/no-bare-spawn.sh) matches on the literal text `tokio::spawn(` and `tauri::async_runtime::spawn(`. `use tokio::task::spawn; spawn(future)` bypasses the gate entirely. Fix shape: either ban the `use tokio::task::spawn;` import outright (via the same script), or upgrade the gate to a syntactic check using `cargo-expand` or `syn`. Low-cost first pass: append `grep -n 'use .*tokio::\\(task::\\)\\?spawn;' …` to the script's scan set.

### 3.4 Release-engineering lens findings

**R-2 (Medium) — release workflow has no CI-needs gate.** [`.github/workflows/release.yml`](../../.github/workflows/release.yml) triggers on tag push and does not declare `needs: [ci, e2e]`. An admin merging a PR with a failing CI status (or a tag pushed manually from a broken branch) kicks off the release immediately. Fix: add a `wait-for-ci` job that polls the merge commit's check-run set and blocks on any non-success.

**R-3 (Medium) — no fast-forward guard on `git push origin HEAD:master`.** The release workflow pushes the bump commit to `master` without `--force-with-lease`. A concurrent merge between the workflow's clone and push windows overwrites the intervening commit. Fix: `git push --force-with-lease=master:<sha-before>`, where `<sha-before>` is captured at the start of the workflow run.

**R-4 (Low) — `check-semver-label` is advisory.** The `exactly-one-semver-label` workflow fails loudly on a PR that lacks a `semver:*` label, but branch protection on `master` does not list it as required; a maintainer with merge rights can bypass. Fix: add it to the required-status-checks list in repo settings.

**R-5 (Low) — `no-bare-spawn.sh` matrix coverage.** The CI job in [`ci.yml`](../../.github/workflows/ci.yml) runs only on `ubuntu-latest`. A future change to the script that assumes GNU-grep-only flags would fail on a macOS runner; the release-matrix `macos-latest` lane doesn't catch it because the gate is wired into `ci.yml`, not `e2e.yml`. Add a `macos-latest` leg to the script's CI job.

### 3.5 Performance lens findings

The performance subagent's pass found **no actionable regressions** against v0.4-tip. `supervised_spawn`'s overhead is one extra `tokio::spawn` + one `.await` per supervised task; this is invisible next to the orchestrator's own per-source HTTP latency. `SerdeDefaultAudit`'s compile-time cost was measured at +3.4s for the dayseam-core build — well within the plan's "acceptable" band for a compile-time audit. `updater.check()` startup latency is dominated by network round-trip to GitHub Releases (~200–600ms); the measurement was done cold-cache and is expected. `HttpClient` pooling via DI (DAY-111) is a net win, not a cost. No deferred findings.

### 4.2 Trigger check against the decision doc's five re-open conditions

1. **Workspace size roughly doubles (>~25 crates or >~600 source files).** Crates: 17 (68% of the threshold). Source files: 383 (64% of the threshold). **Not fired** — the trajectory is clear but the absolute thresholds aren't crossed yet. LOC has doubled (+101%) but LOC isn't the rubric criterion.
2. **Contributor count crosses ~3 concurrent.** Human contributors: still 2 (solo developer + github-actions bot). The parenthetical applies to AI agents — "if parallel agent sessions become a regular pattern rather than an occasional one, re-evaluate." v0.5 saw a material uptick in parallel agent orchestration (this very review dispatches 5 parallel lens subagents). **Approaching threshold but not fired** — the agent orchestration pattern is "regular at capstone moments" rather than "regular across all development", and the graph would still be stale between capstones without a refresh CI.
3. **Architecture docs drifting from code.** v0.5's execution revealed no architectural drift, but did reveal **process drift** (DAY-114 skipped, DAY-117 bypassed the capstone review). That's a plan-vs-execution drift, not an architecture-vs-code drift; the graphify rubric is about the latter. **Not fired** in the rubric's sense, but the PROC-v0.5-01 finding above is the orthogonal warning.
4. **LLM-agent orchestration becomes first-class dev surface.** No in-app agent tooling. **Not fired.**
5. **Zero-LLM-cost alternative ships.** No change. **Not fired.**

### 5.1 End-to-end updater smoke-test (deferred — requires physical macOS verification)

The plan's DAY-115 Step 12 mandates an end-to-end `v0.6.0 → v0.6.1` updater smoke-test on a laptop running the DAY-108 post-merge release. Since v0.6.0 released, three more releases have shipped (v0.6.1, v0.6.2, v0.6.3), so the realistic smoke-test is now `vX → v0.6.3` where `vX` is whatever version the user has installed locally. The smoke-test cannot run inside the agent environment (no macOS laptop, no ability to observe Gatekeeper prompts or Keychain approvals). **The checklist below is for manual execution by the maintainer, post-merge.**

**Smoke-test checklist (manual):**

1. On a laptop running any earlier `v0.6.x` (confirm via _Dayseam → About → Version_ or `/usr/libexec/PlistBuddy -c 'Print :CFBundleShortVersionString' /Applications/Dayseam.app/Contents/Info.plist`), wait up to the poll interval or trigger the updater check manually via _Dayseam → Check for Updates_ (the menu entry DAY-119 added).
2. Confirm the update banner renders with the correct target version (`v0.6.3` as of this review).
3. Click **Update now**. Confirm the download progress UI updates and doesn't hang.
4. Confirm Ed25519 signature verification passes. (Observable symptom: the update actually installs. A failure would surface as an "update failed" banner with no install. Do not physically tamper with the DMG — verification is pinned at the tauri-plugin-updater layer and a tampered-DMG test belongs in CI, not manual smoke.)
5. Confirm the app restarts into the target version: `/usr/libexec/PlistBuddy -c 'Print :CFBundleShortVersionString' /Applications/Dayseam.app/Contents/Info.plist` returns the expected string.
6. Confirm `state.db` survives the update: open the app, a persisted source (any pre-existing connector chip) and a saved report should both be visible.
7. **macOS ad-hoc-signature caveat lens**: record whether Gatekeeper re-prompts on first launch after the auto-update. Known issue — tracked at [#59](https://github.com/vedanthvdev/dayseam/issues/59) / [#108](https://github.com/vedanthvdev/dayseam/issues/108) for the Developer ID structural fix. The expected post-v0.5 behaviour is one Keychain prompt per connector family (the three-prompt cascade deferred per [decision doc](../decisions/2026-04-20-keychain-prompts-defer-developer-id.md)).

Steps 1–6 passing = updater healthy. Step 7's prompt observation is an accepted cost per the deferral decision.

## 6. Process note carried from v0.4 — partially validated, partially broken

v0.4's §6 carried a process note that **15 consecutive `semver:none` merges across v0.3 + v0.4 with zero drift confirmed the "capstone-only release" discipline holds without mechanism backing**. v0.5's record:

- DAY-108 (correctly `semver:minor` for the updater feature) — **on-policy**.
- DAY-109..DAY-113 (5 PRs, all correctly `semver:none`) — **on-policy**.
- DAY-116..DAY-117 (2 PRs, DAY-116 `semver:none`, DAY-117 `semver:patch` cutting `v0.6.1`) — **on-policy** per the four-bump release cadence, but the `v0.6.1` cut happened via an **ad-hoc release-engineering PR** (DAY-117) rather than the planned capstone PR (DAY-115). That's a new failure mode the discipline hasn't previously exhibited: the label policy stayed correct, but the phase-ending *review* was bypassed.
- DAY-119 (correctly `semver:patch` for `v0.6.2`) + DAY-120 (`semver:patch` but functionally a release-unblock rather than a feature) + DAY-121 (`semver:patch` for `v0.6.3`) — **on-policy** individually, but the pattern of three consecutive patch releases in four days for user-reported bugs is itself a v0.5 hardening signal: the plan's DAY-114 (class-level meta-tests) and DAY-115 (multi-lens review) were the two gates designed to catch these *before* ship, and both were skipped / deferred.

**Carried forward finding:** the label discipline is at 20 merges without a label-level drift, but the *review discipline* (reviews gate releases) is newly demonstrated as fallible. The v0.7 plan header should require the capstone review's written artefact as a pre-merge gate for the phase-ending release.
