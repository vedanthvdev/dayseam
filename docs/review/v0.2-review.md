## 1. Why this is a hotfix, not a new cut

The v0.2.0 tag was created **before** this review ran. The previous task ([DAY-83 orchestrator registry wiring](https://github.com/vedanthvdev/dayseam/pull/82)) merged with a `semver:minor` label, which the release workflow interpreted — correctly per its own policy — as "bump to `0.2.0` and publish". By the time the capstone review started, `v0.2.0` was already on the [Releases page](https://github.com/vedanthvdev/dayseam/releases/tag/v0.2.0).

That shifts this PR's shape. Instead of *cutting* `v0.2.0` and including the review findings in the same release, the review surfaced seven P0/HIGH findings that break the v0.2 user promise ("Atlassian works end-to-end from the desktop UI") and a long tail of Mediums/Lows that do not. The inlined fixes ship as **v0.2.1** (`semver:patch`). The Mediums/Lows are filed as themed GitHub umbrella issues and deferred to v0.3.

This is the mirror image of the [Phase 3 addendum](./phase-3-review-addendum.md): that document ran the deeper pass *after* v0.1.0 shipped because the formal Phase 3 review missed a class of bug; this document runs the deeper pass *after* v0.2.0 shipped because the release happened before the review. The structural fix — block the capstone PR on the review battery *before* the `semver:minor` label lands — is captured as a process note in §7.

## 4. Cross-cutting observations (for v0.3's plan)

Three patterns showed up across multiple lenses and warrant attention before a fifth connector (GitHub) lands.

### 4.1 The `connector-confluence` normaliser needs a second editorial pass

Four of the nine silent-failure findings (CORR-v0.2-02, -04, -05 + the comment-entity P0 CORR-v0.2-01) cluster in `connector-confluence::normalise`. The Jira normaliser has exactly one (CORR-v0.2-03). The pattern suggests CQL's envelope has more optional fields and the normaliser's "bail loudly on shape drift" discipline — documented in its own module docstring — is inconsistently applied. The DAY-71 invariant (*"silent empty report is the worst outcome"*) survives only when every mandatory field uses `ok_or_else(shape_error(…))?`.

### 4.2 Cross-source symmetry is drifting

`CONS-v0.2-01`, `-02`, `-03`, `-04`, `-05` each flag a place where a new pattern shipped for Atlassian without being back-ported to GitLab (or vice versa): Atlassian got identity seeding but not the `sources_update` + backfill pair; GitLab's `map_status` is missing 404/429 arms; `EntityRef.kind` has three conventions; `ActivityKind` naming has three conventions; 5xx classification diverges between connectors. The symmetry fix for `CONS-v0.2-01` and `-02` is inlined here; the remaining three are about SDK-level conventions and want a single "connector conventions ADR" in v0.3 before the GitHub connector starts.

### 4.3 The template review battery does not exercise the full seam

Both DAY-71 (Phase 3) and DOG-v0.2-01/02 (this review) are cases where per-crate tests + UI tests were green but nothing drove the full `desktop → IPC → Orchestrator → Mux → walker → persist → render → ReportDraft` path end-to-end through the real Rust code. The E2E mock short-circuits the Rust layer entirely; the orchestrator's `fanout.rs` uses `MockConnector` under `LocalGit`/`GitLab` only. `TST-v0.2-05` is the direct follow-up. Until that lands, the only defence against a DAY-71-class regression is manual dogfood.

## 5. Inlined fix narratives

### 5.1 DOG-v0.2-01 — wire `build_source_auth` for Jira/Confluence

**Symptom.** Journey A: user adds Jira + Confluence sources with a shared PAT, sees the "Connected as Vedanth" ribbon, restarts the app, clicks **Generate report** — the entire report fails with `CONNECTOR_UNSUPPORTED_SYNC_REQUEST`. If GitLab sources were also selected, their bullets disappear too (same-run blast radius).

**Root cause.** `apps/desktop/src-tauri/src/ipc/commands.rs:283-313` still had the DAY-74-era stub: `SourceKind::Jira | SourceKind::Confluence => Err(DayseamError::Unsupported { … })`. The stale comment even pointed at DAY-76/79 as the "will land later" work, both of which merged weeks ago. `BasicAuth::atlassian` (DAY-74), `SourceConfig::Jira` / `Confluence` with `email` (DAY-76/79), and the keychain seed (DAY-82) all shipped — but nobody replaced the `Unsupported` arm with a real `BasicAuth` construction.

**Fix.** [`apps/desktop/src-tauri/src/ipc/commands.rs`](../../apps/desktop/src-tauri/src/ipc/commands.rs) — the Jira / Confluence arm now mirrors the GitLab arm: pull `email` from `SourceConfig::{Jira,Confluence}`, resolve `secret_ref` via `state.secrets.get(&secret_store_key(&secret_ref))`, return `Arc::new(BasicAuth::atlassian(email, token, keychain_service, keychain_account))`. Missing-secret-ref and empty-keychain-slot paths reuse the GitLab recovery shape so the user gets the same `action_hint: "reconnect"` with an `atlassian.auth.invalid_token`-style code.

**Test.** `build_source_auth_returns_basic_auth_for_jira_and_confluence_sources` in the same file — constructs a test `AppState`, seeds a Jira source with a keychain-backed PAT, asserts the returned auth downcasts to `BasicAuth`.

### 5.2 DOG-v0.2-02 — hydrate Jira/Confluence muxes at startup + upsert on add

**Symptom.** Even after fixing DOG-v0.2-01 locally, the next run produced a *different* failure: `JiraMux::sync` looked up `ctx.source_id` in its empty `HashMap<SourceId, JiraConnector>` and returned `InvalidConfig` ("no Jira source registered for id …"). The add flow wrote the DB + keychain + identity rows correctly, but never reached into the in-memory orchestrator to register the source with its mux.

**Root cause.** Two symmetric writers are required by the `JiraMux::new` contract, and neither existed: (a) `resolve_registry_config` in `apps/desktop/src-tauri/src/startup.rs` hardcoded `jira_sources: Vec::new()` / `confluence_sources: Vec::new()`, (b) `atlassian_sources_add_impl` had no `mux.upsert(…)` call after the DB/keychain transaction committed. DAY-83's "closes the loop" commit only wired the *default* registry; the E2E mock directly re-derives the report from the in-memory sources array, so the test passed despite the production wiring being broken.

**Fix.**
1. [`apps/desktop/src-tauri/src/startup.rs`](../../apps/desktop/src-tauri/src/startup.rs) — `resolve_registry_config` now matches `(SourceKind::Jira, SourceConfig::Jira { workspace_url, email, .. })` and the Confluence twin, pushing `JiraSourceCfg` / `ConfluenceSourceCfg` onto the accumulating vec.
2. [`apps/desktop/src-tauri/src/ipc/atlassian.rs`](../../apps/desktop/src-tauri/src/ipc/atlassian.rs) — `atlassian_sources_add_impl` now calls `state.orchestrator.jira_mux.upsert(…)` / `confluence_mux.upsert(…)` after the DB/keychain transaction commits, removing the "restart required" toast as an implicit reload signal.

**Test.** `resolve_registry_config_populates_jira_and_confluence_sources` in `startup.rs`; `atlassian_sources_add_upserts_source_into_mux` in `ipc/atlassian.rs`.

### 5.3 DOG-v0.2-03 — enforce `.atlassian.net` in workspace-URL normalisation

**Symptom.** Security-shaped. A user who typos `compay` (or pastes a phishing link) has the PAT shipped to `https://evil.com/rest/api/3/myself` on the **Validate** button press, carrying `Authorization: Basic <base64(email:token)>`. Client validator accepted any dotted hostname; server `parse_workspace_url` only checked `scheme == https`.

**Root cause.** `apps/desktop/src/features/sources/atlassian-workspace-url.ts:57-66` third regex branch silently prepends `https://` to any dotted hostname; `apps/desktop/src-tauri/src/ipc/atlassian.rs:93-140` server-side check is not actually defence-in-depth. The dialog's own doc-comment says *"there is no self-hosted Atlassian Cloud"* — the code didn't pin that.

**Fix.** Both helpers now require the resolved host to end in `.atlassian.net` (case-insensitive, post-IDN):
- [`apps/desktop/src/features/sources/atlassian-workspace-url.ts`](../../apps/desktop/src/features/sources/atlassian-workspace-url.ts) — new `WorkspaceUrlNormalisation::invalid` reason: *"Only Atlassian Cloud hosts (e.g. `company.atlassian.net`) are supported."*
- [`apps/desktop/src-tauri/src/ipc/atlassian.rs`](../../apps/desktop/src-tauri/src/ipc/atlassian.rs) — `parse_workspace_url` now rejects non-Atlassian hosts with `IPC_ATLASSIAN_INVALID_WORKSPACE_URL`.

**Test.** Vitest `evil.com → invalid`, `other-cloud.example.net → invalid`, `acme.atlassian.net → valid` alongside the existing six cases in `atlassian-workspace-url.test.ts`; Rust parser test mirrors each case.

### 5.4 CORR-v0.2-01 — push `confluence_page` entity in `normalise_comment`

**Symptom.** A user who commented on five different Confluence pages in one space on one day sees five bullets rendered correctly, but the rollup collapses them into **one** synthetic `ConfluencePage` artifact whose `page_id` is the sentinel string `"UNKNOWN"`. Evidence popover clicks resolve to that single artifact for every comment.

**Root cause.** `crates/connectors/connector-confluence/src/normalise.rs:333-336` — `normalise_comment` sets `entities = vec![space_entity, comment_entity]`. The rollup at `crates/dayseam-report/src/rollup.rs:165-174` looks specifically for a `confluence_page` entity and falls through to `"UNKNOWN"` when absent. The comment *does* carry `parent_external_id = Some("page:{id}")` via `comment_parent_page_id`, but `orphan_key` only reads `event.entities`, never `parent_external_id`. The rollup module's own invariant #1 ("each issue / page gets its own artifact with its own `event_ids`") is violated in the comment-only case.

**Fix.** [`crates/connectors/connector-confluence/src/normalise.rs`](../../crates/connectors/connector-confluence/src/normalise.rs) — `normalise_comment` now pushes a `confluence_page` entity onto the event's entities vec, extracted via a new `comment_parent_page_ref` helper that robustly pulls the parent page id + title from the comment's `content.ancestors[]` or `content.container` shape. Two additive lines in the hot path; zero changes to `orphan_key`.

**Test.** `confluence_comment_on_different_pages_bucket_into_separate_synthetic_artifacts` in `dayseam-report/src/rollup.rs` tests — emits two `ConfluenceComment` events in the same space pointing at different pages, asserts two groups (not one) and neither's `page_id` is `"UNKNOWN"`.

### 5.5 CONS-v0.2-01 — Atlassian self-identity seeding parity

**Symptom.** An Atlassian PAT rotation via `sources_update` re-writes the keychain but never touches `source_identities`. If the `AtlassianAccountId` row is ever missing — pre-DAY-82 install, partial-rollback bug in a future refactor, orchestrated delete+restore — the render-stage self-filter silently drops every event. DAY-71 silent-empty shape, re-introduced for Atlassian.

**Root cause.** Two sibling writers that exist for GitLab do not exist for Atlassian:
- `commands.rs::sources_update` has a `SourceKind::GitLab` arm that calls `ensure_gitlab_self_identity` using the patched config (or falling back to the persisted config, so a pure label edit or PAT rotation still re-seeds pre-DAY-71 rows). No Atlassian arm.
- `startup.rs` runs `backfill_gitlab_self_identities(&pool)`. No `backfill_atlassian_self_identities` counterpart.

**Fix.**
1. [`apps/desktop/src-tauri/src/ipc/commands.rs`](../../apps/desktop/src-tauri/src/ipc/commands.rs) — new `ensure_atlassian_self_identity(&state, source_id)` helper, idempotent on the unique index `(person_id, source_id, kind, external_actor_id)`. It re-asserts the existing `AtlassianAccountId` row from the DB rather than re-fetching from the API (no network hop on rename). Called from `sources_update` for `SourceKind::Jira | SourceKind::Confluence`. If the identity is missing, a `tracing::warn!` fires with a clear message so the next boot's backfill picks it up.
2. [`apps/desktop/src-tauri/src/startup.rs`](../../apps/desktop/src-tauri/src/startup.rs) — new `backfill_atlassian_self_identities(&pool)` mirrors `backfill_gitlab_self_identities`, invoked right after it in `resolve_registry_config`.

**Test.** `ensure_atlassian_self_identity_is_idempotent_on_repeat_calls` in `commands.rs` tests; `backfill_atlassian_self_identities_reasserts_existing_identities` in `startup.rs` tests.

### 5.6 TST-v0.2-01 — walk_day auth-mapping tests for Jira + Confluence

**Symptom.** The existing Atlassian auth tests live in `tests/auth.rs` (`validate_auth_401…`, `validate_auth_403…`) and `tests/cloud.rs` (`discover_cloud_maps_401/403…`). Neither `connector-jira/tests/walk.rs` nor `connector-confluence/tests/walk.rs` has a 401/403 case. A well-meaning refactor that converts `walk_day`'s non-success branch to "log + `Ok(SyncOutcome::empty())`" (so a mid-sync token revoke doesn't fail a run hard) would pass every existing test — and re-introduce DAY-71 silent-empty for Atlassian.

**Root cause.** TST-addendum-01 (phase-3 addendum) explicitly called this out as the Phase-4-seeding lens: per-connector + per-auth tests were both green, but the *intersection* wasn't tested.

**Fix.**
- [`crates/connectors/connector-jira/tests/walk.rs`](../../crates/connectors/connector-jira/tests/walk.rs) — `walk_day_maps_401_to_atlassian_auth_invalid_credentials` + `walk_day_maps_403_to_atlassian_auth_missing_scope`. Each mounts `POST /rest/api/3/search/jql` with the target status and asserts the returned `DayseamError` is `Auth { code: ATLASSIAN_AUTH_INVALID_CREDENTIALS | ATLASSIAN_AUTH_MISSING_SCOPE }`.
- [`crates/connectors/connector-confluence/tests/walk.rs`](../../crates/connectors/connector-confluence/tests/walk.rs) — same shape, mounting `GET /wiki/rest/api/search` (CQL endpoint).

**Test hook.** The tests *are* the test hook. ≈30 lines per product.

### 5.7 CONS-v0.2-02 — 404/429 arms in `gitlab::map_status`

**Symptom.** A `404` from `/api/v4/projects/:id` for a project the PAT can't see flows into the `_ =>` catch-all of `gitlab::map_status` and gets reported as `gitlab.upstream_5xx` — a flat lie. A `429` that bypasses the SDK's retry loop (future code path that calls `map_status` directly) also falls into `Upstream5xx`. The error-card routing reads the middle segment and renders the wrong copy.

**Root cause.** `crates/connectors/connector-gitlab/src/errors.rs:92-103` — `map_status` only explicitly handles `UNAUTHORIZED`, `FORBIDDEN`, and 5xx. `NOT_FOUND` and `TOO_MANY_REQUESTS` both fall to the catch-all, which lies in the error message. Atlassian's equivalent (`connector-atlassian-common::errors::map_status`) handles all of `{401, 403, 404, 429, 5xx}` explicitly.

**Fix.**
1. [`crates/dayseam-core/src/error_codes.rs`](../../crates/dayseam-core/src/error_codes.rs) — new `GITLAB_RESOURCE_NOT_FOUND = "gitlab.resource_not_found"`, registered in the `ALL` array so the snapshot test catches accidental drops.
2. [`crates/connectors/connector-gitlab/src/errors.rs`](../../crates/connectors/connector-gitlab/src/errors.rs) — new `GitlabUpstreamError::ResourceNotFound { message }` variant → `DayseamError::Network { code: GITLAB_RESOURCE_NOT_FOUND }`. `map_status` now routes `NOT_FOUND` to `ResourceNotFound` and `TOO_MANY_REQUESTS` to `RateLimited { retry_after_secs: 0 }` (conservative default; the SDK layer is the source of truth for the real value from the `Retry-After` header).
3. [`crates/connectors/connector-gitlab/src/auth.rs`](../../crates/connectors/connector-gitlab/src/auth.rs) — the explicit 429 arm in `validate_pat_is_active` now has an updated comment explaining it is a *defensive* layer (redundant with the `map_status` routing after this fix, but kept so a future refactor that drops the 429 arm from `map_status` is still caught here).

**Test.** `map_status_routes_404_to_resource_not_found` + `map_status_routes_429_to_rate_limited_with_zero_retry_after_as_conservative_default` in `errors.rs`; `error_taxonomy_matches_design` updated; `error_codes::tests::registry_snapshot` (insta snapshot) regenerated to include the new code.

**Cross-connector parity.** The remaining taxonomy drift — dot-separator depth, `invalid_token` vs `invalid_credentials`, 5xx routing to `Network` vs `UpstreamChanged` — is the subject of [#87](https://github.com/vedanthvdev/dayseam/issues/87) (CONS-v0.2-03/04/05), to be settled in v0.3 before the GitHub connector adds a fifth axis of divergence.

### 5.8 DOG-v0.2-04 — v0.2.0 → v0.2.1 Confluence email upgrade backfill

**Symptom (caught in dogfood, before PR #88 merged).** After upgrading from v0.2.0 to the v0.2.1 build on `DAY-84-v0.2.0-capstone`, a user who had connected Jira + Confluence on v0.2.0 saw the next report render Jira bullets normally but surface *"Atlassian credentials rejected — the email or API token Dayseam stored was rejected by Atlassian"* for Confluence. The shared API token was fine; Jira used it without complaint in the same run.

**Root cause.** v0.2.0 persisted `SourceConfig::Confluence { workspace_url }` — no email field. DOG-v0.2-01's fix required an email (BasicAuth needs both halves), so this PR added `email: String` with `#[serde(default)]` so old rows still deserialise. The deserialised row then had `email = ""`, and `build_source_auth` explicitly refuses empty emails with `atlassian.auth.invalid_credentials` + `action_hint: "reconnect"` — *before* any network call. I had even written a test (`build_source_auth_errors_when_atlassian_email_is_blank`) pinning that behaviour. What I hadn't written was the migration that keeps existing users from hitting it. Jira rows were fine because `SourceConfig::Jira` has carried an `email` field since DAY-76 / v0.2.0.

**Fix.** [`apps/desktop/src-tauri/src/startup.rs`](../../apps/desktop/src-tauri/src/startup.rs) — new `backfill_atlassian_confluence_email` runs on every boot (after the self-identity backfills, before `build_orchestrator`). For each `Confluence` row with empty email, it looks for a sibling Atlassian source (Jira or Confluence) whose `secret_ref` matches *exactly* — same `keychain_service` + `keychain_account` — and copies that sibling's email into the Confluence row via `SourceRepo::update_config`. The `secret_ref`-match is load-bearing: two Atlassian sources that share a keychain slot share a credential by construction (Journey A / Journey C mode 1), which means they share a user, which means they share an email. Matching on workspace URL alone would risk copying identity across two independently-added tenants on the same host.

**Residual failure modes + UX.** Confluence-only installs (no sibling), or installs where the user rebuilt the DB to a different `secret_ref` but kept the row, land in the `tracing::warn!("... user must reconnect manually")` branch — the row's empty email is preserved, `build_source_auth` keeps returning `atlassian.auth.invalid_credentials`, and the Reconnect card is the right UX for that case because the email is genuinely not recoverable from any on-disk state.

**Test.** Five unit tests in `startup.rs`:

1. `..._copies_from_jira_sibling_sharing_secret_ref` — the user's case: Jira has email, Confluence has `email: ""`, shared `secret_ref`, backfill copies across.
2. `..._leaves_row_alone_when_no_sibling` — Confluence-only install, no Jira to copy from; empty email preserved so `build_source_auth` routes to Reconnect.
3. `..._is_noop_when_email_already_present` — idempotent; boot N+1 doesn't corrupt row from boot N.
4. `..._skips_sibling_with_different_secret_ref` — two independent tenants on same host must *not* share emails (identity-leak guard).
5. `..._migrates_raw_v0_2_0_shape_json` — the regression test for this class of bug: writes raw `{"Confluence":{"workspace_url":"..."}}` JSON directly into `sources.config_json` via sqlx (bypassing the struct-literal path the other four tests use) and asserts (a) `SourceRepo::get` still deserialises it via `#[serde(default)]`, (b) the resulting `email` is `""`, (c) the backfill then copies the sibling Jira email across. If any future refactor drops `#[serde(default)]` from `SourceConfig::Confluence::email` — which would silently brick every v0.2.0 install on upgrade — this test fails loudly.

Test 5 is the "test that would have caught this if it had existed before the rollout". The other four prove the *fix* works; test 5 pins the *upgrade contract* with v0.2.0. Tests 1–4 construct `SourceConfig` via struct literals, which serialise with `"email":""` present — a strict superset of v0.2.0's wire shape, so they would not have caught a regression in the serde-default handling.

**Process lesson.** This is the second Medium-caught-as-P0 in this PR — the first was CONS-v0.2-01 (the GitLab self-identity backfill lens applied to Atlassian). The pattern is the same: "we added a required field / row to v0.2.1 and documented with `#[serde(default)]` that old rows will still deserialise, but didn't write the repair step". The deferred issue [#83](https://github.com/vedanthvdev/dayseam/issues/83) should be retitled to explicitly cover "required-field migration sweep": every `#[serde(default)]` on a `SourceConfig::*` field needs either (a) a boot-time repair helper, (b) a migration in `dayseam-db`, or (c) a test that runs the full `sources_list → build_source_auth` path on a fixture-v0.2.0 row and asserts the outcome is a clean Reconnect prompt, not a silent empty report. Picked this up in dogfood purely by accident — the `SILENT` lens' calibration notes should be updated to list "required-at-runtime fields marked `#[serde(default)]` for backcompat" as a specific thing to hunt for.

## 7. Process note for v0.3

The v0.2.0 release beating the capstone review to the punch is a release-workflow policy question, not a review-doc question. The fix is to gate `semver:minor` / `semver:major` labels behind a "capstone review merged" check — either:

- **Policy:** the capstone PR itself must carry the `semver:minor` label, not the second-to-last feature PR. Then the review doc, CHANGELOG entry, and version bump all land atomically.
- **Mechanism:** a release workflow guard that refuses to publish a minor/major bump if the most recent PR to `master` is not titled `DAY-N: vX.Y capstone — hardening review + release`.

v0.3's plan should own this decision. This doc is the record that the Phase 3 addendum's "real dogfood before the review closes" lesson needs to be generalised: *real review before the release closes*.
