# Dayseam v0.1 — Design Document

- **Status:** Approved design, pre-implementation
- **Author:** @vedanthvdev
- **Date:** 2026-04-17
- **Tracker:** [DAY-1](https://github.com/vedanthvdev/dayseam/issues/1)
- **Scope:** First shippable version (v0.1). Everything below is committed scope for v0.1 unless explicitly marked as a later-phase hook.

---

## 1. Purpose

Nobody enjoys writing end-of-day reports. Modern developers work across many tools in parallel — several repos, several merge requests in flight, review threads, issue comments — and reconstructing "what did I actually do today" from memory is slow, inaccurate, and a daily source of friction.

Dayseam assembles the reconstruction for you. It connects to the tools you already use, looks at what you did on a given day, and produces an evidence-linked markdown report you can save to disk or drop into your Obsidian vault. You stay in control: the data stays local, the report is a draft, every generated line can be traced back to its source.

This document captures the v0.1 design: a Mac desktop app that covers local git and self-hosted GitLab, saves markdown reports to disk and Obsidian vaults, and lays the architectural groundwork for every later source, sink, scheduler, and LLM hook without committing to ship them yet.

## 2. v0.1 scope

### 2.1. What ships

- **Platform:** macOS desktop app, Tauri-based (Rust core + React/TypeScript UI). Intel + Apple Silicon universal binary.
- **Sources:** local git (multi-root folder scanning, identity auto-detection); self-hosted GitLab (custom domain, Personal Access Token auth via one-click deep link).
- **Template:** one default — "Dev EOD" — grouped by Merge Requests, Commits, Issues, and Reviews. Backed by a pluggable template engine (Handlebars-based) so v0.2 can ship user-editable templates without code changes.
- **Date handling:** single-day, unrestricted date picker (past, today, future). System-local timezone.
- **Rollup:** commits that match an MR's commit list are rolled up into that MR in the default render. Unmatched commits still appear in the Commits section. A verbose toggle is available but off by default.
- **Output:** one configurable markdown-file sink with one or two destination roots (e.g. a plain folder plus an Obsidian vault folder). Output file is `YYYY-MM-DD.md`. HTML-comment marker blocks preserve pre-existing content in the target file across regenerations.
- **UI:** single-window app, inline setup (no wizard), streaming per-section render with spinners and per-section error cards, macOS-native toasts for ephemeral feedback, in-app log drawer, Light / Dark / System theme.
- **Storage:** SQLite for app state, normalised activity, raw payloads, report drafts, and log entries. macOS Keychain for secrets. Standard macOS application support directory.
- **Never fail silently.** Every collector emits structured progress events. Every error carries a stable code, a user-readable message, and a suggested action.

### 2.2. What v0.1 does NOT ship but IS architected for

- **Additional sources:** GitHub, Jira, Confluence, Slack — added later by implementing the `SourceConnector` trait. No core changes required.
- **Additional sinks:** HTML export, PDF export, Notion, Linear — added by implementing the `SinkAdapter` trait.
- **OAuth:** `AuthStrategy` is a trait from day one; v0.1 ships `PatAuth` only, v0.2+ adds `OAuthAuth`.
- **Scheduler:** `Scheduler` trait exists for daily/weekly automation; no implementation in v0.1.
- **LLM rewrite:** `Rewriter` trait exists; no implementation. Templates are the source of truth in v0.1.
- **Date ranges:** v0.1 is single-day by design; range handling adds a grouping dimension to the renderer that is deferred.
- **Category-based redaction:** v0.1 offers a simple per-repo privacy toggle; regex-based redaction lands when Slack/Jira do.

### 2.3. Explicit non-goals for v0.1

- Windows, Linux, web, or mobile builds.
- Team features, admin dashboards, centralised user management.
- Cloud sync of the activity store. The product is local-first, not local-by-default-with-optional-cloud.
- Auto-send to destinations. Saving is always explicit; drafts are the default.
- An Obsidian plugin. We write markdown files into the vault folder, which Obsidian reads natively.
- Visual regression tests or prompt-injection tests — the former is premature, the latter is meaningless without an LLM.

## 3. Principles

Each principle has a test implication, listed here so drift is caught in CI.

1. **Local-first.** No network call the user did not authorise. Zero telemetry in v0.1. *Test:* network-sandbox integration test confirms no outbound traffic without a configured source.
2. **Read-only sources by default.** GitLab PAT scopes requested at setup time are `read_api`, `read_user`, `read_repository`. *Test:* the deep-link URL fixture is asserted against this exact scope string.
3. **No mandatory account.** Nothing to sign up for to use the app. *Test:* happy-path E2E covers "install → connect local-git → generate report" with zero account creation.
4. **Draft-first.** The sink is never invoked automatically; every save is a deliberate user action. *Test:* generating a report never triggers a sink call; only `report:save` IPC commands do.
5. **Evidence-linked.** Every rendered bullet carries a stable `bullet_id`, mapped by the `Evidence` structure to the `ActivityEvent` ids that fed it. The UI exposes this via click-through. *Test:* golden snapshots assert that each bullet in a rendered report has a matching evidence entry.
6. **Never fail silently.** Every collector emits `starting → in_progress → completed | error`. Every error carries `phase`, `code`, `message`, `retryable`, `action_hint`. *Test:* contract test per connector asserts that every error variant surfaces a UI-renderable structure.
7. **Pluggable.** Sources and sinks are self-contained crates implementing small typed traits. Core does not depend on any specific connector. *Test:* a crate-level dependency audit asserts that `dayseam-core` does not transitively depend on any `connector-*` or `sink-*` crate.
8. **Secrets live in the OS keychain.** Never on disk, never in plaintext logs. *Test:* `Secret<T>` wrapper's `Debug` impl is `"***"`; a lint forbids `format!` arguments that include `Secret<T>` directly.

### 3.1. Privacy / redaction in v0.1

Category-based redaction is deferred to v0.2. For v0.1 we ship two narrow, high-value features:

- **Preview before save.** The user sees the exact markdown that will be written and can edit it in-app before saving. This is the primary redaction mechanism.
- **Per-repo private flag.** A local repo can be tagged `is_private = true`. Events from it still count as activity, but commit titles and branch names render as `[redacted]`. Raw event data is stored unchanged so the evidence panel still shows what fed each line (for the user's own audit).

## 4. Architecture

### 4.1. Process & thread model

Tauri application, single OS process, two coordinated runtimes:

- **WebView (Chromium):** hosts the React UI. Cannot do network or disk work directly.
- **Rust core (async, tokio):** owns connectors, database, keychain, sinks, scheduler trait (dormant), and the IPC command/event dispatcher.

The two communicate only through Tauri IPC. The frontend never imports anything that touches the OS directly; the Rust core never imports React. This asymmetry is intentional — it makes the same core trivially reusable under a CLI, a web shell, or a future Windows/Linux build.

Async Rust is chosen over sync because sync would serialise the connectors: if GitLab fetch and local-git scan ran back-to-back the "streaming per-section render" UX would collapse. Tokio lets both run in parallel with bounded concurrency per upstream endpoint.

**Rationale:**

- Every connector is its own crate, not a module inside `core`. Adding Jira = `crates/connector-jira/` + one registration call. Deleting a connector = `rm -rf`. No cross-contamination of dependencies.
- `dayseam-events` is its own crate because both the orchestrator and each connector produce events, and the frontend log panel subscribes — shared bus, neutral location.
- `test-fixtures` is a sibling of crates, so Rust tests and Playwright E2E both read from the same source of truth for fixture data.
- `ipc-types` is generated code. Rust types in `dayseam-core` derive `ts-rs`; `build.rs` emits TypeScript type definitions. Renames in Rust surface as TS compile errors.

### 4.3. IPC surface

The frontend can only invoke a small set of commands. It never calls connectors or sinks directly.

**Commands:**

```
invoke('sources:list')                                         → Source[]
invoke('sources:add', { kind, config })                        → Source
invoke('sources:remove', { id })                               → void
invoke('sources:rescan', { id })                               → void

invoke('identities:list')                                      → Identity[]
invoke('identities:update', { emails, gitlabUserIds })         → void

invoke('report:generate', { date, sourceIds, template, verbose }) → ReportJob
invoke('report:get', { jobId })                                → ReportDraft
invoke('report:save', { jobId, destinationIds })               → SaveResult

invoke('settings:get')                                         → Settings
invoke('settings:update', { patch })                           → Settings

invoke('logs:tail', { since })                                 → LogEntry[]
```

**Events emitted Rust → TS** (subscribed via Tauri `listen`):

```
'progress'   { jobId, phase, sourceId, status, message?, error?, fetchedCount? }
'log'        { level, ts, source, message, contextJson }
'toast'      { kind: 'success'|'info'|'warn'|'error', message, action? }
```

Every IPC command is idempotent where feasible. `report:generate` returns immediately with a `jobId`; the UI subscribes to `'progress'` events filtered by that id for the streaming render.

### 5.2. SQLite schema (v1)

Managed by `sqlx::migrate!`. All timestamps are ISO-8601 UTC strings; dates are `YYYY-MM-DD` local-timezone strings. JSON columns are plain TEXT with convention names ending in `_json`.

SQLite is opened with `PRAGMA journal_mode = WAL; PRAGMA synchronous = NORMAL; PRAGMA foreign_keys = ON;`.

### 5.3. Invariants (each has a test)

1. **Deterministic event id.** `uuid_v5(namespace = source_id, name = external_id + kind)`. Re-syncing the same day twice produces zero duplicate rows.
2. **Dates are UTC on disk, local at render.** Timezone conversion is the report engine's job; connectors see UTC windows only.
3. **Secrets never enter SQLite.** `sources.secret_ref` is only a Keychain item name; the token is read lazily at sync time via `dayseam-secrets`.
4. **Raw payloads are debug-only.** Reports render from `activity_events` exclusively. Deleting `raw_payloads` never breaks a report.
5. **Report drafts are immutable.** Regeneration creates a new row. History is preserved until the retention policy prunes it.
6. **Privacy flag is sticky.** Events from a repo where `is_private = 1` are inserted with `privacy = 'redacted_private_repo'`. Unticking private requires an explicit re-sync; we do not retroactively rewrite rows.

### 5.4. Data retention (v0.1 defaults)

| Table | Policy | Who prunes |
|---|---|---|
| `raw_payloads` | Keep last 30 days | App startup sweep |
| `log_entries` | Keep last 7 days OR 10,000 rows, whichever is larger | App startup sweep |
| `activity_events` | Not auto-pruned; user-triggered "Forget before date" deferred to v0.2 | Manual (v0.2) |
| `report_drafts` | Keep last 90 days | App startup sweep |

## 6. Connectors

### 6.1. The SDK (`crates/connectors-sdk`)

One trait, one context, one auth abstraction. Every source implements the trait; nothing else in the system knows about specific sources.

```rust
#[async_trait]
pub trait SourceConnector: Send + Sync {
    fn kind(&self) -> SourceKind;
    async fn healthcheck(&self, ctx: &ConnCtx) -> Result<HealthInfo, DayseamError>;
    async fn fetch_day(
        &self,
        ctx: &ConnCtx,
        date: NaiveDate,
    ) -> Result<Vec<ActivityEvent>, DayseamError>;
}

pub struct ConnCtx<'a> {
    pub source_id: SourceId,
    pub identities: &'a [Identity],
    pub auth: Box<dyn AuthStrategy>,
    pub events: EventBus,
    pub raw_store: RawStore,
    pub clock: Clock,
    pub http: HttpClient,
}

#[async_trait]
pub trait AuthStrategy: Send + Sync {
    async fn inject(&self, req: HttpRequest) -> Result<HttpRequest, DayseamError>;
    fn describe(&self) -> AuthDescriptor;
}

pub struct PatAuth { pub keychain_ref: SecretRef }
// v0.2+: pub struct OAuthAuth { ... }
```

**Key design decisions:**

- One sync method, `fetch_day`, not an incremental cursor. v0.1 is date-picker-driven and always fetches a bounded window. Incremental sync arrives with the scheduler in v0.2.
- Progress events flow through `ConnCtx.events`, not through the return value. A connector can emit `starting → fetching MRs → fetching commits → completed` as it works; the UI renders streaming progress without extra threading code.
- The `HttpClient` is pre-configured with retries, timeouts, and structured logging. Connectors never touch `reqwest` directly. Every HTTP request is auto-logged; every 429/5xx response is auto-retried with exponential backoff; every call times out at 30s.
- `raw_store` is an injected dependency so tests can swap in an in-memory implementation without touching SQLite.

### 6.2. GitLab connector (`crates/connector-gitlab`)

#### 6.2.1. Auth (v0.1)

Personal Access Token via a one-click deep link. On "Connect GitLab," the UI asks for a base URL (e.g. `gitlab.example.com`), then opens:

```
https://<base>/-/user_settings/personal_access_tokens?name=Dayseam&scopes=read_api,read_user,read_repository
```

The user creates the token, copies it, and pastes it back. We validate with `GET /api/v4/user`, store `user_id` and `username` in `Source.config`, and store the token in Keychain under `dayseam.gitlab.<source_id>`.

`AuthStrategy` is a trait from day one so v0.2 can add `OAuthAuth` without a rewrite.

#### 6.2.2. `fetch_day(date)` algorithm

The GitLab Events API (`GET /api/v4/users/:user_id/events`) is the spine: it returns pushes, MR events, issue events, and comments in one stream. We enrich per-target for details.

1. Compute UTC window from the local date.
2. Emit `progress { phase: "gitlab.events", status: "starting" }`.
3. Paginate `GET /api/v4/users/:user_id/events?after=<day-1>&before=<day+1>`. The ±1 day buffer handles timezone-boundary events; we filter precisely in step 8.
4. Filter by `action_name` and target to the in-scope set (see mapping table in §6.2.3).
5. Group by target (MR iid, issue iid) to batch detail fetches.
6. Enrich in parallel, concurrency capped at 6:
   - MRs: `GET /projects/:id/merge_requests/:iid` and `/commits` (for rollup matching).
   - Issues: `GET /projects/:id/issues/:iid`.
   - Commit events: `GET /projects/:id/repository/commits/:sha` for each pushed SHA. Bounded to 50 per push; beyond that we summarise as "pushed 200 commits to branch X" without per-commit details.
7. Normalise into `ActivityEvent`s.
8. Filter precisely to the local-day window.
9. Persist raw payloads to `raw_payloads`.
10. Emit `progress { phase: "gitlab.events", status: "completed", fetched_count: N }` and return events.

#### 6.2.3. Event mapping

| GitLab event | `ActivityKind` | Title template |
|---|---|---|
| Push w/ commit SHAs | `CommitAuthored` (one per SHA, author-filtered to identity) | `{commit.title}` |
| MR `opened` | `MrOpened` | `{mr.title}` |
| MR `merged` | `MrMerged` | `{mr.title}` |
| MR `closed` | `MrClosed` | `{mr.title}` |
| MR `approved` | `MrApproved` | `{mr.title}` |
| Note on MR (review thread) | `MrReviewComment` | `comment on {mr.title}` |
| Issue `opened` | `IssueOpened` | `{issue.title}` |
| Issue `closed` | `IssueClosed` | `{issue.title}` |
| Note on issue | `IssueComment` | `comment on {issue.title}` |

Excluded in v0.1 (under feature flags, default off, surfaced in Settings → Advanced): pipelines triggered, wiki edits, snippets, notes on MRs the user is not part of a review thread on.

#### 6.2.4. Rate limiting

Self-hosted GitLab often disables rate limits. GitLab.com enforces 2000 req/min/user. We respect `RateLimit-*` response headers; on a rate-limit response we back off and emit `progress { message: "GitLab rate-limited, waiting Xs" }` so the user sees the cause of the delay.

#### 6.2.5. Error taxonomy

| Code | Meaning | UI surface |
|---|---|---|
| `gitlab.auth.invalid_token` | 401 from `/user` | Source chip red → "Reconnect" deep link |
| `gitlab.auth.missing_scope` | 403 | "GitLab token is missing read_api scope" |
| `gitlab.url.dns` | DNS fail | "Can't reach {host}" |
| `gitlab.url.tls` | Cert error | "Couldn't verify TLS certificate for {host}" |
| `gitlab.rate_limited` | 429 | Toast "Rate-limited, waiting Xs" |
| `gitlab.upstream_5xx` | 5xx | "GitLab returned an error. Retrying…" |
| `gitlab.upstream_shape_changed` | Schema mismatch | "GitLab response didn't match expected shape" |

### 6.3. Local git connector (`crates/connector-local-git`)

Uses `git2` (libgit2 bindings). No process shell-out.

#### 6.3.1. Phase 1 — discovery

Called at setup time and on "Rescan":

1. User adds one or more scan roots (e.g. `~/Code`, `~/work`).
2. Walk each root recursively, depth-limited to 8, skipping `node_modules/`, `.venv/`, `target/`, `.build/`, `dist/`, `build/`, and any directory matching `.git/*`.
3. A directory containing `.git` (or a `.git` file pointing to a worktree) is a candidate.
4. For each candidate, run the equivalent of `git log --since="90 days ago"` via libgit2; collect unique author emails and per-email commit counts.
5. **Identity matching:** return candidates to the UI with their path, commit count by "me" (union of user-approved identity emails), and total recent commit count. Repos with zero "me" commits are pre-unticked but still shown so the user can opt in to a repo they intend to contribute to.
6. Progress emitted per scan root: `progress { phase: "local-git.scan.<root>", status: "..." }`.

Approved repos → `local_repos` table.

#### 6.3.2. Phase 2 — `fetch_day`

For each approved repo:

1. Open with `Repository::open()`.
2. Revwalk all local branches + HEAD + tags (`refs/*`).
3. For each commit where `author.when()` falls in the local-day UTC window and `author.email()` is in the identity email set:
   - Create `CommitAuthored` with `external_id = sha`, `title = first line of message`, `body = remaining lines`, `parent_external_id = None` (filled by the rollup step if GitLab is also connected).
   - If the repo's `is_private = 1`, set `privacy = RedactedPrivateRepo`. Raw stored unchanged; render is redacted.
4. Emit per-repo progress.

**Documented limitations for v0.1:**
- Detached worktrees are not scanned (only the primary repo per path).
- Submodules are not descended into.
- Signed-commit verification is not performed.

### 6.4. Identity application

Connectors do not individually decide "is this me." They receive `&[Identity]` in `ConnCtx`. The filter logic is defined once in the core, ensuring GitLab username match and local-git email match use consistent rules.

### 6.5. Error taxonomy (unified)

Every connector maps its errors to the shared `DayseamError` enum:

```rust
pub enum DayseamError {
    Auth { code: &'static str, message: String, retryable: bool, action_hint: Option<String> },
    Network { code: &'static str, message: String },
    RateLimited { code: &'static str, retry_after_secs: u64 },
    UpstreamChanged { code: &'static str, message: String },
    InvalidConfig { code: &'static str, message: String },
    Io { code: &'static str, path: Option<PathBuf>, message: String },
    Internal { code: &'static str, message: String },
}
```

Every variant has a stable `code` (e.g. `gitlab.auth.invalid_token`, `local_git.repo_locked`) that tests assert on.

## 7. Report engine & sink

### 7.1. `dayseam-report` — the three-stage pipeline

```
Vec<ActivityEvent>  ──[ rollup ]──▶  RolledActivity  ──[ render ]──▶  RenderedReport
```

#### 7.1.1. Stage 1 — rollup

Inputs: all events for the local day from all sources. Outputs: a `RolledActivity` tree where commits are attached to their matching MR when we can prove the match.

**Rule:**

1. Bucket events by kind.
2. For each MR event, read the MR's commit-SHA list (already fetched during GitLab enrichment).
3. For every `CommitAuthored` (from GitLab push OR local-git) whose SHA appears in any MR's commit list, annotate it `rolled_into_mr: Some(mr_external_id)`.
4. If `verbose_mode == false`, suppress rolled-up commits from the rendered output. They remain in the evidence graph so click-through still works.
5. Unmatched commits (WIP branches, unpushed local work) still render in the Commits section.
6. `MrReviewComment` events are never rolled up into the MR they comment on — they represent separate work ("I reviewed Alice's MR").
7. `IssueComment` events are collapsed per target issue: N comments on issue `#42` render as one line `3 comments on #42 — {issue title}`.

**Dedup across sources:** if both GitLab (push event) and local-git emit a `CommitAuthored` for the same SHA, we keep one event — the one with richer metadata (GitLab adds branch and project; local-git may add the full message body). Their `links` and `entities` are unioned.

**Idempotency:** `rollup(rollup(events)) == rollup(events)`. Property-tested.

#### 7.1.2. Stage 2 — template engine

Templates are `.md` files with Handlebars-style placeholders. v0.1 ships the `dev-eod` template bundled into the binary. v0.2 will add user-authored templates in `~/Library/Application Support/Dayseam/templates/`.

**Default `dev-eod.md.hbs`:**

```handlebars
# Work log — {{date.long}}

{{#unless sources_ok}}
> ⚠ Partial report: {{failed_sources_list}} could not be reached.
{{/unless}}

{{#if merge_requests}}
## Merge requests
{{#each merge_requests}}
- [!{{iid}}]({{url}}) — {{title}} ({{state_label}}){{#if commit_count}} · {{commit_count}} commits{{/if}}{{#if rolled_up_commits}} · [{{rolled_up_commits.length}} rolled-up commits]{{/if}}
{{/each}}
{{/if}}

{{#if commits}}
## Commits
{{#each commits_by_repo}}
- **{{repo_label}}**: {{summary_line}} ({{commit_count}} commits)
{{#each commits}}
  - `{{sha_short}}` {{title}}
{{/each}}
{{/each}}
{{/if}}

{{#if issues}}
## Issues
{{#each issues}}
- [#{{iid}}]({{url}}) — {{title}}{{#if notes_count}} · {{notes_count}} comments{{/if}}
{{/each}}
{{/if}}

{{#if review_activity}}
## Reviews
{{#each review_activity}}
- [!{{iid}}]({{url}}) — {{action}} on "{{title}}" by {{author}}
{{/each}}
{{/if}}

{{#unless has_any_activity}}
*No activity found for this date across the configured sources.*
{{/unless}}
```

**Why Handlebars:** boring, predictable, well-maintained Rust crate. Users who have ever touched a dashboard tool know the syntax.

**Available helpers** (deliberately small to avoid Turing-complete templates): `date.long`, `date.short`, `date.iso`, `sha_short` (first 7), `{{#if}}`/`{{#each}}`/`{{#unless}}`, and `{{link label url}}` (safe URL escape).

No arbitrary expressions. Templates are data; imported text never executes.

#### 7.1.3. Stage 3 — rendering

Emits:

```rust
pub struct RenderedReport {
    pub markdown: String,
    pub sections: Vec<RenderedSection>,
    pub evidence: Vec<Evidence>,
    pub source_states: HashMap<SourceId, SourceRunState>,
    pub metadata: RenderMeta,
}
```

Each `RenderedSection` owns `Vec<RenderedBullet>`, each with a stable `bullet_id`. The UI uses this id for click-through; `Evidence` maps it to the contributing `event_ids`.

### 7.2. Template versioning

Each template has `template_id` (e.g. `dev-eod`) and `template_version` (e.g. `1.0.0`). Stored on the draft. Changing the default template's render output in v0.2 bumps the version; old drafts preserve the version they were generated with. Golden snapshot tests are keyed by `(template_id, template_version)`.

### 7.3. `sink-markdown-file`

Handles both the plain markdown folder and the Obsidian vault folder — they are two configured destination roots of one sink. Frontmatter is a per-destination toggle, off by default (adding frontmatter to an existing note can collide with Obsidian daily-notes plugin's own frontmatter).

#### 7.3.1. Marker block contract

```
<!-- dayseam:start 2026-04-17 v1 -->
...rendered markdown...
<!-- dayseam:end 2026-04-17 -->
```

- Marker includes the date so multiple days could theoretically coexist in one file.
- Marker includes `v1` so format changes in v0.2 can be detected.
- Start and end are symmetrical so a simple regex can find both.

#### 7.3.2. Write algorithm

```
fn export(draft, cfg) -> ExportReceipt:
    if !cfg.dest_dir.exists():
        mkdir -p cfg.dest_dir    # auto-create missing parent directories

    path = cfg.dest_dir / format!("{}.md", draft.date)
    new_block = format!(
        "<!-- dayseam:start {date} v1 -->\n{body}\n<!-- dayseam:end {date} -->",
        date = draft.date,
        body = rendered_markdown,
    )

    if !path.exists():
        atomic_write path = frontmatter_if_enabled() + new_block + "\n"
        return Created

    current = read path
    match find_dayseam_block(current):
        Some((start, end)) =>
            atomic_write path = current[..start] + new_block + current[end..]
            return UpdatedMarkerBlock { preserved_bytes: ... }
        None =>
            atomic_write path = current + "\n\n" + new_block + "\n"
            return AppendedNewSection { preserved_bytes: current.len() }
```

**Atomic write:** write to `path.md.dayseam-tmp`, fsync, rename. If rename fails, the original file is untouched; a toast surfaces the cause.

**Parent-directory policy:** if the destination directory does not exist, we create it with `mkdir -p` semantics. This is safe because the user explicitly configured the destination path; we are never creating a directory the user did not nominate.

#### 7.3.3. Save UX

1. User clicks Save.
2. Per destination: toast `info` "Saving to {path}…".
3. On success: toast `success` "Saved to {path} — updated marker block, preserved N bytes of your notes."
4. On failure: toast `error` "Couldn't save to {path}: {reason}" with a Retry action.

#### 7.3.4. Sink error codes

| Code | Meaning |
|---|---|
| `sink.fs.not_writable` | EACCES on destination |
| `sink.fs.destination_missing` | Destination folder does not exist |
| `sink.malformed_marker` | Existing file has partial/duplicate markers — hard fail, do not write |

#### 7.3.5. Tests (high priority — user data loss risk)

- New file, no frontmatter.
- New file, frontmatter enabled.
- Existing file with existing marker block → outside content preserved byte-for-byte.
- Existing file without marker block → appended at end with blank line separation.
- File with two marker blocks → `sink.malformed_marker` error, no write.
- Destination dir missing → auto-created (mkdir -p semantics).
- Destination dir not writable → `sink.fs.not_writable` error.
- Simulated power loss mid-write → temp file cleanup, no corruption.
- Marker-pattern inside a fenced code block → not matched (regex is line-anchored on start of line).

`~/Library/Application Support/Dayseam/`:

### 9.3. Packaging

## 10. Errors, logs, and diagnostics

- Every error carries a stable `code`, a `message`, `retryable`, and an optional `action_hint`.
- Every `DayseamError` variant has at least one test that triggers it and asserts the surface in the UI (progress event or error card).
- Logs are persisted to `log_entries` and surfaced live in the in-app log drawer.
- Toasts are for `success` and `info` moments; error states use per-section cards so they do not fade out of view.
- The log drawer can export the last N entries to a file — useful for GitHub Issue attachments.

## 11. Branching, versioning, release

### 11.1. Branching & PR policy (enforced by GitHub branch protection)

- **Never commit directly to `master`.** PR required, at least one approval, all status checks green, linear history, no force-push.
- **Branch naming:** `DAY-<issue>-<kebab-title>`. Prefix `DAY` is the dayseam tracker prefix. Branch ties back to a GitHub Issue by number.
- **One commit per branch.** Amend and force-push the same branch; the PR timeline remains a single commit.
- **Commit format:**
  ```
  <branch>: <Title>

  <Paragraph explaining what and why.>
  ```
- **PR title:** `DAY-<issue>: <Title>`.
- **PR body template** (committed at `.github/PULL_REQUEST_TEMPLATE.md`): problem, approach, evidence/tests, breaking changes, alpha-build link (auto-injected by the alpha workflow).
- **Branch deletion on merge:** automatic.
- **Exactly one `semver:*` label required before merge** (enforced by `check-semver-label.yml`):
  - `semver:major` — breaking change, bumps `X.y.z`.
  - `semver:minor` — new feature, bumps `x.Y.z`.
  - `semver:patch` — bugfix / refactor, bumps `x.y.Z`.
  - `semver:none` — docs/CI/chore, no release.

### 11.2. Version source of truth

`VERSION` at repo root. Single line, single version string. Initial value: `0.0.0` (nothing shippable yet). `0.1.0` is reserved for the first feature-complete v0.1 release.

Everything else derives from it at release time:
- `Cargo.toml` workspace version.
- `apps/desktop/package.json` version.
- `apps/desktop/src-tauri/tauri.conf.json` version.
- Any other `packages/*/package.json` versions.

### 11.3. Release workflow (`.github/workflows/release.yml`)

Triggers on push to `master`:

1. Identify the PR that produced this merge commit.
2. Read the PR's `semver:*` label. `semver:none` → exit, no release.
3. Read `VERSION`; compute new version by bumping the labelled segment, resetting lower segments.
4. Write `VERSION`; propagate to `Cargo.toml`, `package.json` files, and `tauri.conf.json`.
5. Commit `"chore(release): vX.Y.Z"` on `master` using a deploy-key-authenticated bot. **This is the one exception to "never commit to `master`":** the automated release bump, signed with a key limited to this workflow. The bot is the only principal listed under branch-protection "Allowed to bypass required pull requests." Its deploy key has `contents:write` only for this repo and nothing else. The original PR was the trigger; the bump does not go through another PR because PR-gating the version bump produces an infinite loop.
6. Tag `vX.Y.Z`.
7. Build the Mac app: `cargo tauri build` + codesign + notarise.
8. Generate release notes from PR titles merged since the previous tag.
9. Append to `CHANGELOG.md` ([Keep a Changelog](https://keepachangelog.com/) format).
10. `gh release create vX.Y.Z` with `.dmg` and `.app.zip` artefacts attached.

### 11.4. Alpha pre-release per PR (`.github/workflows/alpha.yml`)

Triggers on PR open / sync:

1. Compute alpha version: `<target>-alpha.<short-sha>`. Example: if `VERSION` is `0.1.0` and the PR is labelled `semver:minor`, alpha is `0.2.0-alpha.abc1234`.
2. Build `cargo tauri build --debug` with ad-hoc codesign (no notarisation — too slow per-PR).
3. Embed the alpha version string in the About dialog for reviewer clarity.
4. Upload `.dmg` and `.app.zip` as GitHub Actions artefacts (30-day retention).
5. A comment on the PR includes:
   - Download links.
   - "How to open an unsigned alpha" instructions.
   - Expected About-dialog version string.
6. On PR close/merge, default artefact retention cleans up.

Artefacts are PR-scoped; there is no long-running `dayseam-alpha.dmg` channel that silently drifts.

### 11.5. Branch protection rules (to enable after the first merge)

- Require pull request before merging (1 approval).
- Require status checks to pass: `rust`, `frontend`, `e2e`, `check-semver-label`.
- Require linear history.
- Require conversation resolution before merge.
- Include administrators.
- Allow squash-and-merge only (one commit per PR in `master` history).
- Automatically delete head branches.

## 13. Test strategy

### 13.1. Rust core

- **Unit tests** — `cargo test`, colocated. Every pure function has tests: normalisation, rollup, dedup, marker-block replacement, template rendering.
- **Golden / snapshot tests** — `insta`. Fixture events → rendered markdown → compared to checked-in `.snap`. Template drift caught in CI.
- **Contract tests with recorded fixtures** — `wiremock`. Recorded GitLab payloads in `test-fixtures/gitlab/` are replayed against the connector. Schema drift is a deliberate fixture update, not a flaky test.
- **Property-based tests** — `proptest` for rollup idempotency and dedup commutativity.

### 13.2. Frontend

- **Vitest** — component and unit tests.
- **React Testing Library** — behaviour tests ("clicking Generate shows spinners, then replaces them").
- **MSW** — mocks Tauri IPC so frontend tests run without the Rust side.

### 13.3. End-to-end

- **Playwright** with Tauri's webdriver. One happy path in v0.1: add a mock GitLab source + a fixture local-git repo → pick a date → generate → verify marker block written to a temp destination.

### 13.4. Coverage targets

- Rust crates: **≥ 85%** line coverage, enforced in CI via `cargo-llvm-cov`.
- Frontend: **≥ 75%** line coverage, Vitest coverage threshold.
- Connectors: every fixture shape must have a normalisation test (not a percentage — a structural requirement).

### 13.5. Pre-commit + CI gates (all must pass)

1. `cargo fmt --check`
2. `cargo clippy -- -D warnings`
3. `cargo test` across the workspace
4. `cargo-llvm-cov` threshold
5. `pnpm -r lint`
6. `pnpm -r test`
7. `pnpm e2e` (one path)
8. `cargo audit`
9. `cargo deny check`
10. `check-semver-label`

### 13.6. Explicit test non-goals for v0.1

- Visual regression (Percy/Chromatic) — premature.
- Load / performance benchmarks — premature.
- Prompt-injection hardening — no LLM yet.
- Migration tests — there is only a v1 schema; migration tests begin when v2 is introduced.

## 14. Risk register

| Risk | Likelihood | Impact | Mitigation in v0.1 |
|---|---|---|---|
| GitLab Events API does not cover everything we need | medium | high | Per-endpoint fallbacks for MRs/issues. Real-instance fixtures catch schema drift. |
| Token accidentally logged | low | very high | `Secret<T>` wrapper's Debug is `"***"`. Custom lint forbids `format!` of `Secret<T>`. Token reads are centralised in `dayseam-secrets`. |
| Marker-block bug causes user data loss | low | very high | Atomic writes. Exhaustive file-state tests. `sink.malformed_marker` hard fails rather than overwrites. |
| Unsigned binary blocked by Gatekeeper | high | medium | README instructions for first run; sign before calling v0.1 shipped. |
| SQLite corruption on crash | low | medium | WAL mode, `synchronous=NORMAL`. Migrations via `sqlx`. |
| AGPL scares away contributors | medium | low | Acceptable cost; consistent with commercial intent. CONTRIBUTING.md explains. |
| Performance on huge monorepos (100k+ commits) | medium | medium | `fetch_day` is day-bounded, not repo-size-bounded. libgit2 revwalk with time filter is O(recent). |
| Generated `ts-rs` types drift from committed files | low | low | `build.rs` regenerates; CI fails if generated output differs. |

## 16. Glossary

- **ActivityEvent:** normalised record of one thing the user did on one source.
- **Connector:** crate implementing `SourceConnector`. Fetches from an upstream and normalises into `ActivityEvent`s.
- **Sink:** crate implementing `SinkAdapter`. Writes a `ReportDraft` somewhere.
- **Rollup:** process that attaches commits to their MR at render time so a single unit of work is rendered as a single line by default.
- **Marker block:** HTML-comment-delimited region in a destination file that the sink owns and replaces on each regeneration; everything outside the markers is preserved.
- **Identity:** the set of emails and upstream user ids that represent "me" across all sources.
- **Alpha artefact:** a per-PR build of the Mac app, uploaded to the PR as a downloadable artefact for sandbox testing.
