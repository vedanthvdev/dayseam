# v0.4 GitHub connector + v0.3 carry-over hardening — Implementation Plan

> **For agentic workers:** this plan is executed PR-by-PR. Each task below is one PR on its own branch. Follow the one-commit-per-branch rule: the first change on a branch is a real `git commit`; every subsequent change is `git commit --amend --no-edit` followed by `git push --force-with-lease`. When a step below says "Commit," obey that rule.

**Goal:** at the end of v0.4, a Dayseam user can connect a GitHub account with a single personal-access-token credential and see their GitHub activity (pull-requests opened / merged / closed / reviewed, issues opened / closed / commented / assigned, review comments, PR-linked push activity) fused into the same daily EOD report as their GitLab, local-git, Jira, and Confluence events — with cross-source enrichment that links a GitHub PR titled `CAR-5117: …` to Jira issue `CAR-5117` by the same mechanism v0.2 shipped for GitLab. GitHub PRs and GitLab MRs both get promoted to a first-class `ArtifactPayload::MergeRequest` variant with their own `## Merge requests` section (the v0.3 seam DAY-86 explicitly left open). v0.4 ships as `v0.4.0` at the capstone.

**Why v0.4 and not v0.3.x:** this is **an additive minor release** — new connector, new event kinds, new UI affordance, one new public IPC surface (`github_validate_credentials`). It also pays down the four deferred findings from the v0.3 capstone review (TST-v0.3-01, PERF-v0.3-01, CONS-v0.3-01, CORR-v0.3-01) plus the two v0.3 plan-level scope deferrals that naturally land alongside the GitHub work (first-class `ArtifactPayload::MergeRequest` variant; `EntityKind::Workspace`). Per the v0.2 review §7 / v0.3 process correction, `semver:minor` is carried only by the capstone PR (DAY-101); every feature PR between DAY-92 and DAY-100 ships as `semver:none`, so the release workflow holds fire until the capstone review has run.

**Architecture:** v0.3 shipped four connectors (`connector-local-git`, `connector-gitlab`, `connector-jira`, `connector-confluence`). v0.4 adds one more (`connector-github`) and does **not** introduce a shared parent crate — GitHub's REST / GraphQL surface is distinct enough from both GitLab's and Atlassian's that sharing a common crate would be speculative factoring. Precedent: `connector-atlassian-common` exists because Jira + Confluence share workspace URL parsing, the ADF walker, cloudId discovery, cursor pagination, and 401/403 classification; GitHub shares none of those with either GitLab or Atlassian. If a future v0.5 adds Bitbucket, a `connector-forge-common` crate becomes justifiable; in v0.4 it would be premature.

The orchestrator registers `github` as a new `SourceKind`; the `sources` table learns one new kind value via migration `0006_github_sources.sql`. The `SourceConfig::GitHub` variant carries `{ api_base_url: Url }` where `api_base_url` defaults to `https://api.github.com` for github.com tenants and accepts `https://<host>/api/v3` for GitHub Enterprise Server — the first connector to natively support a non-single-vendor hostname. PAT auth reuses `PatAuth` with a new `PatAuth::github(token)` constructor (the `Authorization: Bearer <token>` header shape) — no new `AuthStrategy` trait impl needed. Identity seeds a single `SourceIdentityKind::GithubUserId` row per source, using `GET /user` at validate time (same shape as GitLab's `/user` self-lookup). The walker polls `GET /users/{login}/events` augmented by targeted `GET /search/issues?q=…involves:<login>+updated:<window>` queries — the GitHub equivalent of Atlassian's JQL-driven day-window query; the REST events endpoint alone is not enough because it truncates to the most recent 90 days and drops types we care about (review comments, review approvals).

v0.4 also delivers **two first-class seams the v0.3 capstone called out as deferred scope**: (a) `ArtifactPayload::MergeRequest` promotion — GitLab MRs move out of the `Commits` roll-up suffix into their own payload + `ReportSection::MergeRequests`, and GitHub PRs land there from day one; (b) `EntityKind::Workspace` — the enum variant DAY-89 deferred now lands with a concrete call site (every GitHub identity references its GitHub account, which is the same "workspace" concept as a GitLab group or Atlassian cloud instance).

**Tech stack additions on top of v0.3.0:** no new top-level crate families. `reqwest` + `wiremock` + `serde` + `tokio` all carry over at their v0.3 versions. One new internal helper: a `LinkHeaderPaginator` lives inside `connector-github::pagination` (parses the `Link: <…>; rel="next"` header shape GitHub uses; ~40 LoC including tests). Base64 / ADF / cursor-paginator helpers are untouched.

**GitHub transport specifics:**

- **Hostname:** `https://api.github.com` (github.com). GitHub Enterprise Server users pass `https://<host>/api/v3` in the `SourceConfig::GitHub::api_base_url` field. v0.4 tests the github.com path end-to-end; GitHub Enterprise is a documented-but-unexercised code path until a user reports issues.
- **Auth header:** `Authorization: Bearer <token>` (classic PATs and fine-grained PATs both work with this header; the `token <token>` legacy shape is equivalent but Bearer is the documented-forward shape).
- **Required scopes (classic PAT):** `read:user`, `repo` (private-repo read), `read:org` (for events on org repos the user belongs to). Fine-grained PATs: read-only `Metadata`, `Issues`, `Pull requests`, `Contents`.
- **Pagination:** `Link` header with `rel="next"`. GraphQL's cursor shape is not used in v0.4 — REST is sufficient and simpler to mock.
- **Rate limits:** 5000/hr authenticated; headers `X-RateLimit-Remaining`, `X-RateLimit-Reset`. 429 with `Retry-After` handled by existing `connectors-sdk::HttpClient` retry loop.
- **Body format:** GitHub issue / PR / review-comment bodies are Markdown already. No normalisation needed — we preserve the first ~140 chars verbatim for the bullet, exactly the shape `connector-gitlab` uses for MR titles.

## Plan / design alignment

v0.4 consumes the contracts v0.3.0 shipped verbatim. The following pieces are referenced by tasks below and **must not drift**:

- `connectors-sdk::{SourceConnector, SyncRequest, SyncResult, SyncStats, Checkpoint, ConnCtx, HttpClient, AuthStrategy, RawStore, Clock}` — v0.4 adds one trait implementor (`GithubConnector`). No new `AuthStrategy` variant needed (`PatAuth::github` is a constructor, not a new struct). `SyncRequest::Day` is implemented; `Range` / `Since` return `Err(DayseamError::Unsupported)` with `CONNECTOR_UNSUPPORTED_SYNC_REQUEST` exactly the way the other four connectors do.
- `dayseam-core::types::{source, identity, activity, artifact, entity}` — v0.4 adds additively: `SourceKind::GitHub`, `SourceIdentityKind::GithubUserId`, `EntityKind::GithubRepo`, `EntityKind::GithubPullRequest`, `EntityKind::GithubIssue`, `EntityKind::Workspace` (the v0.3 deferred variant), `ArtifactKind::GithubPullRequest`, `ArtifactKind::GithubIssue`, `ArtifactPayload::MergeRequest` (the cross-connector variant the v0.3 seam wanted), plus six new `ActivityKind` variants (`GithubPullRequestOpened`, `GithubPullRequestMerged`, `GithubPullRequestClosed`, `GithubPullRequestReviewed`, `GithubIssueOpened`, `GithubIssueClosed`, `GithubIssueCommented`). - `dayseam-secrets` — v0.3's `SecretString` + Keychain wrapper stays frozen. GitHub PATs keyed at `dayseam.github.<source_id>`; one source = one token row, no cross-source sharing (no Journey A / B / C split — GitHub is single-product).

- `dayseam-orchestrator` — v0.3's `generate_report` lifecycle, fan-out, and `SyncRun` state machine stay frozen. v0.4 adds `GithubMux` and registers the GitHub `SourceKind` in the default registry; no public-type change.

- `dayseam-report` — Task 7 reshapes the renderer to render `## Merge requests` as its own section (fed by `ArtifactPayload::MergeRequest`), moves the existing `(rolled into !42)` suffix to be purely a "merged commits rolled into this MR" annotation on the MR bullet, and adds `ReportSection::Workspace` affordance for the `EntityKind::Workspace` case. ## Task 5 (DAY-96): `connector-github::walk` — events walker + normaliser

**Why fifth:** Task 4 provides the shell; Task 5 is where "what happened in GitHub today" actually lives. Keeping it split from Task 4 mirrors the v0.2 precedent (Tasks 4 / 5 for Jira, 7 / 8 for Confluence) and keeps per-PR review scope tight.

**Files to create:**

- `crates/connectors/connector-github/src/walk.rs` — `async fn walk_day(ctx: WalkCtx<'_>, date: NaiveDate, tz: Tz) -> Result<Vec<ActivityEvent>, DayseamError>`. Day-window bounds via `dayseam_core::time::day_bounds_utc`. **Query strategy:** two calls.
  1. `GET /users/{login}/events?per_page=100` — the public events stream covers PushEvent, PullRequestEvent (opened/closed/merged), IssuesEvent (opened/closed), IssueCommentEvent, PullRequestReviewEvent, PullRequestReviewCommentEvent. Paginate via `LinkHeaderPaginator` until a `created_at` value is older than the day-window lower bound. The endpoint only returns public events + events on repos the user has access to via their PAT.
  2. `GET /search/issues?q=involves:<login>+updated:<start>..<end>&per_page=100` — catches private-repo PR / issue activity the events endpoint misses (when the repo owner restricts the events API via org policy), plus older events the 90-day truncation drops.
  Dedup by `(event_type, source_url, created_at)`; the search endpoint's richer data wins on conflict.
- `crates/connectors/connector-github/src/normalise.rs` — maps GitHub event payloads into `ActivityEvent`s. One match arm per event type:
  - `PullRequestEvent.action: "opened"` → `GithubPullRequestOpened`.
  - `PullRequestEvent.action: "closed" + merged: true` → `GithubPullRequestMerged`.
  - `PullRequestEvent.action: "closed" + merged: false` → `GithubPullRequestClosed`.
  - `PullRequestReviewEvent.action: "submitted"` → `GithubPullRequestReviewed { state: "approved" | "changes_requested" | "commented" }`.
  - `IssueCommentEvent` on a PR (has `pull_request` field) → `GithubPullRequestCommented`.
  - `IssueCommentEvent` on an issue (no `pull_request` field) → `GithubIssueCommented`.
  - `IssuesEvent.action: "opened"` → `GithubIssueOpened`.
  - `IssuesEvent.action: "closed"` → `GithubIssueClosed`.
  - `IssuesEvent.action: "assigned"` + `assignee.login == self` → `GithubIssueAssigned`.
  - `PushEvent` **is not directly normalised** — commits render through the existing local-git pipeline when the repo is also a local-git source; GitHub push events that don't correspond to a local-git source are currently out of scope (recorded as an explicit non-goal; the workaround is to add the repo as a local-git source).
- `crates/connectors/connector-github/src/rollup.rs` — **rapid-review collapse**: multiple `GithubPullRequestReviewed` events on the same PR by the same author within `RAPID_REVIEW_WINDOW_SECONDS` (default 60) collapse to one event with `review_count = N` and the final `state`. Direct parallel to `connector-jira`'s rapid-transition collapse.
- `crates/connectors/connector-github/tests/walk.rs` — wiremock fixtures + tests.
- `crates/connectors/connector-github/tests/fixtures/walk/` — recorded JSON responses for at least one of each event type. Start with responses scrubbed of PII that isn't ours.

### Invariants proven by tests

1. **Day window honoured.** Events outside `[start, end)` are dropped. Test: `walk_day_drops_events_outside_window`.
2. **Dedup across endpoints.** An event present in both `/users/{login}/events` and `/search/issues` appears exactly once. Test: `walk_day_dedups_events_across_endpoints`.
3. **Rapid-review collapse.** Three approvals within 30s fold to one. Test: `walk_day_collapses_rapid_reviews`.
4. **Self-filter.** An `IssueCommentEvent` where the commenter is **not** the self user is dropped. Test: `walk_day_drops_events_from_other_users`.
5. **Wire-format tolerance.** An `IssuesEvent` with an unknown `action` field emits a warn log + drops the event (does **not** fail the walk — DAY-88 silent-failure discipline applies here too). Test: `walk_day_warns_and_drops_on_unknown_issue_action`.
6. **401 during walk.** A 401 from any paginated endpoint mid-walk fails the walk with `github.auth.invalid_credentials` — **not** "log + empty outcome" (TST-v0.2-01 shape, applied to GitHub). Test: `walk_day_maps_401_to_github_auth_invalid_credentials`.
