# v0.1 Phase 3: GitLab + polish + release — Implementation Plan

> **For agentic workers:** this plan is executed PR-by-PR. Each task below is one PR on its own branch. Follow the one-commit-per-branch rule: the first change on a branch is a real `git commit`; every subsequent change is `git commit --amend --no-edit` followed by `git push --force-with-lease`. When a step below says "Commit," obey that rule.

**Goal:** at the end of Phase 3, Dayseam is a **shippable v0.1.0** that an external user can download from GitHub Releases and run on their own Mac. They can connect a self-hosted GitLab instance with a Personal Access Token, see per-source error cards when something goes wrong (red chip → "Reconnect" deep link), generate a report that fuses GitLab activity with their local-git commits (cross-source deduplicated), save it to a markdown file, and trust that the binary they downloaded has not been tampered with — modulo the one explicit caveat that v0.1.0 ships *unsigned* with a documented Gatekeeper-bypass path, with codesigning + notarisation tracked as a Phase 3.5 / v0.1.1 follow-up.

No scheduler (v0.3), no LLM rewrite (v0.5), no second GitLab auth strategy (OAuth lands in v0.2). Phase 3 closes with **one tagged GitHub release** and the project crossing into post-0.1.0 semver discipline.

**Architecture:** Phase 1 laid the development spine and Phase 2 lit up the local-git slice. Phase 3 plugs in the second connector (`connector-gitlab`), proves the cross-source contract the design has carried since day one (a single `CommitAuthored` may have *both* a GitLab push and a local-git revwalk as evidence; the orchestrator dedups them), wires up the per-source error UI surface that the GitLab connector's richer failure modes finally make load-bearing, and packages the result as a real binary with a real release workflow. The phase ends by formally crossing the v0.1 boundary: the capstone PR carries `semver:minor`, the release workflow auto-tags `v0.1.0`, and from that point on the design's "every feature PR carries a `semver:*` label and each merge auto-releases" rule replaces today's blanket `semver:none` cadence.

**Tech stack additions on top of Phase 2:** `reqwest` 0.12 (already in the dep tree via `connectors-sdk::HttpClient` from Phase 1) gets its first real consumer; `wiremock` 0.6 (Phase 1's contract-test crate, dormant through Phase 2) becomes the spine of `connector-gitlab`'s test suite; `playwright` (via `@playwright/test` and Tauri's webdriver) joins `apps/desktop/` for the one E2E happy path; Tauri's bundler stays at the version Phase 1 pinned and is invoked through `cargo tauri build --target universal-apple-darwin` to produce a universal `.dmg`.

## Plan / design alignment

Phase 3 consumes the contracts Phases 1–2 shipped verbatim. The following pieces are referenced by tasks below and **must not drift**:

- `connectors-sdk::{SourceConnector, SyncRequest, SyncResult, SyncStats, Checkpoint, ConnCtx, HttpClient, AuthStrategy, RawStore, Clock}` — Phase 3 adds one trait implementor (`GitlabConnector`), never a trait change. `GitlabConnector` handles `SyncRequest::Day` and returns `Err(DayseamError::Unsupported)` with `CONNECTOR_UNSUPPORTED_SYNC_REQUEST` for `Range`/`Since`, exactly the way `LocalGitConnector` does today.
- `dayseam-core::types::{source, identity, activity, artifact, run}` — Phase 2 sized `SourceKind`, `ActivityKind`, and `ArtifactKind` to anticipate the GitLab variants documented in design §6.2.3 and §7. Phase 3 *uses* the variants that already exist (`SourceKind::Gitlab`, `ActivityKind::MrOpened`/`MrMerged`/`MrClosed`/`MrApproved`/`MrReviewComment`/`IssueOpened`/`IssueClosed`/`IssueComment`, `ArtifactKind::MergeRequest`/`Issue`/`Thread`); it does not add new enum variants, so the `ts-rs` drift guard in `crates/dayseam-core/tests/ts_types_generated.rs` only sees additive *behaviour* and stays green by accident of design.
- `dayseam-secrets` — Phase 1's `SecretString` + Keychain wrapper. Phase 3's PAT lives at `dayseam.gitlab.<source_id>` per design §13; the redaction lints (PAT never crosses the IPC boundary, never leaks into `log_entries`) carry over from `phase-1-review.md` SEC-04 and `phase-2-review.md` SEC-04.
- `dayseam-events::{RunStreams, AppBus}` — Phase 3 runs use `RunStreams` exactly as shipped. The Phase 2 deferral ARC-03 (RunStreams ownership convergence between `generate_report` and `save_report`) is closed in Task 4 below.
- `dayseam-orchestrator` — Phase 2's `generate_report` lifecycle, fan-out, and `SyncRun` state machine stay frozen at the trait/IPC level. Phase 3 adds two **internal** behaviours: cross-source dedup of `CommitAuthored` events (Task 2) and the per-source error-card payload that the UI consumes (Task 3). Both are pure rollup additions; neither changes a public type.
- `dayseam-db` — Phase 2's migrations `0001_initial.sql`, `0002_artifact_syncrun.sql`, `0003_person_source_identity.sql` stay frozen. All schema changes in Phase 3 are **additive** migrations (`0004_*.sql`) so a laptop upgrading from a Phase 2 build never drops its `state.db`. The Phase 1 `identities` legacy table stays in place; v0.2 retires it.
- The IPC surface — every Phase 3 command goes through `apps/desktop/src-tauri/src/ipc/commands.rs`, is added to `apps/desktop/src-tauri/capabilities/default.json`, and has a typed entry in `@dayseam/ipc-types`. The Phase 1 capability / `Commands` parity guard and the dev-commands feature gate continue to hold.

- [`phase-2-review.md`](../review/phase-2-review.md) §4: ARC-01, ARC-03, MNT-02, PERF-14, TST-05 are all dispositioned "Defer to Phase 3." Task 4 closes ARC-03, MNT-02, PERF-14, TST-05 by name. ## Task 1: `connector-gitlab` — PAT auth + Events API spine

**Why first:** Tasks 2–3 read GitLab events and render them into bullets. Task 4's `RunStreams` convergence has nothing to merge until both sides exist. The error-card UI in Task 3 has no error codes to render until the connector emits them. Landing the connector first means every subsequent task either consumes its output or hardens its surfaces.

**Files to create:**

- `crates/connectors/connector-gitlab/Cargo.toml` — declares `connectors-sdk`, `dayseam-core`, `dayseam-secrets`, `reqwest`, `serde`, `chrono`, `tracing`, plus dev-deps `wiremock`, `tokio`, `serial_test`.
- `crates/connectors/connector-gitlab/src/lib.rs` — re-exports `GitlabConnector`, `GitlabConfig`, `GitlabError`, `register()` (the SDK registration shim that Phase 1 settled on).
- `crates/connectors/connector-gitlab/src/connector.rs` — `GitlabConnector` struct, `SourceConnector` impl. The `sync` method runs the §6.2.2 algorithm. Identity matching reads `SourceIdentity` rows where `kind == SourceIdentityKind::Gitlab` and the `external_actor_id` parses as `i64` matching `event.author.id`.
- `crates/connectors/connector-gitlab/src/auth.rs` — `PatAuth` (consumes `dayseam-secrets::SecretString`, never `String`), `validate_pat(host, pat)` that hits `GET /api/v4/user` and returns `(user_id: i64, username: String)` or one of the `gitlab.auth.*` error codes.
- `crates/connectors/connector-gitlab/src/events.rs` — typed wrappers around the Events API response shape: `GitlabEvent`, `GitlabEventAction` (enum over `pushed`, `opened`, `closed`, `merged`, `approved`, `commented`), `GitlabEventTarget` (enum over `MergeRequest`, `Issue`, `Note`, `Push`). All `serde(rename_all = "snake_case")` to match the API.
- `crates/connectors/connector-gitlab/src/walk.rs` — the day-window walker. Computes UTC bounds from the local date (same helper as `connector-local-git::walk` — extract the helper from `local-git` into `dayseam-core::time` if and only if this is the second use-site, which it is, per MNT-02 reasoning).
- `crates/connectors/connector-gitlab/src/normalise.rs` — maps `GitlabEvent` → `ActivityEvent` per design §6.2.3. Each `ActivityEvent::id` is `deterministic_id(source_id, external_id, kind)` so re-running the same day reproduces the same row verbatim (same invariant that backs `INSERT OR IGNORE` in `dayseam-db::ActivityRepo` post-DAY-52).
- `crates/connectors/connector-gitlab/src/errors.rs` — `GitlabError` enum, conversion `impl From<GitlabError> for DayseamError`, mapping to the seven `gitlab.*` codes from design §6.2.5.
- `crates/connectors/connector-gitlab/tests/sync.rs` — wiremock-backed integration tests, one per error code plus the happy-path "two pushes + one MR + one issue → four ActivityEvents" case.
- `crates/connectors/connector-gitlab/tests/fixtures/` — recorded JSON payloads under `events/`, `merge_requests/`, `issues/`, `commits/`, `user/` mirroring design §4 (`test-fixtures/gitlab/` was the design-time path; the actual crate-local layout is the one tests load relative to `CARGO_MANIFEST_DIR`).
- `crates/dayseam-core/src/error_codes.rs` — add the seven `gitlab.*` codes from design §6.2.5 verbatim.
- `crates/dayseam-core/src/snapshots/dayseam_core__error_codes__tests__registry_snapshot.snap` — accept the snapshot diff in the same commit.

**Files to modify:**

- `Cargo.toml` (workspace) — add `connector-gitlab` to `members`.
- `crates/dayseam-orchestrator/src/registry.rs` — register `connector-gitlab` alongside `connector-local-git`. Boot-only registry hydration (ARC-01) is unchanged; the new entry just appears in the static map.
- `crates/connectors/connector-local-git/src/walk.rs` (only if MNT-02 helper extraction lands in this PR) — replace the inlined day-window helper with a call to the new `dayseam_core::time::day_bounds_utc(local_date, tz)` helper.
- `CHANGELOG.md` — one line under Unreleased.

### Invariants proven by tests

1. **PAT redaction across the IPC boundary.** A test that simulates `sources_add` with a PAT in `config_json` asserts that the PAT never appears in any `log_entries` row, never in any `tracing` event captured by `tracing_test`, and never in the `Source` value returned to the IPC caller. (Mirrors `phase-1-review.md` SEC-01 / SEC-02.) Test: `pat_never_leaks_to_logs_or_ipc`.
2. **Determinism.** Replaying the same recorded fixture day twice produces byte-identical `Vec<ActivityEvent>` (after sorting by `id`). Test: `replay_is_deterministic`.
3. **Identity match by GitLab user id, not username.** A fixture where the user's email rotates but `event.author.id` is stable still attributes the events to `is_self`. The reverse — same email, different user id — does not. Test: `identity_match_uses_user_id`.
4. **Per-target enrichment cap.** A push event with 200 commit SHAs triggers at most 50 commit-detail fetches (design §6.2.2 step 6) and degrades the remainder to a single "pushed N commits to branch X" `ActivityEvent::body`, not 200 separate ones. Test: `large_push_caps_enrichment`.
5. **Rate-limit backoff is observable.** A wiremock 429 response with `RateLimit-Reset: <epoch+5>` produces a `progress { message: "GitLab rate-limited, waiting Xs" }` event (with `X` derived from the header), the connector waits the indicated duration *up to a 30 s ceiling* (so a malicious server cannot hang a run indefinitely), then retries successfully. Test: `rate_limit_emits_progress_and_retries_with_ceiling`.
6. **Each `gitlab.*` error code maps to its documented HTTP signal.** Seven tests, one per row in design §6.2.5. Test: `error_taxonomy_matches_design`.
7. **Day-window precision.** Events with `created_at` in the ±1 day buffer but outside the local-day window are dropped in step 8 of the algorithm (design §6.2.2). Test: `day_window_filter_drops_buffer_events`.
8. **Schema-drift surfaces a typed error, not a panic.** A fixture where the API returns an unexpected field (e.g. `target_type: "WikiPage"`) decodes via `#[serde(other)]` and produces `gitlab.upstream_shape_changed`, never a `serde::de::Error` propagated as a generic `DayseamError`. Test: `unknown_target_type_yields_shape_changed`.
9. **`raw_payloads` honours retention.** Each enriched response is persisted via `RawStore::put` with the raw bytes intact; the existing Phase 1 retention sweep (`dayseam-orchestrator::retention`) prunes them per the configured TTL. No new persistence path. Test: `raw_payloads_round_trip` (mock `RawStore`, assert call count + bytes).
10. **`SyncRequest::Range` and `::Since` return `Unsupported` with the right code.** Same shape as the `connector-local-git` test that holds this line today. Test: `non_day_sync_request_unsupported`.

## Task 2: GitLab enrichment + cross-source `CommitAuthored` dedup + `rolled_into_mr` rollup

**Why next:** Task 1 makes GitLab events *appear* in a draft. Task 2 makes them *correct* when local-git is also connected. The design has carried "if both GitLab (push event) and local-git emit a `CommitAuthored` for the same SHA, we keep one event" since day one (§10.3); Phase 3 is when that contract finally has two producers to dedup between.

**Files to create:**

- `crates/dayseam-report/src/dedup.rs` — pure-function rollup helper: `dedup_commit_authored(events: Vec<ActivityEvent>) -> Vec<ActivityEvent>`. Takes the union of two sources' events; for any pair sharing `(external_id, kind == CommitAuthored)`, keeps the row whose `body` is longer (richest message), unions `links`, and unions `entities`. Pure, no IO; trivial to property-test.
- `crates/dayseam-report/src/rollup_mr.rs` — pure-function rollup helper: `annotate_rolled_into_mr(events: &mut [ActivityEvent], mrs: &[MergeRequestArtifact])`. For each MR, reads its commit-SHA list (already fetched during GitLab enrichment), and for every `CommitAuthored` whose `external_id` matches a SHA, sets `event.parent_external_id = Some(mr.external_id)`. Pure, no IO.
- `crates/dayseam-report/tests/dedup.rs` — property tests: input order does not affect output (sort and compare); idempotent (`dedup(dedup(x)) == dedup(x)`); never duplicates a SHA across sources; never invents a SHA that wasn't in the input; preserves the per-event `privacy` field (the louder side wins; `RedactedPrivateRepo` from local-git beats a non-redacted GitLab push event when the local repo is marked private).
- `crates/dayseam-report/tests/rollup_mr.rs` — golden test: a fixture with two pushes + one MR with two commits → the two `CommitAuthored` rows have `parent_external_id == Some(mr_external_id)`; a third push not in any MR is left at `None`.

**Files to modify:**

- `crates/dayseam-report/src/lib.rs` — re-export `dedup_commit_authored` and `annotate_rolled_into_mr` so the orchestrator can call them.
- `crates/dayseam-orchestrator/src/generate.rs` — between `split_fan_out(per_source)` and the `ActivityRepo::insert_many` call DAY-52 added, run `dedup_commit_authored(events)` and then `annotate_rolled_into_mr(&mut events, &artifacts)`. Order matters: dedup first (so the MR-rollup pass sees the canonical one event per SHA), MR-rollup second.
- `crates/dayseam-report/src/templates/mod.rs` — bump `DEV_EOD_TEMPLATE_VERSION` to `2026-04-22` (or whatever the merge date ends up being). Add a docstring entry: `2026-04-22 — Phase 3 Task 2: cross-source CommitAuthored dedup + rolled_into_mr annotation visible in verbose mode.`
- `crates/dayseam-report/src/templates/dev_eod.rs` — in verbose mode, `CommitAuthored` bullets whose `parent_external_id` is set get a `(rolled into !{mr_iid})` suffix, where the iid is parsed from the MR external_id. Non-verbose mode is unchanged.
- `crates/dayseam-report/tests/golden.rs` — accept snapshot diffs for the existing tests (they all run with single-source today; the snapshots only need a header version bump). Add new test `dev_eod_dedups_commitauthored_across_sources` with a fixture where local-git and GitLab both produced the same SHA: only one bullet appears, with the union of links.
- `CHANGELOG.md` — one line under Unreleased.

### Invariants proven by tests

1. **Dedup is set-like.** `dedup(a ++ b) == dedup(b ++ a)` for any input lists, after sort. Property test, 256 cases.
2. **Dedup never invents or loses a SHA.** `set(sha for e in dedup(input) if e.kind == CommitAuthored) == set(sha for e in input if e.kind == CommitAuthored)`. Property test.
3. **Dedup picks the richer body.** When two events share a SHA, the one with the longer `body` field wins. If both bodies have identical length, tie-break is `source_id` lexicographic (so the choice is deterministic across runs). Test: `dedup_picks_richer_body_with_lex_tiebreak`.
4. **Dedup unions `links` and `entities`.** The kept event has `links = a.links ∪ b.links` and likewise for `entities`. Test: `dedup_unions_links_and_entities`.
5. **Dedup respects privacy.** If either input row is `RedactedPrivateRepo`, the output is `RedactedPrivateRepo`. Test: `dedup_respects_redacted_private_repo`.
6. **MR rollup never invents a parent.** If a `CommitAuthored` SHA is not in any MR's commit list, its `parent_external_id` stays `None`. Property test.
7. **MR rollup is order-independent.** Calling `annotate_rolled_into_mr` twice has no effect beyond the first call. Test: `mr_rollup_is_idempotent`.
8. **Template version bump is captured by the existing snapshot tests.** The seven `golden__dev_eod_*.snap` files all show the new version in their header; this is itself the regression guard against silently bumping the template without bumping the version.

## Task 3: GitLab admin UI + per-source error cards + Reconnect deep link

**Why next:** Tasks 1 and 2 make GitLab *work*. Task 3 makes it *configurable* and *recoverable*. The error cards earn their keep here: GitLab's `gitlab.auth.invalid_token` is the first error in the codebase whose only sensible UI surface is a chip with a clickable "Reconnect" deep link, not a toast that disappears. Local-git errors (path missing, repo locked) were toast-shaped because the user has no recovery action besides "fix the path"; GitLab errors are recovery-shaped, and the UI catches up here.

## Task 5: Playwright E2E happy path + `pnpm e2e` CI job

**Why next:** Tasks 1–4 leave the app feature-complete and internally clean. Task 5 is the first PR that exercises *the user's own click path* — Tauri webview, IPC, real Rust orchestrator, real SQLite, real markdown sink — with a single green-or-red signal. Before release-engineering (Task 6) starts producing a `.dmg`, we want a CI gate that knows whether the built binary still executes the happy path end to end.

> **Scope deviation, recorded 2026-04-20.** The written plan below targets `tauri-driver` + WebDriverIO against the built `.app` bundle on macOS. That path was swapped for **Playwright + Chromium against the `vite preview` bundle with the Tauri IPC boundary mocked in-page**. Rationale:
>
> - WKWebView's WebDriver story on macOS 13+ is still thin enough that `tauri-driver` flakes on every-PR runs; we do not retry-mask (invariant 5.4).
> - The mocked-IPC variant runs on Linux (~30s cold, ~1.2s warm), which lets every PR carry an E2E gate without paying macOS-runner minutes.
> - The Rust side retains its own coverage: `dayseam-orchestrator::multi_source_dedups_commitauthored` covers the cross-source rollup, `sink-markdown-file`'s marker-block tests cover the round-trip. What Playwright adds here is the **frontend click path + IPC contract** surface, which neither Rust test can see.
> - A follow-up issue tracks the real native-driver path (tauri-driver + WebDriverIO) so that coverage is not lost; it just does not block v0.1.0.
>
> Concrete shape that shipped:
>
> - The suite lives in a dedicated root-level pnpm workspace package (`e2e/`, `@dayseam/e2e`) alongside `apps/`, `crates/`, `packages/`. Promoting it out of `apps/desktop/` keeps the test layer legible in the folder tree and keeps neither `@playwright/test` nor `playwright-bdd` sitting in the desktop app's dep closure.
> - Scenarios are **Gherkin `.feature` files** under `e2e/features/<domain>/`, compiled at test time into runnable Playwright specs by [`playwright-bdd`](https://vitalets.github.io/playwright-bdd/) (`bddgen && playwright test`). Every scenario reads as a stack of plain-English Given/When/Then sentences; the catalogue of what E2E covers is human-readable, not buried in `.spec.ts` bodies.
> - Layout (`features/`, `steps/ui-steps/<domain>-steps.ts`, `page-objects/<domain>/<domain>-page.ts` + `<domain>-page-locators.ts`, `fixtures/pages/` + `fixtures/runtime/`) and naming conventions (`@tag` scenarios, `<domain>-*.ts` files) match Modulr's `customer-portal-v2` suite so a reader familiar with one can navigate the other. Page objects keep selectors out of step bodies; `fixtures/base-fixtures.ts` is the single `createBdd(test)` entry point.
> - `e2e/playwright.config.ts` uses `defineBddConfig({ features, steps, statefulPoms: true })` and boots `vite preview` on `127.0.0.1:4173` via `pnpm --filter @dayseam/desktop exec`, Chromium-only, 3-minute per-test budget, `retries: 0`, HTML report uploaded as CI artifact on failure.
> - `e2e/fixtures/runtime/tauri-mock-init.ts` is injected via `page.addInitScript` and mirrors `@tauri-apps/api/mocks`'s public surface (`invoke`, `transformCallback`, `unregisterCallback`, `runCallback`, `plugin:event|{listen,emit,unlisten}`). It drives the `report_generate` flow via `Channel`-emitted progress events and a `report:completed` window event, then captures the `report_save` payload on `window.__DAYSEAM_E2E__.captured.saveCalls` so a Then step asserts the IPC shape end-to-end via the `tauriMock.capturedSaveCalls()` fixture helper.
> - The shipped feature file is `e2e/features/happy-path/generate-and-save-report.feature` (tags `@happy-path @smoke @save-ipc-contract`): onboarded background → Generate → wait for the streaming preview → Save to the markdown sink → assert the receipt → assert the captured save-IPC arg shape → assert `no console or page errors were captured during the run` (filtering only the React DevTools install notice via the `diagnostics` fixture).
> - `.github/workflows/e2e.yml` runs on Ubuntu, caches `~/.cache/ms-playwright` against `pnpm-lock.yaml`, builds the frontend bundle via `pnpm --filter @dayseam/desktop build`, runs `pnpm --filter @dayseam/e2e e2e` (which calls `bddgen` first), and uploads the Playwright HTML report on failure with 14-day retention.
> - `e2e/README.md` documents the rationale, the BDD authoring recipe (feature-first, then step, then page object), the tag-based `--grep` workflow, and the fixture-refresh process.
>
> The "Files to create" / "Files to modify" / "Invariants" / "Steps" lists below describe the original intent. Where the shipped artefact differs, the invariant intent holds (wall-clock budget, no retry-masking, fail-loud on console errors), but the mechanics map onto the Playwright-BDD path documented above.
