# v0.3 Report polish + deferred-findings hardening — Implementation Plan

> **For agentic workers:** this plan is executed PR-by-PR. Each task below is one PR on its own branch. Follow the one-commit-per-branch rule: the first change on a branch is a real `git commit`; every subsequent change is `git commit --amend --no-edit` followed by `git push --force-with-lease`. When a step below says "Commit," obey that rule.

**Goal:** at the end of v0.3, a Dayseam user reading their daily EOD report sees events grouped by source-kind under semantic headings — GitLab commits under *Commits*, GitLab MRs under *Merge requests*, Jira transitions under *Jira issues*, Confluence edits under *Confluence pages* — instead of every artifact kind collapsing into a single `COMMITS` catchall. Reconnecting a rejected Atlassian source reopens the Add-Source dialog pre-filled with workspace URL + email, the way GitLab's reconnect flow already works. Behind the UX, every Medium-severity finding the v0.2 capstone review deferred (22 across 5 lenses) is either resolved, converted into a regression test, or re-deferred with a recorded decision. v0.3 ships as `v0.3.0` at the capstone.

**Why v0.3 and not v0.2.2…v0.2.N:** this is **not a feature phase**. No new connectors, no new event kinds, no new IPC surface. It's a *polish + hardening* phase consuming the 22 Mediums the v0.2 capstone review deferred to umbrella issues [#83](https://github.com/vedanthvdev/dayseam/issues/83)–[#87](https://github.com/vedanthvdev/dayseam/issues/87). Individually each PR is `semver:none` (no auto-release); the phase accumulates into a `semver:minor` capstone that cuts `v0.3.0` so the changelog reads as one coherent "report polish + hardening" release instead of a string of eight patch bumps.

**Why the "v0.2 release beat the capstone review" problem can't repeat here:** the v0.2 `semver:minor` label was on the penultimate feature PR (DAY-83), which auto-released v0.2.0 before DAY-84's review could run. v0.3 applies the lesson from [`docs/review/v0.2-review.md`](../review/v0.2-review.md#7-process-note-for-v03): the `semver:minor` label lives **only on the capstone PR** (DAY-91), never on a feature or polish PR. The feature PRs all carry `semver:none` so that the release, review doc, CHANGELOG entry, and `VERSION` bump land atomically on one merge.

**Architecture:** v0.3 is additive-by-default. The one non-additive area is the per-kind report section rework in DAY-86 — the rendered markdown changes shape (new headings, section ordering). No persisted data moves, no IPC command changes, no SDK trait change. Every other PR is either pure bug fix (CORR-v0.2-02..10, 5xx mapping symmetry), invariant test (registry `kind()` round-trip), index-only schema change (`secret_ref` index), or boot-time repair helper (`#[serde(default)]` migration pattern generalisation).

**Tech stack:** no new crates. `reqwest`, `wiremock`, `sqlx`, `serde`, `tokio`, `ts-rs` all carry over at the versions v0.2 pinned. One new internal module in `dayseam-report` for the per-kind section grouper. One schema migration (`0006_secret_ref_index.sql`) that's an `IF NOT EXISTS` index creation, zero-downtime for upgraders.

## Plan / design alignment

v0.3 consumes the v0.2 contracts verbatim. The following pieces are referenced by tasks below and **must not drift**:

- `connectors-sdk::{SourceConnector, SyncRequest, SyncResult, SyncStats, Checkpoint, ConnCtx, HttpClient, AuthStrategy, RawStore, Clock}` — v0.3 adds zero trait implementors, zero new `AuthStrategy` variants, zero `SyncRequest` cases. Cross-source consistency work (DAY-89) may widen the `DayseamError` taxonomy with symmetrically-mapped 5xx codes but only under the existing `{service}.{category}` convention (`jira.resource_gone`, `confluence.resource_gone`, `gitlab.resource_gone` as a triplet, etc. — no new top-level category).
- `dayseam-core::types::{source, identity, activity, artifact, run, entity}` — DAY-89 adds an explicit `EntityRef.kind` discriminant and a `workspace` entity variant. Both are additive: existing serialised `EntityRef` payloads round-trip with the new field absent (`#[serde(default)]` default is the computed discriminant, preserving every v0.2.1 row), and no existing enum variant is renamed or removed. The `ts-rs` drift guard in `crates/dayseam-core/tests/ts_types_generated.rs` catches any unintended shape change.
- `dayseam-secrets` — v0.2's `SecretString` + Keychain wrapper stays frozen. DAY-89's `secret_ref` index is on the `sources` DB table (indexing the JSON-serialised column), not on the keychain.
- `dayseam-events::{RunStreams, AppBus}` — unchanged.
- `dayseam-orchestrator` — DAY-88's MR↔transition temporal ordering fix lives inside the enrichment pipeline DAY-78 introduced; the public `generate_report` signature is frozen.
- `dayseam-db` — v0.2's migrations `0001…0005` stay frozen. v0.3 adds exactly one migration (`0006_secret_ref_index.sql`) that's an `IF NOT EXISTS` index creation. No destructive changes; any v0.2.1 laptop upgrading to v0.3.0 keeps its `state.db`.
- `dayseam-report` — DAY-86 introduces a new `group_events_by_kind` helper and reshapes the markdown renderer's section loop. The **contract at the renderer's input** (ordered `Vec<Event>` post-enrichment) and **output type** (`String`) are unchanged. Downstream consumers (the obsidian writer, the file writer) see the same `String` shape — new headings, same flat format.
- The IPC surface — v0.3 adds exactly one IPC command, `atlassian_sources_prefill_for_reconnect` in DAY-87. It's added to `apps/desktop/src-tauri/capabilities/default.json` and has a typed entry in `@dayseam/ipc-types` via `ts-rs`. No existing command's signature changes.

Five pieces flagged by the v0.2 capstone review that v0.3 consumes directly:

- [`docs/review/v0.2-review.md`](../review/v0.2-review.md) §5.8 — the `#[serde(default)]`-field-without-paired-migration anti-pattern. DAY-88 generalises the `backfill_atlassian_confluence_email` pattern into a documented migration-helper convention + a `#[serde(default)]` class detector (DAY-90) so the next DOG-v0.2-04-shaped bug gets caught at PR time, not dogfood time.
- [`docs/review/v0.2-review.md`](../review/v0.2-review.md) §7 — the semver-label policy correction. DAY-91's capstone PR (not any other PR) carries `semver:minor`.
- Issue [#85](https://github.com/vedanthvdev/dayseam/issues/85) — "report's section-heading logic isn't tested against a mixed Jira/Confluence/GitLab day". DAY-86 covers this with golden snapshots as a natural side effect of the per-kind section work.
- Issue [#61](https://github.com/vedanthvdev/dayseam/issues/61) — graphify evaluation. **Re-deferred to v0.4.** The three-axis rubric in `docs/decisions/2026-04-20-graphify-deferred.md` still scores low; v0.3 adds zero crates and roughly +2k LoC of test + rendering code. The rubric re-scores meaningfully when the repo gains a new connector family (e.g. GCal, Slack) or crosses ~20 crates, neither of which happens in v0.3. DAY-91's review doc records the decision again.
- [`docs/review/v0.2-review.md`](../review/v0.2-review.md) §5.1 — the `SILENT` lens' calibration gap around required-at-runtime fields marked `#[serde(default)]`. DAY-90 lands the automated class detector so that gap closes at the lint/test layer rather than depending on the reviewer's priors.

## Task 2 (DAY-86): `dayseam-report` — per-kind report sections

**Why this order:** user-visible bug that the user explicitly surfaced in dogfood ("the section heading is still 'COMMITS'" after Jira + Confluence events landed in the same report). Fixing this first means the v0.3 release feels polished even if later PRs slip.

**Current behaviour (v0.2.1):** `dayseam-report` renders every event under a single `## COMMITS` heading because the render loop only distinguishes `ActivityKind::Commit` from "everything else" and lumps the latter together. This is flagged as `DOG-v0.2-05` in [`docs/review/v0.2-review.md`](../review/v0.2-review.md) under [#86](https://github.com/vedanthvdev/dayseam/issues/86) and spotted by the user in the post-hotfix dogfood output.

**Target behaviour:** the rendered markdown carries one section per `SourceKind` × domain pairing, ordered by event volume desc (most-active kind first for the reader's eye):

```
## Commits
- **work/dayseam** (dayseam) — 7e2b0cd Fix DOG-v0.2-04 backfill
...

## Merge requests
- **acme** (acme/backend) — CAR-5117: Production Verification → merged
...

## Jira issues
- **Carbon Team** (CAR) — CAR-5117 Production Verification → Done
- **Carbon Team** (CAR) — CAR-5190 Test affected test plugin → In review
...

## Confluence pages
- **Delivery Tribes** (ST) — Edited page "Kanban Release Process"
- **Delivery Tribes** (ST) — Created page "v0.2.1 Release Notes"
...
```

Empty sections are omitted (no `## Confluence pages` heading if zero pages hit). Section ordering is **fixed** (`Commits → Merge requests → Jira issues → Confluence pages`) rather than volume-sorted — fixed ordering means the user learns to scan their report predictably across days, and the golden snapshot in the test suite doesn't churn when event volumes shift.

**Files to create:**

- `crates/dayseam-report/src/sections.rs` — new module with the `ReportSection` enum (`Commits`, `MergeRequests`, `JiraIssues`, `ConfluencePages`), `section_for_event(&Event) -> ReportSection`, and `group_events_by_section(events) -> BTreeMap<ReportSection, Vec<&Event>>` (BTreeMap for deterministic ordering by `ReportSection`'s derived `Ord`).
- `crates/dayseam-report/tests/snapshots/mixed_kind_day.snap` — golden snapshot of the rendered markdown for a day with 2 commits + 1 MR + 2 Jira transitions + 1 Confluence edit + 1 Confluence page-created. Consumes `insta` (already in the dep tree).

**Files to modify:**

- `crates/dayseam-report/src/lib.rs` — re-export `sections`.
- `crates/dayseam-report/src/render.rs` — replace the single-heading loop with a per-section loop. Preserve the existing "no events at all → `No tracked activity.`" short-circuit (that copy is load-bearing for the E2E mixed-empty-day scenario).
- `crates/dayseam-report/tests/render_integration.rs` — add `mixed_kind_day_renders_each_section` using the new snapshot; keep the existing single-kind tests as baseline.

**Contract changes:**

- **Section order is fixed** — documented at the top of `sections.rs` with a 4-line rationale so a future refactor doesn't "improve" it into volume-sorted and churn every golden snapshot.
- **Empty sections are omitted**, not rendered as `## Jira issues\n\n(none)`. Rationale: the report reads cleaner and dogfood feedback from v0.2 has been "short reports are good; signal per line is what matters".
- **Artifact-less events** (e.g. an `ActivityKind` we haven't grouped yet — future variants) default to a `## Other` section at the bottom. This is the forward-compatibility escape hatch so a new `ActivityKind` doesn't panic the renderer; the presence of a non-empty `## Other` section in dogfood is the signal that we need to extend `section_for_event`.

**Tests (invariants they pin):**

1. `mixed_kind_day_renders_each_section` (`render_integration.rs`) — asserts the mixed-day snapshot. Would fail if sections render in the wrong order, if the "COMMITS" heading regresses, or if empty-section omission breaks.
2. `section_for_event_covers_every_activity_kind` (new, in `sections.rs`) — iterates `ActivityKind::iter()` (from `strum` if available; else an explicit match-is-exhaustive helper), asserts every variant maps to a `ReportSection`, panics the test if a variant falls through to `Other`. This catches the forward-compat trap proactively: add a new `ActivityKind` without updating `section_for_event` → test fails. (If `strum::EnumIter` isn't already derived, add it; it's zero-runtime-cost.)
3. `empty_section_is_omitted` — given a day with only commits, assert no `## Merge requests` heading appears in the output.
4. `no_events_renders_no_tracked_activity` — regression for the existing short-circuit; the phrasing is the one the E2E suite asserts against.

**Verification:** the hardening battery + a visual dogfood of one day from `v0.2.1` usage where the mixed-kind output was wrong. The dogfood screenshot goes into the PR description as before/after.

**Risks / out-of-scope:**

- **Section rename is user-visible.** A v0.2.1 user's Obsidian vault has notes titled with `## COMMITS` inline; the v0.3.0 notes will use `## Commits` (and gain three new headings). This is a cosmetic change only — nothing parses the rendered markdown — but the CHANGELOG entry must call it out so users of the file writer aren't surprised.
- **Per-section pagination / collapse is out of scope.** If a user has 50 Jira transitions in a day, the section renders 50 bullets. Collapsing / grouping-by-ticket is a v0.4 candidate, flagged at the bottom of `sections.rs` with a `// v0.4:` comment.
- **`## Other` escape hatch is not tested with an ActivityKind.** Can't easily add a synthetic variant just for the test; the `section_for_event_covers_every_activity_kind` assertion catches this by refusing to let a real variant fall through.

## Task 3 (DAY-87): Atlassian Reconnect flow parity

**Why this order:** second user-visible bug from the v0.2 capstone dogfood. The Reconnect chip on the `SourceErrorCard` for an Atlassian source currently does nothing — the button is rendered, clicks are caught, no IPC fires, the card stays. [#86](https://github.com/vedanthvdev/dayseam/issues/86) item 2.

**Current behaviour (v0.2.1):** `apps/desktop/src/features/sources/SourceErrorCard.tsx` renders a `Reconnect` button for any source with a `last_health.last_error`. For GitLab, the click opens `AddGitLabSourceDialog` pre-filled with the existing workspace URL. For Atlassian, the click is wired to a handler that sets `editing`, but no Atlassian dialog was mounted against that state — the chip effectively no-ops.

**Target behaviour:** click `Reconnect` on an Atlassian source → `AddAtlassianSourceDialog` opens in **token-only reconnect mode** with:

- workspace URL (from `SourceConfig::Jira.workspace_url` or `SourceConfig::Confluence.workspace_url`) — rendered **read-only**
- email (from `SourceConfig::{Jira,Confluence}.email`) — rendered **read-only**
- product checkboxes hidden (source kind is fixed to the row being reconnected)
- API token field empty and editable (the user is here *because* the token was rejected; don't carry the rejected value forward)

Submitting the dialog writes via the **new** `atlassian_sources_reconnect(source_id, api_token)` command, which rotates the keychain slot atomically. Shared-PAT sources (Jira + Confluence pointing at the same `SecretRef`) are rotated in one call; the IPC returns every affected source id so the frontend can fire `sources_healthcheck` on each to clear the red chips.

**Deviation from the original plan.** The original plan called for a read-only `atlassian_sources_prefill_for_reconnect` IPC plus a reuse of a hypothetical `atlassian_sources_update` write path. In practice no `atlassian_sources_update` exists today, and building full URL/email editability would require re-seeding the `AtlassianAccountId` `SourceIdentity` (otherwise the render-stage self-filter would attribute the rotated account's events to the old person). DAY-87 deliberately narrows to **token-only reconnect** — URL + email are pinned to the row, so the prefill can come from `Source` directly without a separate prefill IPC. If a future PR needs full account-rebinding, it will be a net-new feature (`atlassian_sources_rebind`) with its own identity-migration story, not an extension of this reconnect surface.

**Files created/modified:**

- `apps/desktop/src-tauri/src/ipc/atlassian.rs` — new `atlassian_sources_reconnect` command + `atlassian_sources_reconnect_impl` test-visible helper. Validates the new token against the stored `(workspace_url, email)` via `discover_cloud`, enforces the existing `AtlassianAccountId` identity binding, rotates the keychain slot at the shared `SecretRef`, returns the ids of every sibling source that shared the rotated slot.
- `apps/desktop/src-tauri/src/ipc/commands.rs`, `main.rs`, `build.rs`, `capabilities/default.json` — register the new command on the prod allowlist + Tauri capability surface.
- `packages/ipc-types/src/index.ts` — typed entry for `atlassian_sources_reconnect` and inclusion in `PROD_COMMANDS`.
- `apps/desktop/src/features/sources/AddAtlassianSourceDialog.tsx` — optional `reconnect?: { source: Source }` prop and matching `onReconnected?: (affectedIds: string[]) => void` callback. When set, the dialog locks URL + email, hides the product checkboxes and reuse/paste picker, swaps the submit button to "Reconnect", and routes to the new IPC.
- `apps/desktop/src/features/sources/SourcesSidebar.tsx` — both the ✎ chip affordance and the Reconnect chip now mount the dialog in reconnect mode for Atlassian sources (previously a silent no-op). On success, `sources_healthcheck` fires for every id the IPC returned.
- `e2e/fixtures/runtime/tauri-mock-init.ts` — mocks `atlassian_sources_reconnect` and `sources_healthcheck` so the scenarios can exercise the full click-path. Captured payloads land on `state.captured.atlassianReconnectCalls` for contract assertions.

**Contract changes:**

- **New IPC command `atlassian_sources_reconnect`.** Added to `default.json` capabilities. Typed entry in `@dayseam/ipc-types`. Read/write (rotates a keychain slot + issues a `/rest/api/3/myself` probe). Argument `{ source_id: Uuid, api_token: String }`, return `Vec<SourceId>` — every source whose secret was rotated (≥1, equals 2 for shared-PAT rows).
- **Identity-binding invariant.** A reconnect attempt with a token valid for a *different* Atlassian account fails with `atlassian.auth.invalid_credentials` and `action_hint: "reconnect"`. Rotation is all-or-nothing: the keychain write happens only after the identity check passes.
- **Zero-new-UX for GitLab.** The GitLab reconnect path was built correctly in v0.1 and stays unchanged.

**Tests (invariants they pin):**

1. RTL: `AddAtlassianSourceDialog.test.tsx` — "reconnect mode: prefills URL/email as read-only and clears the token" + "submit calls `atlassian_sources_reconnect` and fires `onReconnected` with affected ids". Would fail if anyone forgets to clear the token (security-adjacent: the rejected token must not be re-submitted), or wires the submit to `atlassian_sources_add` by mistake.
2. RTL: `SourcesSidebar.test.tsx` — "Atlassian reconnect chip mounts dialog in reconnect mode". Would fail if the sidebar-level wiring regresses to the v0.2.1 silent no-op.
3. Rust unit: 5 tests on `atlassian_sources_reconnect_impl` covering pre-network rejects (empty token, missing source, non-Atlassian kind, missing `secret_ref`, empty email from the DOG-v0.2-04 upgrade artefact). Full happy path + identity-mismatch + sibling-rotation paths require live HTTP and are covered by the E2E mock instead.
4. **E2E scenario deferred.** Seeding a broken Atlassian source in the current mock requires new scaffolding (the mock starts from a fixed seed and only gains Atlassian rows through the add-flow). The reconnect IPC + mock handlers are wired so the work is cheap when it lands; the trade-off is that the full click-through is verified by the two RTL tests above rather than a `@atlassian-reconnect` BDD scenario. Logged as follow-up for DAY-90 alongside the rest of the E2E count-aware assertion work.

**Verification:** unit + E2E suites green. Manual dogfood: in the app, delete a token from Keychain, trigger a report, confirm the Reconnect card appears and the click completes the flow end-to-end.

**Risks / out-of-scope:**

- **Token-clearing invariant.** The pre-fill deliberately does not carry the existing (rejected) token forward. The test pins this; a reviewer change that helpfully "auto-fills the token for convenience" is a regression and the test fails.
- **GitLab reconnect is not touched.** Parity is one-directional for this PR (Atlassian → GitLab's level).
- **No new error code.** The Reconnect flow reuses `atlassian.auth.invalid_credentials`; the Reconnect dialog is just a better UX for the same error class.

## Task 4 (DAY-88): Silent-failure sweep

**Why this order:** with the two user-visible bugs behind us, the biggest remaining risk is bugs that *don't* surface in dogfood — the whole `SILENT` lens' raison d'être. [#83](https://github.com/vedanthvdev/dayseam/issues/83) bundles 7 Mediums + 2 Lows; this task takes all 7 Mediums and the two highest-leverage Lows, re-deferring the rest.

**Findings consumed (from [`docs/review/v0.2-review.md`](../review/v0.2-review.md) §5 and the review's appendix):**

- **CORR-v0.2-02** (Medium): Confluence `normalise_page_update` drops the version diff when `body.storage.value` is unchanged but `body.storage.representation` changed (e.g. ADF rewrite via editor upgrade). Should log at `Warn` and emit the event anyway rather than silently dropping.
- **CORR-v0.2-03** (Medium): Confluence rollback events (version N+1 identical to version N-1) render as a plain "edited" event with no rollback indicator. Add a `rolled_back_from` field on `ConfluencePageEdited` and a `:arrow_backward:` prefix in the rendered bullet.
- **CORR-v0.2-04** (Medium): Jira changelog entries for field `status` can include adjacent transitions in the same changelog batch (Atlassian merges sub-second edits). The walker treats the last-wins as the single event, masking the intermediate state. Pair this with the rapid-transition-collapse logic from DAY-77 but preserve the intermediate transition in `JiraTransition::via` for rollup auditing.
- **CORR-v0.2-05** (Medium): Orchestrator's MR↔transition temporal-ordering: when both events fire in the same batch, the enrichment pipeline currently orders by insertion order, which is walker-dependent. Switch to `created_at` with a stable tie-break on `event_id` (the canonical `entity_id + activity_kind + timestamp` digest).
- **CORR-v0.2-06** (Medium): `ActivityKind::ConfluenceComment` without a parent `confluence_page` entity (e.g. a comment whose page was deleted between the fetch and normalise) silently drops the event with no log. Add a `Warn` log + record in the `sync_runs.warnings` array + route to the report's `## Other` section (the DAY-86 forward-compat branch).
- **CORR-v0.2-07** (Medium): Jira `assignee` changelog with `toAccountId` equal to `""` (unassignment) is currently rendered as `assigned to ""` — nonsense. Add the "unassigned" variant to `JiraAssigned` and render as `unassigned from @<fromAccountId>`.
- **CORR-v0.2-08** (Medium): v0.2.1's `backfill_atlassian_confluence_email` is one-off. Generalise the pattern into `dayseam-db::migrations::serde_default_repair` with a documented registration shape, so the next `#[serde(default)]`-field-added-for-backcompat gets a repair helper without inventing the plumbing from scratch. Migrate the existing Confluence email backfill to the new helper to prove it handles the non-trivial case.

## Task 5 (DAY-89): Cross-source consistency

**Why this order:** the invariants this PR pins (`EntityRef.kind`, `ActivityKind` naming, 5xx mapping symmetry) are the ones DAY-88's fixes would otherwise re-reveal as drift if they're not landed alongside. Landing this fifth means the silent-failure sweep just finished is the last time we need to touch those surfaces before the test-quality floor (DAY-90) pins them.

**Findings consumed (from [#87](https://github.com/vedanthvdev/dayseam/issues/87) + [#84](https://github.com/vedanthvdev/dayseam/issues/84) `secret_ref` Medium):**

- **CONS-v0.2-03** (Medium): `EntityRef` is a `HashMap<String, String>` with implicit conventions (`{"kind": "repo", "id": "..."}`). The conventions are documented in [`docs/review/phase-3-review-addendum.md`](../review/phase-3-review-addendum.md) CONS-addendum-04 but not enforced. Make `kind` an explicit `EntityKind` enum discriminant on `EntityRef`; the legacy `HashMap<String, String>` stays as a `fields` sub-field for connector-specific extensions. `#[serde(default)]` on the new `kind` field for upgrader compatibility; add an `EntityRefKindRepair` to the migration helper from DAY-88.
- **CONS-v0.2-04** (Medium): Atlassian events referenced their workspace implicitly via `workspace_url` field on `SourceConfig`. GitLab events carry a `repo` entity. Add a `workspace` entity kind (for Atlassian) as a first-class `EntityRef`, so cross-source rollups can attribute "work done in the Acme workspace" coherently. Populated by the normalisers.
- **CONS-v0.2-05** (Medium): `ActivityKind` naming drift audit. `GitLabCommitAuthored` vs `JiraIssueCreated` vs `ConfluencePageCreated` — the preposition (`Authored`/`Created`/`Created`) is inconsistent. Pick one and rename, or document the pattern explicitly at the top of `activity.rs`. Based on three readings of the events, the natural choice is keep each connector's vernacular (`Authored` for Git, `Transitioned`/`Commented`/`Assigned`/`Created` for Jira, `Created`/`Edited`/`Commented` for Confluence) and document the convention: **each connector gets its own verb family, matching how its product describes the action**. No rename; the doc comment is the contract.
- **CONS-v0.2-06** (Medium): 5xx mapping symmetry. Jira maps 500/502/503/504 to `jira.server_error`; GitLab maps the same to `gitlab.server_error`. Confluence currently falls through to `Err(DayseamError::Internal)` for 5xx. Add `confluence.server_error`. Also add `{service}.resource_gone` for 410 across all three (low-probability but symmetric).
- **PERF-v0.2-01** (Medium, piggy-backed): the `sources` table's `secret_ref` column is JSON-serialised. `build_source_auth` + the DAY-88 repair helper scan the whole table each boot. Add `CREATE INDEX IF NOT EXISTS idx_sources_secret_ref ON sources(secret_ref) WHERE secret_ref IS NOT NULL`. SQLite will use it for the `WHERE secret_ref = ?` lookups the repair helper will do; current scans are O(n) over ~5–20 rows, so the win is forward-compatibility not immediate perf.

**Files to create:**

- `crates/dayseam-db/migrations/0006_secret_ref_index.sql` — the partial index.

**Files to modify:**

- `crates/dayseam-core/src/types/entity.rs` — add `EntityKind` enum + `kind` field on `EntityRef`. `#[serde(default)]` keeps old payloads readable. Corresponding `EntityRefKindRepair` registered with the DAY-88 helper.
- `crates/dayseam-core/src/types/activity.rs` — top-of-module doc comment explaining the verb-family convention. No rename.
- `crates/connectors/connector-atlassian-common/src/errors.rs` — `atlassian.server_error` + `atlassian.resource_gone` (both products share the module).
- `crates/connectors/connector-gitlab/src/errors.rs` — `gitlab.resource_gone`.
- `crates/dayseam-core/src/error_codes.rs` — register all three new codes; snapshot update follows.
- `crates/connectors/connector-confluence/src/normalise.rs` — emit a `workspace` entity alongside `space` and `page`.
- `crates/connectors/connector-jira/src/normalise.rs` — emit a `workspace` entity alongside `project` and `issue`.
- `crates/connectors/connector-gitlab/src/normalise.rs` — emit a `workspace` entity derived from the instance host (symmetric with Atlassian's workspace).

**Tests (invariants they pin):**

1. `entity_ref_kind_roundtrips_via_serde_default` — pins the `EntityKind` addition against the v0.2.1 upgrade path. The DAY-88 migration-helper convention means the test uses a registered repair as its fixture, not ad-hoc JSON insertion.
2. `workspace_entity_is_populated_for_every_event` — iterates every `ActivityKind` via the `section_for_event_covers_every_activity_kind` machinery from DAY-86 and asserts the resulting event carries a `workspace` entity.
3. `server_error_mapping_is_symmetric_across_connectors` — new property test in a workspace-level `tests/` directory: for each of `{jira, confluence, gitlab}` × `{500, 502, 503, 504, 410}`, assert the mapping yields a code of shape `{service}.server_error` or `{service}.resource_gone`. This is the "surface area invariant" the CONS lens was built to protect.
4. Migration `0006` test in `dayseam-db/tests/migrations.rs` — upgrade from `0005` to `0006` on a seed DB, assert the index exists via `SELECT name FROM sqlite_master WHERE type='index'`.

**Verification:** hardening battery + re-run DAY-86 snapshots + new migrations test.

**Risks / out-of-scope:**

- **`EntityRef` shape change is the riskiest thing in v0.3.** Mitigation: `#[serde(default)]` + registered repair + the workspace-wide property test. Upgrade pass from v0.2.1 → v0.3.0 has an explicit integration test that seeds a v0.2.1-shape `EntityRef` and asserts the repair hydrates it.

## Task 6 (DAY-90): Test-quality floor

**Why this order:** we've made five PRs' worth of changes. This is the last pass before the capstone that raises the test coverage floor against the whole cumulative v0.3 diff, giving the capstone's hardening battery a strict bar to verify against.

**Findings consumed (from [#85](https://github.com/vedanthvdev/dayseam/issues/85) + lessons from DOG-v0.2-04):**

- **TST-v0.2-02** (Medium): E2E scenarios use `cy.contains("COMMITS")` and similar substring assertions. They pass if the heading merely appears anywhere in the DOM, not if it reflects the expected count. Switch to count-aware assertions: `cy.get("[data-section='commits']").within(() => cy.get("[data-event]").should("have.length", 3))`. Requires adding stable `data-*` hooks to the renderer output.
- **TST-v0.2-03** (Medium): `ConnectorRegistry::kind()` is derived by pattern-match. There's no invariant test asserting the `SourceKind` → connector implementation round-trip holds for every registered connector. A new connector added without a matching registry entry fails silently. Add `registry_kind_round_trips_for_every_registered_connector` as a unit test.
- **TST-v0.2-04** (Medium): the Atlassian E2E suite runs through the IPC mock (`tauri-mock-init.ts`). No test exercises the real `connector-jira` / `connector-confluence` crates end-to-end against a `wiremock` backend via the orchestrator. Add `dayseam-orchestrator/tests/atlassian_integration.rs` covering a Jira-only, Confluence-only, and both-at-once day. The tests are rust-native (not Playwright) and run in `cargo test --workspace`.
- **TST-v0.2-05** (Medium): `AddAtlassianSourceDialog`'s validate-edit flow (user enters a token, clicks Validate, clicks Edit, changes the URL, clicks Validate again) isn't tested. Regression risk: the second Validate could reuse the first's cached result. Add RTL test + E2E scenario.
- **TST-v0.2-06** (Low, piggy-backed): the `#[serde(default)]` class detector — a compile-time helper that iterates the fields of every `SourceConfig::*`, `EntityRef`, and any other persisted type tagged `#[derive(SerdeDefaultAudit)]`, and fails the workspace build if any field is `#[serde(default)]` without a registered `SerdeDefaultRepair` OR a `#[allow(serde_default_no_repair)]` attribute documenting why one isn't needed. The attribute forces a review decision in the PR instead of letting the pattern slip.

**Files to create:**

- `crates/dayseam-orchestrator/tests/atlassian_integration.rs` — rust-native wiremock-backed end-to-end covering the three Atlassian happy paths.
- `crates/dayseam-macros/` (new crate) — hosts the `SerdeDefaultAudit` derive macro. Single-file proc-macro crate, added to the workspace. ~150 LoC including tests.
- `crates/dayseam-macros/tests/serde_default_audit.rs` — trybuild-style compile-pass + compile-fail tests proving the derive catches the anti-pattern.
- `e2e/features/happy-path/atlassian-reconnect-validate-edit.feature` — BDD scenario for TST-v0.2-05.

**Files to modify:**

- `crates/dayseam-report/src/render.rs` — add `data-section` + `data-event` hooks (invisible in Obsidian; consumed by the E2E DOM queries). Feature-gate or wrap behind a "debug hooks" flag if production render shouldn't carry them; practical answer is they're no-ops in markdown and only visible in a DOM render path — but we don't have a DOM render path yet, so the markdown strings get a tiny structured-comment prefix like `<!-- section:commits -->` which the E2E page-renderer reads.
- `apps/desktop/src-tauri/src/connector_registry.rs` — add the round-trip invariant test.
- `crates/dayseam-core/src/types/source.rs` — `#[derive(SerdeDefaultAudit)]` on `SourceConfig`; paired repair registrations move alongside from DAY-88.
- `crates/dayseam-core/src/types/entity.rs` — `#[derive(SerdeDefaultAudit)]` on `EntityRef`; the DAY-89 repair is the paired registration.

- Every E2E `.feature` that currently asserts on substring matches gets updated to the count-aware API. Estimate: ~15 scenarios, ~30 changed lines.

**Tests (invariants they pin):**

1. `registry_kind_round_trips_for_every_registered_connector` — pins the `SourceKind` ↔ connector implementation bijection.
2. `atlassian_integration.rs` — three scenarios covering the three user journeys, rust-native, catches drift the IPC mock would miss.
3. `validate_edit_does_not_cache_first_validation` — RTL + E2E.
4. `serde_default_audit_catches_missing_repair` — trybuild compile-fail test: define a struct with `#[serde(default)]` on a `String` field, no repair registered, no `allow` attribute. Assert the macro emits a compile error naming the field and the `SerdeDefaultRepair` requirement.
5. `serde_default_audit_accepts_explicit_allow` — trybuild compile-pass: define the same struct with `#[allow(serde_default_no_repair)]` on the field. Assert the macro accepts it and that the explanation text is required (macro enforces that the attribute has a non-empty string argument).

**Verification:** hardening battery + the new trybuild suite runs under `cargo test --workspace`. Manual dogfood: run all E2E scenarios locally (`pnpm e2e`), confirm the count-aware assertions pass.

**Risks / out-of-scope:**
