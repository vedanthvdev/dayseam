# v0.1 Phase 1: Foundations — Implementation Plan

> **For agentic workers:** this plan is executed PR-by-PR. Each task below is one PR on its own branch. Follow the one-commit-per-branch rule: the first change on a branch is a real `git commit`; every subsequent change is `git commit --amend --no-edit` followed by `git push --force-with-lease`. When a step below says "Commit," obey that rule.

**Goal:** Build the development spine so every later phase just drops code into already-wired crates. At the end of Phase 1, `pnpm tauri dev` launches a Mac window with a themed, empty layout; `cargo test --workspace` and `pnpm -r test` both pass; CI is green on every PR; branch protection forbids direct commits to `master`.

**Architecture:** Tauri desktop app (Rust core + React/TS frontend) in a dual-workspace monorepo (Cargo + pnpm). All IO lives in Rust async crates; the frontend only talks through typed Tauri IPC. No user-visible behaviour yet — this phase establishes interfaces, persistence, secret storage, event plumbing, and the app shell.

**Tech Stack:** Rust pinned to 1.88, Tokio, `sqlx` 0.8 (SQLite), `keyring` (macOS Keychain), `reqwest`, `ts-rs` 12, `thiserror`, `insta` (snapshots), `proptest` (property-based), React 18, TypeScript 5, Tailwind 3, Vitest, MSW.

## Plan / design alignment

This plan was drafted before `ARCHITECTURE.md` (merged in [PR #14](https://github.com/vedanthvdev/dayseam/pull/14)) was rewritten against reviewer feedback. The design takes precedence over the plan — per [`docs/plan/README.md`](./README.md), "if a plan step and the design conflict, the design is the contract." Where the two disagreed, this document has been aligned with the revised architecture:

- Task 5 now documents the two-transport event model (`RunStreams` per-run mpsc + `AppBus` broadcast) that actually shipped, not the original single-`EventBus` shape.
- Task 6's `SourceConnector` contract is `sync(SyncRequest) -> SyncResult`, not `fetch_day`. `ConnCtx` carries `run_id`, `person`, `source_identities`, per-run `ProgressSender` / `LogSender`, and a `CancellationToken`.
- Task 7's `SinkAdapter` exposes `SinkCapabilities` so the v0.3 scheduler can refuse to run sinks that aren't `safe_for_unattended`.
- Task 8 now explicitly requires a Tauri capabilities entry for every command.
- Task 9 forwards per-run streams through Tauri `Channel<T>` command arguments and app-wide broadcasts through `Manager::emit`, not a single event channel.
- Task 10 (phase-end hardening + cross-cutting review) is the new gate between Phase 1 and Phase 2.

- [ ] **Step 2.6: Write the error enum with stable codes**

  `crates/dayseam-core/src/error.rs`:
  ```rust
  use thiserror::Error;

  #[derive(Debug, Error, Clone, serde::Serialize, serde::Deserialize, ts_rs::TS)]
  #[ts(export, export_to = "../../packages/ipc-types/src/generated/")]
  #[serde(tag = "variant", content = "data")]
  pub enum DayseamError {
      #[error("auth error [{code}]: {message}")]
      Auth { code: String, message: String, retryable: bool, action_hint: Option<String> },
      #[error("network error [{code}]: {message}")]
      Network { code: String, message: String },
      #[error("rate-limited [{code}]: retry after {retry_after_secs}s")]
      RateLimited { code: String, retry_after_secs: u64 },
      #[error("upstream changed [{code}]: {message}")]
      UpstreamChanged { code: String, message: String },
      #[error("invalid config [{code}]: {message}")]
      InvalidConfig { code: String, message: String },
      #[error("io error [{code}]: {message}{}", path.as_ref().map(|p| format!(" at {}", p.display())).unwrap_or_default())]
      Io { code: String, path: Option<std::path::PathBuf>, message: String },
      #[error("internal error [{code}]: {message}")]
      Internal { code: String, message: String },
  }

  impl DayseamError {
      pub fn code(&self) -> &str {
          match self {
              Self::Auth { code, .. }
              | Self::Network { code, .. }
              | Self::RateLimited { code, .. }
              | Self::UpstreamChanged { code, .. }
              | Self::InvalidConfig { code, .. }
              | Self::Io { code, .. }
              | Self::Internal { code, .. } => code,
          }
      }
  }
  ```

## Task 5: `dayseam-events` — per-run streams + app-wide broadcast ✅ shipped

**Status:** merged in [PR #16](https://github.com/vedanthvdev/dayseam/pull/16). The description below has been rewritten to match what actually landed, so later tasks (Task 6, Task 9) reference the real surface.

**Why:** connectors need ordered, per-run progress and log streams; the Tauri layer needs a second channel for app-wide signals like toasts. Two transports, not one — see `ARCHITECTURE.md` §11.3.

**What shipped:**

- `crates/dayseam-events/Cargo.toml`
- `crates/dayseam-events/src/lib.rs`
- `crates/dayseam-events/src/run_streams.rs` — `RunStreams`, `ProgressSender`, `ProgressReceiver`, `LogSender`, `LogReceiver`
- `crates/dayseam-events/src/app_bus.rs` — `AppBus`, `ToastSubscribeError`
- `crates/dayseam-events/tests/bus.rs` — cross-module integration tests
- Event types (`RunId`, `ProgressEvent`, `ProgressPhase`, `LogEvent`, `ToastEvent`, `ToastSeverity`) are defined in `dayseam-core::types::events` so `ts-rs` can generate the bindings in the same export pass as every other IPC type.

### Invariants (as implemented)

1. **Per-run streams are mpsc, not broadcast.** Each `RunStreams` pairs a `run_id` with a `ProgressSender`/`ProgressReceiver` and a `LogSender`/`LogReceiver` built on `tokio::sync::mpsc::unbounded_channel`. Senders are cheap to clone across tasks; receivers are single-consumer. Ordering within a run is preserved.
2. **Publishers never block.** `send` on a closed channel returns an error the caller can log; it does not panic.
3. **End-of-run is observable.** Once every `ProgressSender` / `LogSender` for a run is dropped, the matching receiver observes `None` and the UI knows the run is finished cleanly.
4. **`AppBus` is broadcast fanout for small app-wide signals.** Built on `tokio::sync::broadcast` with a default capacity of 64. Slow subscribers observe `Lagged` explicitly (they are never silently dropped), and the publisher never blocks.
5. **Every event carries its `run_id` (or `None` for app-global).** Stale events from a superseded run cannot paint over a newer run's UI.

### Surface (sketch)

```rust
// Per-run transport.
pub struct RunStreams { /* run_id, paired sender/receiver handles */ }
impl RunStreams {
    pub fn new(run_id: RunId) -> Self;
    pub fn progress(&self) -> ProgressSender;
    pub fn logs(&self) -> LogSender;
    pub fn take_receivers(self) -> (ProgressReceiver, LogReceiver);
}

// App-wide transport.
#[derive(Clone)]
pub struct AppBus { /* broadcast::Sender<ToastEvent> */ }
impl AppBus {
    pub fn new() -> Self;
    pub fn with_capacity(capacity: usize) -> Self;
    pub fn publish_toast(&self, event: ToastEvent);
    pub fn subscribe_toasts(&self) -> broadcast::Receiver<ToastEvent>;
}
```

### Tests that locked the contract

- `run_streams::tests` — FIFO order, `run_id` stamping, closed-receiver behaviour.
- `app_bus::tests` — fanout, no-subscriber publishes, dropped subscribers.
- `tests/bus.rs` — cloned senders all deliver to the same receiver; receiver yields `None` once every sender is dropped; progress and log streams are independent; a lagged broadcast subscriber observes `Lagged` and recovers without blocking the publisher; multi-publisher multi-subscriber.

Later tasks (Task 6, Task 9) consume this surface directly. If any of the invariants above changes, `ARCHITECTURE.md` §11.3 is the source of truth and this section must be updated in the same PR.

## Task 9: IPC bindings (channels + broadcast) + log drawer + toast system

**Why last:** this is the first PR that ties Rust and TS together end-to-end. It's also the PR that makes the "never fail silently" principle physically visible in the UI — landing it as the capstone of Phase 1 means Phase 2 benefits from the log/toast plumbing from its very first fetch.

Per `ARCHITECTURE.md` §6 and §11.3, the IPC uses **two** Tauri transports, not one:

- **Per-run streams** (progress and log) flow through Tauri `Channel<T>` arguments on the command that starts the run. The Rust side holds a `RunStreams` for each run and forwards each `ProgressEvent` / `LogEvent` into the corresponding `Channel` so the React side observes them in order for that specific `run_id`.
- **App-wide broadcasts** (toasts, and future global signals) flow through `AppBus` on the Rust side and `tauri::Manager::emit` on the frontend boundary. Every window receives every broadcast.

Phase 1 has no real run to stream — Task 6 adds the first — so the plumbing is built and exercised with an internal `dev:emit_*` command that lets the frontend tests fire synthetic events.

**Files to create/modify:**

- Modify: `apps/desktop/src-tauri/src/main.rs` (register commands)
- Create: `apps/desktop/src-tauri/src/ipc/{mod.rs,commands.rs,run_forwarder.rs,broadcast_forwarder.rs}`
- Create: `apps/desktop/src-tauri/src/state.rs` (`AppState` holding `SqlitePool`, `AppBus`, `SecretStore`, and a `RunRegistry` mapping `RunId` to its active `RunStreams` receivers)
- Modify: `apps/desktop/src-tauri/capabilities/default.json` (add every command declared below)
- Modify: `packages/ipc-types/src/index.ts` (add command typings)
- Create: `apps/desktop/src/ipc/{invoke.ts,useRunStreams.ts,useToasts.ts,useLogsTail.ts}`
- Create: `apps/desktop/src/components/{LogDrawer.tsx,ToastHost.tsx,Toast.tsx}`
- Modify: `apps/desktop/src/App.tsx` (mount LogDrawer and ToastHost)
- Create: matching `__tests__` files

### Invariants

1. The frontend cannot call any command whose identifier does not appear in `apps/desktop/src-tauri/capabilities/default.json`. Adding a command requires touching the command, the capabilities file, and the TS `Commands` type in the same PR; any of the three missing fails CI.
2. Progress and log events for run `R` observed by the React hook `useRunStreams(R)` arrive in the exact order `dayseam-events::RunStreams` produced them. Events from a different run can never leak into `R`'s stream.
3. A `ToastEvent` published on `AppBus` reaches every open window.
4. Clicking "View log" (or `⌘L`) opens a drawer showing the last 100 persisted log entries (queried via `logs_tail`), newest first, filterable by level.
5. A `DayseamError` rendered as a toast shows its `code` and message with a Copy-to-clipboard action — never a Rust `Debug` output.

### Steps

- [ ] **Step 9.1: `AppState` wires `AppBus` and a `RunRegistry`**

  ```rust
  pub struct AppState {
      pub pool: sqlx::SqlitePool,
      pub app_bus: dayseam_events::AppBus,
      pub secrets: std::sync::Arc<dyn dayseam_secrets::SecretStore>,
      pub runs: tokio::sync::RwLock<RunRegistry>,
  }

  pub struct RunRegistry { /* RunId -> Handle<RunStreams receivers, cancel token> */ }

  fn main() {
      tauri::Builder::default()
          .setup(|app| {
              let handle = app.handle().clone();
              let rt = tokio::runtime::Handle::current();
              rt.block_on(async move {
                  let pool = dayseam_db::open(&data_dir().join("state.db")).await?;
                  let app_bus = dayseam_events::AppBus::new();
                  let secrets = std::sync::Arc::new(dayseam_secrets::KeychainStore::new("app.dayseam.desktop"));
                  app.manage(AppState {
                      pool,
                      app_bus: app_bus.clone(),
                      secrets,
                      runs: Default::default(),
                  });
                  ipc::broadcast_forwarder::spawn(handle, app_bus);
                  Ok::<_, Box<dyn std::error::Error>>(())
              })?;
              Ok(())
          })
          .invoke_handler(tauri::generate_handler![
              ipc::commands::settings_get,
              ipc::commands::settings_update,

              ipc::commands::dev_emit_toast,        // dev-only, gated behind a feature flag
              ipc::commands::dev_start_demo_run,    // dev-only, exercises the Channel<T> wiring
          ])
          .run(tauri::generate_context!())
          .expect("error while running tauri application");
  }
  ```

## Task 10: Phase 1 hardening + cross-cutting review

**Why last:** Tasks 1–9 each shipped in isolation with their own PR-level review. Before Phase 2 starts piling connectors on top of this scaffold, we deliberately stop, look at the **cumulative diff** from the very first Phase 1 commit to the current `master`, and hunt for the class of bugs that only appears when the pieces are seen together — circular dependencies we didn't notice incrementally, panics one crate makes reachable from another, capabilities gaps, leaked secrets in logs, migrations that don't compose, error codes that overlap, feature flags that accidentally break release builds. Nothing in Phase 2 is blocked on a connector — it's blocked on *Phase 1 being right*. This task is the gate.

> **This task sets the template for every future phase.** See `docs/plan/README.md` for the generalised process. Every phase plan must end with a "Phase N hardening + cross-cutting review" task that repeats this shape, tuned to that phase's domain.

**Intent, not scope:** this task is the one place where we deliberately allow ourselves to fix multiple small findings in a single PR. The single commit that lands it must enumerate each finding and its resolution in the commit body, so the audit trail is preserved even without per-finding PRs. Findings that are large enough to deserve their own PR get their own PR and a reference back from this task.

**Files to create/modify:**

- Create: `docs/review/phase-1-review.md` — the written review artifact described below.
- Modify: any crate where findings trigger a fix.
- Modify: `CHANGELOG.md` — summarise the hardening pass.

### Process

- [ ] **Step 10.1: Inventory**

  List every PR merged into `master` since the Phase 1 baseline (PR #6 `DAY-5-monorepo-scaffold`) in `docs/review/phase-1-review.md`, with the commit SHA range. Capture `git diff <baseline>..master --stat` in the review doc — that is the surface under review.

- [ ] **Step 10.2: Multi-persona deep review**

  Review the cumulative diff through at least these independent lenses. Each lens gets its own section in `docs/review/phase-1-review.md`. If you are running this task as an agentic worker, dispatch one subagent per lens with the full cumulative diff and have each emit a structured finding list; then aggregate.

  - **Correctness.** Logic errors, edge cases, state-machine bugs, error propagation, intent-vs-implementation mismatches. Walk every `unwrap` / `expect` / `panic!` in non-test code and justify or replace each.

  - **Project standards.** `CONTRIBUTING.md` still accurate. `CHANGELOG.md` entries match what shipped. `CLAUDE.md` / `AGENTS.md` rules honoured. Every Tauri command appears in `capabilities/default.json`. Every `#[tauri::command]` has a TS counterpart in `packages/ipc-types`.
