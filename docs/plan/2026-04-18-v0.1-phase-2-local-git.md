# v0.1 Phase 2: Local-git end-to-end — Implementation Plan

> **For agentic workers:** this plan is executed PR-by-PR. Each task below is one PR on its own branch. Follow the one-commit-per-branch rule: the first change on a branch is a real `git commit`; every subsequent change is `git commit --amend --no-edit` followed by `git push --force-with-lease`. When a step below says "Commit," obey that rule.

**Goal:** at the end of Phase 2, `pnpm tauri:dev` is a tool the author uses on their own machine for their own end-of-day reports — not a demo. The user picks a date, clicks Generate, watches progress and log lines stream into the preview from one or more local git repositories, clicks Save, and a marker-block-delimited markdown file lands in one or two configured folders (including an Obsidian vault, because an Obsidian vault *is* "a folder with `.md` files in it" to the sink).

No GitLab (Phase 3), no scheduler (v0.3), no LLM rewrite (v0.5), no signed / notarised bundle (Phase 3). Everything ships behind `semver:none` so we keep accumulating toward the first real `0.1.0` release.

**Architecture:** Phase 1 laid the development spine — core types, DB, secrets, events, connector/sink SDKs, Tauri shell, IPC with per-run `Channel<T>` streams and an `AppBus` broadcast. Phase 2 fills the middle: a first real connector (`connector-local-git`), a first real sink (`sink-markdown-file`), the report engine, and the orchestrator that threads them together behind typed IPC. The end-to-end request lifecycle from `ARCHITECTURE.md` §3.2 goes live for one connector family.

**Tech stack additions on top of Phase 1:** `git2` 0.19 (libgit2 bindings, bundled) for repo walks; `handlebars` 6 for the Dev EOD template renderer; `tempfile` for atomic writes; `insta` golden snapshots continue from Phase 1; `wiremock` stays unused in Phase 2 (no HTTP connector yet) but the SDK contract stands.

## Plan / design alignment

Phase 2 consumes the contracts Phase 1 shipped verbatim — it does not rewrite them. The following shipped pieces are referenced by tasks below and **must not drift**:

- `dayseam-core::types::{source, identity, report, sink, events, activity, repo, settings}` — all Phase 1-shipped public types round-trip through JSON and through `ts-rs`; the drift guard in `crates/dayseam-core/tests/ts_types_generated.rs` stays green.
- `connectors-sdk::{SourceConnector, SyncRequest, SyncResult, SyncStats, Checkpoint, ConnCtx, HttpClient, AuthStrategy, RawStore, Clock, MockConnector}` — Phase 2 adds one trait implementor (`LocalGitConnector`), never a trait change. Incremental / range sync land later; `LocalGitConnector` handles `SyncRequest::Day` and returns `Err(DayseamError::Unsupported)` with `CONNECTOR_UNSUPPORTED_SYNC_REQUEST` for `Range`/`Since`.
- `sinks-sdk::{SinkAdapter, SinkCtx, MockSink, SinkCapabilities, SinkConfig, SinkKind, WriteReceipt}` — Phase 2 adds one trait implementor (`MarkdownFileSink`) and wires it into the orchestrator through the dispatch map documented in the SDK.
- `dayseam-db` — Phase 1's `0001_initial.sql` stays frozen; all schema changes in Phase 2 are **additive** migrations (`0002_*.sql`, `0003_*.sql`) so a laptop upgrading from a Phase 1 build never has to drop its local `state.db`.
- `dayseam-events::{RunStreams, AppBus}` — Phase 2 runs use `RunStreams` exactly as Task 5 defined it; deferred **PERF-08** from [`phase-1-review.md` §3.5](../review/phase-1-review.md#35-performance) is re-assessed in Task 8 under real run volume.
- The IPC surface — every Phase 2 command goes through `apps/desktop/src-tauri/src/ipc/commands.rs`, is added to `apps/desktop/src-tauri/capabilities/default.json`, and has a typed entry in `@dayseam/ipc-types`. The dev-only commands `dev_start_demo_run` and `dev_emit_toast` remain available behind `--features dev-commands`; the real run path replaces *uses* of them, not the commands themselves, until Task 8 formally retires the dev-run UI wiring.

Two pieces flagged in Phase 1 that Phase 2 picks up directly:

- [`phase-1-review.md`](../review/phase-1-review.md) PERF-08 — log amplification on a wedged UI — is formally re-reviewed under real run volume in Task 8. The review either closes it or records the new mitigation.
- Nits left unresolved in [PR #31](https://github.com/vedanthvdev/dayseam/pull/31) (L-B / L-C / N-B / N-C) are not refreshed by the body of Phase 2; Task 8 sweeps them with the rest of the cross-cutting review and either fixes or explicitly defers each.

## Task 1: Schema v2 — `Artifact` / `SyncRun` core types + repos + self-`Person` bootstrap

**Why first:** every other Phase 2 task writes into or reads out of these tables. `connector-local-git` emits `Artifact`s alongside `ActivityEvent`s; the orchestrator keeps a `SyncRun` row per run; the report engine reads artifacts as its middle layer (`ARCHITECTURE.md` §7A). Landing the persistence shape first means Tasks 2–5 only add behaviour.

**Files to create:**

- `crates/dayseam-core/src/types/artifact.rs` — `Artifact`, `ArtifactKind`, `ArtifactId` (all `ts-rs`-exported).
- `crates/dayseam-core/src/types/run.rs` — `SyncRun`, `SyncRunTrigger`, `SyncRunCancelReason`, `PerSourceState` (aligning with `dayseam-core::types::report::SourceRunState` — naming overlap is deliberate; see invariants below).
- `crates/dayseam-db/migrations/0002_artifact_syncrun.sql` — additive tables + new columns on `activity_events` (`artifact_id TEXT NULL`) and on `report_drafts` (`sync_run_id TEXT NULL`). All columns are nullable so the v1 rows a Phase 1 laptop already has migrate cleanly.
- `crates/dayseam-db/migrations/0003_person_source_identity.sql` — additive tables for `persons` and `source_identities`, plus a backfill statement that seeds exactly one `persons` row from the existing `identities.display_name` if present (so the user does not re-enter their name on upgrade).
- `crates/dayseam-db/src/repos/{artifacts,sync_runs,persons,source_identities}.rs` — typed repositories mirroring the Phase 1 pattern (`insert`, `get`, `list`, and the minimum mutation set each task below needs).
- `crates/dayseam-core/src/error_codes.rs` — add `ORCHESTRATOR_RUN_SUPERSEDED`, `ORCHESTRATOR_RUN_CANCELLED`, `DB_SCHEMA_MIGRATION_FAILED`. Adding a constant is additive; do not rename.
- `crates/dayseam-core/tests/serde_round_trip.rs` — extend to cover the new types.

**Files to modify:**

- `crates/dayseam-core/src/types/mod.rs` — wire the two new modules and re-export `Artifact`, `ArtifactKind`, `ArtifactId`, `SyncRun`, `SyncRunTrigger`, `SyncRunCancelReason` at crate root.
- `crates/dayseam-db/src/lib.rs` — re-export the new repos.
- `packages/ipc-types/src/index.ts` — re-export the new generated TS types next to `SinkCapabilities` et al. (STD-03 from Phase 1 already pins the pattern.)
- `CHANGELOG.md` — one line under Unreleased.

### Invariants proven by tests

1. **Additive migrations.** `pool::open` can open a database created by `0001_initial.sql` alone (simulate by running `0001` via `sqlx::migrate!`, closing, and reopening with the new migrations directory); no migration drops or renames a column. Test: `db::migrations_are_additive_and_idempotent`.
2. **Nullable links.** Rows in `activity_events` with `artifact_id IS NULL` are legal (that's exactly what a commit-that-never-became-an-MR looks like). Rows in `report_drafts` with `sync_run_id IS NULL` are legal for drafts created before Phase 2 shipped. Test: `activity_events_allow_null_artifact_id`, `report_drafts_allow_null_sync_run_id`.
3. **`ArtifactId` is deterministic per `(source_id, kind, external_id)`** — same inputs produce the same UUID across runs, just like `ActivityEvent::deterministic_id` (shipped in Phase 1). Test: `artifact_id_is_stable` + `artifact_id_differs_by_kind`.
4. **`SyncRun` state machine.** `status` transitions only `Running → Completed`, `Running → Cancelled`, or `Running → Failed`. `superseded_by` may only be set when `status == Cancelled`. Test: `sync_run_state_machine` rejects illegal transitions at the repo layer.
5. **Self-`Person` bootstrap is exactly-once.** Running the bootstrap twice does not create a second `persons` row; it no-ops. Test: `bootstrap_self_person_is_idempotent`.
6. **FK cascade parity with Phase 1.** Deleting a source cascades to its artifacts and to `source_identities` rows whose `source_id` matches, exactly the way Phase 1's `activity_events` cascade behaves. Test: `deleting_source_cascades_artifacts_and_source_identities`.

- [ ] **1.3 Write `0002_artifact_syncrun.sql`.** `CREATE TABLE artifacts` (columns: `id TEXT PRIMARY KEY`, `source_id TEXT NOT NULL REFERENCES sources(id) ON DELETE CASCADE`, `kind TEXT NOT NULL`, `external_id TEXT NOT NULL`, `payload_json TEXT NOT NULL`, `created_at TEXT NOT NULL`, `UNIQUE(source_id, kind, external_id)`). `CREATE TABLE sync_runs` (columns: `id TEXT PRIMARY KEY`, `started_at TEXT NOT NULL`, `finished_at TEXT`, `trigger TEXT NOT NULL`, `status TEXT NOT NULL`, `cancel_reason TEXT`, `superseded_by TEXT REFERENCES sync_runs(id) ON DELETE SET NULL`, `per_source_state_json TEXT NOT NULL`). `ALTER TABLE activity_events ADD COLUMN artifact_id TEXT REFERENCES artifacts(id) ON DELETE SET NULL;` followed by `CREATE INDEX idx_events_artifact ON activity_events(artifact_id);`. `ALTER TABLE report_drafts ADD COLUMN sync_run_id TEXT REFERENCES sync_runs(id) ON DELETE SET NULL;`.

- [ ] **1.4 Write `0003_person_source_identity.sql`.** `CREATE TABLE persons` (columns: `id TEXT PRIMARY KEY`, `display_name TEXT NOT NULL`, `is_self INTEGER NOT NULL DEFAULT 0`). `CREATE TABLE source_identities` (columns: `id TEXT PRIMARY KEY`, `person_id TEXT NOT NULL REFERENCES persons(id) ON DELETE CASCADE`, `source_id TEXT REFERENCES sources(id) ON DELETE CASCADE`, `kind TEXT NOT NULL`, `external_actor_id TEXT NOT NULL`, `UNIQUE(person_id, source_id, kind, external_actor_id)`). Backfill:

  ```sql

  SELECT id, display_name, 1 FROM identities WHERE NOT EXISTS (SELECT 1 FROM persons WHERE is_self = 1)
  ```

  - `SyncRunRepo::insert`, `::mark_finished`, `::mark_cancelled(reason)`, `::mark_superseded_by(run_id)`, `::get`, `::list_recent(n)`.

## Task 2: `connector-local-git` — libgit2 discovery + `sync(Day)` + fixture repos

**Why next:** with Task 1 landed, the first real `SourceConnector` can emit `ActivityEvent` *and* `Artifact::CommitSet` rows into the database. This is the "first connector" task in `ARCHITECTURE.md` §8.3. Until it merges there is no real run; everything in Task 3+ is written against fixture repos anyway so the tasks can proceed in parallel once the scan-root shape is frozen.

**Files to create:**

- `crates/connectors/connector-local-git/Cargo.toml` — new workspace member. Depends on `connectors-sdk`, `dayseam-core`, `dayseam-events`, `git2` (with `default-features = false` and only the features we need — `bundled` to avoid linking against host libgit2), `chrono`, `serde`, `tokio`, `async-trait`, `tracing`, `thiserror`.
- `crates/connectors/connector-local-git/src/lib.rs`
- `crates/connectors/connector-local-git/src/connector.rs` — `LocalGitConnector` struct + `SourceConnector` impl.
- `crates/connectors/connector-local-git/src/discovery.rs` — recursive walk under each `scan_root`, respecting `.gitignore` semantics at the walker level, bounded by a configurable `max_depth` (default 6) and a configurable `max_roots` (default 512) to keep pathological setups from stalling.
- `crates/connectors/connector-local-git/src/walk.rs` — per-repo commit walker; emits events + a `CommitSet` artifact per repo per day.
- `crates/connectors/connector-local-git/src/privacy.rs` — the private-repo redaction rule (see invariants). Kept in its own module so the rule is reviewable in isolation.
- `crates/connectors/connector-local-git/tests/fixtures/*.tar` — small, committed-in-repo fixture tarballs (≤20 KB each). Each tarball unpacks to a pre-seeded git repo with known commits, branches, authors, and one "private" marker.
- `crates/connectors/connector-local-git/tests/fixtures.rs` — helper that extracts a fixture tarball into a `tempdir()` and returns its path.
- `crates/connectors/connector-local-git/tests/sync.rs` — integration tests that drive the full `LocalGitConnector` against the fixtures.

**Files to modify:**

- `Cargo.toml` (workspace root) — add `crates/connectors/connector-local-git` to `members` and to `[workspace.dependencies]` alongside the existing crate path deps.
- `crates/dayseam-core/src/error_codes.rs` — add `LOCAL_GIT_REPO_CORRUPT`, `LOCAL_GIT_REPO_NOT_FOUND`, `LOCAL_GIT_NO_SIGNATURE`, `LOCAL_GIT_TOO_MANY_ROOTS` (the previously-registered `LOCAL_GIT_REPO_LOCKED` / `LOCAL_GIT_REPO_UNREADABLE` stay). New codes are **additive**; renaming any existing one is a breaking change (MNT-04 pattern).
- `deny.toml` — verify `git2` license (Apache-2.0 / MIT) is already on the allow list; add `libssh2-sys`, `openssl-sys` etc. if `bundled` still pulls them. If any surface is novel, add a documented rationale entry per MNT-04.
- `CHANGELOG.md` — one line under Unreleased.

### Invariants proven by tests

1. **Discovery is bounded and deterministic.** Given a scan root with N nested repos and N bogus sub-directories, the walker yields the same ordered list of `LocalRepo` rows on every run. `max_depth` and `max_roots` are respected; exceeding `max_roots` emits a `LogEvent::Warn` with `code = LOCAL_GIT_TOO_MANY_ROOTS` and truncates, never panics. Test: `discovery_is_deterministic_and_bounded`.
2. **`SyncRequest::Day` returns only the events whose author or committer email resolves through `ctx.source_identities` to `ctx.person`.** Everything else is excluded from `events` and counted in `stats.filtered_by_identity`. Test: `sync_filters_by_identity`.
3. **One `Artifact::CommitSet` per `(source_id, repo_path, date)` with at least one event.** `artifacts` length equals the number of `(repo, day)` buckets that produced at least one matching commit. Test: `sync_emits_one_commit_set_per_repo_per_day`.
4. **Timezone correctness.** An event whose commit timestamp lands at 01:00 UTC on day D+1 but 20:00 local on day D must be attributed to day D when the user's local timezone is UTC-05. Test: `sync_uses_local_tz_for_day_window`.
5. **Private-repo redaction.** A repo whose `local_repos.is_private = 1` (or that contains a sentinel `.dayseam/private` marker the user added) yields events with `privacy = Privacy::Redacted`, empty `title`/`body`, and a stubbed `RawRef` that does not round-trip the commit message. The commit SHA and author are still captured because the user opted in to "I did work here" signal without "here's what the commit said". Test: `sync_redacts_private_repos`.
6. **Cancellation is prompt.** `ctx.cancel.cancel()` fired mid-walk causes `sync` to return `Err(DayseamError::Cancelled)` within a bounded wall-clock window (allow ≤2 seconds on CI to absorb GC pauses). Test: `sync_aborts_promptly_on_cancel`.
7. **`SyncRequest::Range` / `SyncRequest::Since` return `Err(DayseamError::Unsupported)` with `CONNECTOR_UNSUPPORTED_SYNC_REQUEST`.** The orchestrator can then fall back (or, for v0.1, surface the error to the UI — Phase 2 never issues these requests). Test: `sync_unsupported_variants_return_unsupported`.

- [ ] **2.2 Freeze the fixture shape.** Each tarball unpacks to a git repo with 3–6 commits across two authors, one of which matches `fixture_self_person()`, the other does not. ## Task 3: `dayseam-report` — Dev EOD template, rollup, render, golden snapshots

**Why next:** Tasks 1 and 2 produce `Artifact`s and `ActivityEvent`s; Task 3 turns them into a `ReportDraft`. The engine is pure (`ARCHITECTURE.md` §10): `(events, artifacts, template_id, template_version, person, source_identities) -> ReportDraft`. No IO, no clocks, no randomness. That makes golden snapshots trivial — same inputs always produce the same output, and `insta` keeps drift honest.

**Files to create / modify:**

- `crates/dayseam-report/Cargo.toml` — promote from skeleton. Depends on `dayseam-core`, `handlebars` 6, `serde`, `serde_json`, `chrono`, `thiserror`. `dev-dependencies` add `insta = { features = ["yaml"] }` and `proptest`.
- `crates/dayseam-report/src/lib.rs` — public surface: `pub fn render(input: ReportInput) -> Result<ReportDraft, ReportError>`.
- `crates/dayseam-report/src/rollup.rs` — stage 1: group `ActivityEvent`s by `Artifact` (falling back to a synthetic `CommitSet` for events without one); sort events deterministically by `occurred_at, external_id`. No IO, no state.
- `crates/dayseam-report/src/templates/mod.rs` — registry `{ template_id: &str -> TemplateDef }`, backed by `handlebars::Handlebars`.
- `crates/dayseam-report/src/templates/dev_eod.rs` — ships the Dev EOD template (`template_id = "dayseam.dev_eod"`, `template_version = "2026-04-18"`). One file, two Handlebars partials (`partials/section_commits.hbs`, `partials/evidence_link.hbs`).
- `crates/dayseam-report/src/render.rs` — stage 3: walks the rolled-up input, runs the template, wraps each rendered bullet with a stable `bullet_id` and an `Evidence` edge back to its source `ActivityEvent` ids.
- `crates/dayseam-report/src/clock.rs` — re-export the `Clock` trait from `connectors-sdk` with an assertion that the engine never calls `now()` itself; kept as a lint guard for Task 8 rather than a real dependency (see invariant #1).
- `crates/dayseam-report/tests/golden.rs` + `crates/dayseam-report/tests/snapshots/` — `insta` golden snapshots covering every fixture from Task 2.
- `CHANGELOG.md` — one line under Unreleased.

### Invariants proven by tests

1. **Purity.** Calling `render` with identical inputs produces byte-identical `ReportDraft`s — including `bullet_id`s, which are deterministic hashes of `(template_id, section_id, artifact_id, sorted event_ids)`. Test: `render_is_deterministic` (run 100 times with a fixed seed, all outputs equal).
2. **Verbose mode is additive.** `ReportDraft::verbose_mode = true` only adds bullets; it never changes an existing bullet's `bullet_id` or `evidence`. Test: `verbose_mode_only_adds_bullets`.
3. **Every bullet traces back to at least one event.** No bullet in any rendered draft has `evidence.event_ids.is_empty()`. Test: `every_bullet_has_evidence`, property-based over generated event sets.
4. **Redacted events render as "(private work)"-style bullets** with the repo link intact but no title / body in the output. Test: `redacted_events_render_without_message`.
5. **Empty day path is explicit.** An input with zero events still produces a valid `ReportDraft` with one section whose body is the "No tracked activity for `<date>`" empty-state string documented in `docs/design/2026-04-17-v0.1-design.md` §UX. Test: `empty_day_renders_empty_state`.
6. **Golden snapshots for every fixture from Task 2.** Task 2's tarballs feed directly into Task 3's tests; drift in either half fails the snapshot. Test: `golden/dev_eod_<fixture_name>.md.snap`.

### Steps

- [ ] **3.1 Stub the public surface and write a one-event smoke test.** Fail first, implement just enough to pass. This anchors the snake of red tests that follow.

### Invariants proven by tests

1. **Atomic write.** A crash simulated mid-write (SIGKILL before `persist()`) leaves zero bytes of the target file changed; the temp file is cleaned up on next sink init. Test: `crash_midwrite_leaves_target_intact`.
2. **Marker-block preservation.** Writing a draft for date D into an existing file that already contains marker blocks for D-1 and D-3 appends a new D block; existing user prose between blocks is preserved verbatim (byte-for-byte). Test: `marker_block_preserves_surrounding_user_prose`.
3. **Marker-block replacement.** Re-writing the same date D replaces only the existing D block; adjacent blocks are untouched. Test: `rewriting_same_date_replaces_only_its_block`.
4. **Marker-block shape.** Each block is delimited by `<!-- dayseam:begin date="YYYY-MM-DD" run_id="…" template="dayseam.dev_eod" version="…" -->` and a matching `<!-- dayseam:end -->`. 5. **Two destinations, one consistent result.** `MarkdownFileSink` configured with `dest_dirs = [A, B]` writes the same bytes to both. Test: `second_destination_failure_does_not_rollback_first`.
6. **Obsidian-friendly filenames.** The default filename is `Dayseam <YYYY-MM-DD>.md` (space, capital-D), which Obsidian's daily-note plugin tolerates without configuration. Test: `default_filename_matches_obsidian_convention`.
7. **Concurrent-write refusal.** Two `write` calls targeting the same file overlap: the second observes a lock file (`<file>.dayseam.lock`, POSIX `flock` / Windows FS-level exclusive) and returns `DayseamError::Io { code: SINK_FS_CONCURRENT_WRITE, .. }`. 8. **Capability declaration matches Phase-1 expectation.** `capabilities()` returns `SinkCapabilities { local_only: true, remote_write: false, interactive_only: false, safe_for_unattended: true }`. Test: `capabilities_are_local_only_and_unattended_safe`.

### Invariants proven by tests

1. **Fan-out.** `generate_report(date, sources)` runs each source's `sync` concurrently with a bounded parallelism of `min(sources.len(), 4)`. Each source's output streams progress / logs onto a shared `RunStreams` whose `run_id` is the `SyncRun::id`. Test: `generate_fans_out_and_streams_with_one_run_id`.
2. **Supersede on retry.** If a generate is started while another is already running for the **same `(person_id, date, template_id)` tuple**, the older run's `status` transitions to `Cancelled` with `cancel_reason = SupersededBy(<new_run_id>)`, its per-run streams are drained and dropped, and any late-arriving results are discarded at persist time. Test: `second_generate_supersedes_first_and_late_writes_are_dropped`.
3. **Cancellation propagates.** `orchestrator.cancel(run_id)` fires `ctx.cancel.cancel()` for every connector in the run, waits for them to observe it, marks the `SyncRun` `Cancelled`, and emits a terminal `ProgressPhase::Cancelled` on the per-run stream. Test: `cancel_fires_tokens_and_emits_terminal_progress`.
4. **Partial failure surfaces, does not abort.** One source returning `Err(DayseamError::…)` does not abort the other sources. The failing source's `per_source_state_json` carries the error code and message; the draft is still rendered from the healthy sources' events; the UI toast fires from `AppBus`. Test: `one_source_failure_does_not_abort_the_run`.
5. **`SyncRun` row is durable.** Every transition (`Running -> Completed`, `-> Cancelled`, `-> Failed`) is persisted *before* the next transition is possible. If the process crashes mid-run, the row is left `Running` with `finished_at IS NULL`; the next startup sweeps those rows into `Failed` with `code = INTERNAL_PROCESS_RESTARTED`. Test: `crash_during_run_recovers_to_failed_on_next_startup`.
6. **Retention sweep is safe.** `retention::sweep` deletes `raw_payloads` and `log_entries` older than the configured cutoff (default 30 days, stored in `settings` under key `retention.days`), runs in one transaction per table, and is a no-op on an already-swept DB. Test: `retention_sweep_is_idempotent_and_never_deletes_recent_rows`.
7. **`save_report` is atomic from the orchestrator's point of view.** A failed sink write does not mutate `report_drafts.sections_json`; only `WriteReceipt` rows are appended (see Task 4 invariant #5). Test: `save_on_failing_sink_leaves_draft_untouched`.

### Steps

> This task follows the shape documented in [Phase 1 Task 10](./2026-04-17-v0.1-phase-1-foundations.md#task-10-phase-1-hardening--cross-cutting-review). Only the phase-specific lenses differ.

**Intent, not scope.** This is the one PR where multiple small findings may land together; the single commit that closes it enumerates each finding and its resolution in the commit body.

**Files to create / modify:**

- Create: `docs/review/phase-2-review.md` — the written review artifact, shaped like `phase-1-review.md`.
- Modify: any crate where findings trigger a fix.
- Modify: `CHANGELOG.md` — "Phase 2 hardening pass" entry.

### Phase-specific lenses (additions to the Phase 1 lens list)

- **Data integrity — marker-block preservation.** Fuzz `sink-markdown-file::markers` against hand-crafted and `arbitrary`-generated markdown inputs. Any case where `parse(splice(parse(x), block)) != splice(parse(x), block)` is a finding.
- **Identity linking regression.** For each fixture from Task 2, walk the self-`Person` bootstrap path on a fresh DB and on an upgraded-from-v1 DB; assert the resolved identity set is identical.
- **libgit2 supply chain.** `cargo deny` audit of `git2`, `libgit2-sys`, `libssh2-sys`, `openssl-sys` trees; any new advisory or licence gets a rationale row alongside the Phase 1 ignores.
- **`SyncRun` lifecycle coverage.** Every state transition in Task 5's `SyncRun` state machine has a test; any missing arrow is a finding.
- **PERF-08 closure.** The Phase-1 deferred item is formally resolved (or re-deferred with fresh justification) in this review.
- **Marker-block security.** A marker block sourced from user-editable markdown must not allow JavaScript / HTML escape into a rendered preview (we render markdown in the preview; confirm the sanitiser does not let a maliciously crafted pre-existing marker block inject script).
- **IPC parity.** Every `#[tauri::command]` appears in `capabilities/default.json` and in `Commands`; new commands added during Phase 2 do not regress the ARC-01 guard from Phase 1.

## Exit criteria for Phase 2

Phase 2 is done when **all** of the following are true on `master`:

1. All eight PRs above are merged with `semver:none`.
2. `VERSION` is still `0.0.0`.
3. `cargo fmt --check`, `cargo clippy -D warnings`, `cargo test --workspace --all-features`, `pnpm -r lint`, `pnpm -r typecheck`, `pnpm -r test`, `cargo deny check`, and `cargo audit` all green.
4. `pnpm tauri:dev` opens the Mac window to either the first-run empty state (on a fresh `state.db`) or the real workspace (on an upgraded one), and the full Generate → Save loop against a real local git scan root produces a marker-block markdown file the author uses for their own EOD.
5. `docs/review/phase-2-review.md` is merged, with every finding resolved or explicitly deferred with a linked issue.
6. PERF-08 from [`phase-1-review.md`](../review/phase-1-review.md) is formally closed (resolved or re-deferred with evidence).
7. Phase 3's plan document can be drafted against a reviewed, known-good Phase-2 scaffold.
