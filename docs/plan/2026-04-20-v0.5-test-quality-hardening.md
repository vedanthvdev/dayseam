# v0.5 In-app auto-updater + test-quality floor raise + v0.4 carry-over hardening — Implementation Plan

> **For agentic workers:** this plan is executed PR-by-PR. Each task below is one PR on its own branch. Follow the one-commit-per-branch rule: the first change on a branch is a real `git commit`; every subsequent change is `git commit --amend --no-edit` followed by `git push --force-with-lease`. When a step below says "Commit," obey that rule.

**Goal:** at the end of v0.5, Dayseam **checks GitHub Releases for a newer version on app startup and can download + verify + install the update from within the app without the user having to re-download the DMG**. Alongside this one user-visible feature, every bug the v0.4 capstone review and the v0.4 dogfood sweep surfaced has a regression test that would fail if its fix were reverted, and every *recurring* bug pattern has been promoted to a **class-level meta-test or lint** so the next instance cannot reach a reviewer. Behind the hardening lens, v0.5 consumes the three Medium/Low items the v0.4 capstone review explicitly deferred (TST-v0.4-01, TST-v0.4-02, TST-v0.4-04), audits cross-source enrichment for GitLab/GitHub parity, and generalises the `F-10` supervised-spawn pattern into a reusable helper with a CI gate against bare `async_runtime::spawn`.

**Release cadence:** v0.5 ships in **two releases** (a break from the v0.1–v0.4 one-release-per-phase pattern). DAY-108 — the in-app updater — cuts **`v0.6.0`** at its own merge under `semver:minor`, putting the updater in users' hands the moment it is ready rather than at the end of the phase. DAY-109..DAY-114 each carry `semver:none` and batch into the capstone. The capstone (DAY-115) cuts **`v0.6.1`** under `semver:patch`, packaging the hardening batch.

**Why release-per-feature for DAY-108 specifically:** the updater is the unique v0.5 PR whose value-to-user is unlocked *by merging it*, not by the phase-end release — every intervening day between DAY-108's merge and the capstone's merge is a day the user is still manually re-downloading DMGs even though the fix is sitting on `master`. Splitting DAY-108's release from the capstone's collapses that gap to zero. The pattern for hardening PRs (T3–T8) remains capstone-batched because the hardening's value-to-user manifests at the capstone's aggregate release, not at each individual merge. This is a deliberate one-off exception to the v0.3 / v0.4 capstone-only convention, justified by the feature-shape asymmetry; it is not a new default. Future phases should default back to capstone-batched releases unless a similar shape argument applies.

**Why `v0.6.0` and `v0.6.1`, not `v0.5.3` and `v0.5.4`:** the updater is a user-visible feature — the kind the design doc's semver rule explicitly maps to `semver:minor`. An honest version bump says so. The hardening capstone that follows is a patch release because no further user-visible feature lands in DAY-109..DAY-114. The "v0.5 phase name / v0.6.x release numbers" asymmetry mirrors v0.4 (where DAY-104 landed as a post-capstone `semver:minor` bump to `v0.5.0`) — phase names are about planning cadence, release numbers are about the changelog audience, and the two don't have to agree.

**Why this is one phase, not two:** the updater is one substantial feature (signing keys, release-workflow rework, plugin wiring, UI, E2E); the rest is hardening. They don't share code — a reviewer examining the updater PR should not have to understand the `SerdeDefaultAudit` extension, and vice versa — but they DO share a dogfood arc (the hardening work in T3–T8 gets auto-delivered to the user via the updater that T2 just shipped, so by the capstone the updater has been field-exercised against the phase's own internal releases). Keeping them under one phase name + one plan doc is the simplest discipline that still captures this arc.

**Testing-first lens (new for v0.5):** every hardening task below carries an explicit `**Would-have-caught:**` subsection naming the concrete v0.4-era bug (or bug class) whose fix the task's tests would fail-reproduce if reverted. The updater task (T2) carries the same subsection shape — naming the release-engineering failure modes its tests cover rather than a historical bug, because the updater is new code rather than a fix for a shipped bug. Tasks that don't have a direct would-have-caught mapping (T1 plan draft; T9 capstone) say so explicitly. This isn't bookkeeping — it's the acceptance bar. If a PR reaches review without a would-have-caught claim backed by a failing test on the unreverted fix, the PR is not done.

**Architecture:** v0.5 is mostly additive. No schema migration. No IPC signature change for existing commands. No SDK trait change. No renamed variant. Three structurally non-trivial changes:

1. **T2's Tauri updater plugin wiring.** A new Tauri plugin (`tauri-plugin-updater`) is registered in `lib.rs`. A new `updater` section in `tauri.conf.json` carries the Ed25519 public key and the `latest.json` endpoint URL. A new `updater.json` capability in `apps/desktop/src-tauri/capabilities/` grants `updater:allow-check` + `updater:allow-download-and-install` scopes. A new React component surfaces the update-available UI. The release workflow learns one new step: sign the DMG with `tauri signer sign`, upload the `.sig` file, and regenerate `latest.json` with platform-specific download URLs + signatures.

2. **T7's `supervised_spawn` helper**, which wraps the existing `tauri::async_runtime::spawn` call sites inside a thin awaiting outer task; every call-site's external behaviour (fire-and-forget, returns a `JoinHandle`) is preserved byte-for-byte.

3. **T5's injectable `HttpClient` on `AppState`**, which is additive at the struct level (a new `pub http: Arc<dyn HttpClient>` field with a default-construct path that matches today's behaviour) but lets reconnect-account-rebind and other IPC-level flows be tested without a live network.

**macOS ad-hoc-signature caveat (T2):** Dayseam currently ships ad-hoc signed (no Apple Developer ID — tracked at [#59](https://github.com/vedanthvdev/dayseam/issues/59) / [#108](https://github.com/vedanthvdev/dayseam/issues/108)). Ad-hoc signatures are per-build by default, so an auto-updated binary has a different signature from the installed binary, which can re-trigger Gatekeeper's "this app can't be opened" dialog on first launch after update. The user explicitly accepted this caveat in the v0.5 planning dialog — the occasional re-prompt is tolerable for v0.5's single-user blast radius. **Stabilizing the ad-hoc identity across builds is explicitly deferred** (would reduce the re-prompt odds but not eliminate them); the real fix is #59, which is a separate future phase. T2 documents the caveat in `docs/updater/2026-04-20-macos-unsigned-updater-caveat.md` so a future contributor (or user) can find it.

**Tech stack additions on top of v0.5.2:** two new deps for T2, zero new deps for the hardening tasks.

- `tauri-plugin-updater = "2"` — Rust plugin. Pinned at Tauri's `2.x` major since the workspace already tracks `tauri = "2"`.
- `@tauri-apps/plugin-updater` — frontend binding. Version matches the Rust plugin's major.
- No other deps. `reqwest` + `wiremock` + `sqlx` + `serde` + `tokio` + `ts-rs` + `dayseam-macros` + `proptest` (if T8 pulls it in) carry over.

## Task 2 (DAY-108): In-app auto-updater via GitHub Releases

**Why this order:** feature work first, hardening after. Users (currently: one) have been eating the "download a fresh DMG and re-install" cycle for every patch release since v0.1.0 — six releases in the last four days. Shipping the updater early means (a) you can dogfood the updater itself through the rest of v0.5's capstone release (v0.6.0 → v0.6.1 auto-upgrade at DAY-115 merge is the first end-to-end field exercise), and (b) DAY-108 merging on `semver:minor` cuts `v0.6.0` immediately, putting the updater in your hands the moment it's ready rather than at the end of the phase. The test-quality tasks (T3-T8) produce no user-visible behaviour and batch into the capstone's `v0.6.1` patch release.

**Current behaviour (v0.5.2):** the app has no update-awareness. The release workflow cuts a DMG at `github.com/vedanthvdev/dayseam/releases/latest` but nothing in the installed app knows when a newer DMG is published. The user discovers new versions by watching GitHub or by the CHANGELOG message in a merged PR.

**Target behaviour:** on app startup (after the main window is up, not blocking it — matching the deferred-orphan-audit pattern F-10 established), the app issues a single `GET` to a GitHub-hosted `latest.json` manifest. If the manifest's `version` is newer than the installed `CFBundleShortVersionString`, a non-blocking banner renders at the top of the shell: **"Dayseam v0.6.0 is available · [Update now] · [Remind me later] · [Skip this version]"**. Clicking "Update now" downloads the signed DMG from GitHub, verifies the Ed25519 signature against the public key bundled in the app, installs atomically, and restarts the app. Deferred updates re-prompt on next app launch; skipped versions are suppressed until the next minor version is available.

**Shape of the updater contract (Tauri v2):**

- **Public key** bundled in `tauri.conf.json` under `plugins.updater.pubkey`. Generated once via `tauri signer generate -w <passphrase>`; the public half is committed.
- **Private key** stored as GitHub Actions secret `TAURI_SIGNING_PRIVATE_KEY` + `TAURI_SIGNING_PRIVATE_KEY_PASSWORD`. Used by the release workflow to sign the DMG.
- **Manifest** at `https://github.com/vedanthvdev/dayseam/releases/latest/download/latest.json`, generated at release time. Shape:

```json
{
  "version": "0.6.0",
  "notes": "See CHANGELOG.md for details.",
  "pub_date": "2026-04-28T12:00:00Z",
  "platforms": {
    "darwin-aarch64": {
      "signature": "<ed25519 sig>",
      "url": "https://github.com/vedanthvdev/dayseam/releases/download/v0.6.0/Dayseam-v0.6.0-aarch64.dmg"
    },
    "darwin-x86_64": {
      "signature": "<ed25519 sig>",
      "url": "https://github.com/vedanthvdev/dayseam/releases/download/v0.6.0/Dayseam-v0.6.0-x86_64.dmg"
    }
  }
}
```

- **Endpoint** in `tauri.conf.json` set to `https://github.com/vedanthvdev/dayseam/releases/latest/download/latest.json`. GitHub rewrites `/releases/latest/download/<asset>` to the newest release's asset, so the URL is stable across versions.
- **Capability** grants `updater:allow-check` + `updater:allow-download-and-install`. Add to a new `apps/desktop/src-tauri/capabilities/updater.json` file so the grant is isolated from the existing `default.json` (mirrors the `dialog.json` split pattern the project already uses).
- **Plugin registration** in `lib.rs`: `.plugin(tauri_plugin_updater::Builder::new().build())`.

## Task 3 (DAY-109): Extend `SerdeDefaultAudit` to deep persisted types

**Why this order:** mechanical and high-leverage. The deeper types — `ArtifactPayload` (enum), `ActivityEvent` (struct), `SyncRun` (struct), `PerSourceState` (struct), `LogEntry` (struct) — are exactly where the next `#[serde(default)]` is most likely to land (most activity/inbox surface area happens in these types). **Current behaviour (v0.5.2):** these five types participate in sqlite round-trips (via `serde_json::to_string` at write time, `serde_json::from_str` at read time) but do *not* carry the audit derive. **Helper signature:**

```rust
pub fn supervised_spawn<F>(context: &'static str, future: F) -> tokio::task::JoinHandle<()>
where
    F: Future<Output = ()> + Send + 'static,
{
    tokio::spawn(async move {
        let inner = tokio::spawn(future);
        match inner.await {
            Ok(()) => tracing::debug!(context, "background task completed"),
            Err(e) if e.is_panic() => {
                tracing::error!(context, error = ?e, "background task panicked");
            }
            Err(e) => {
                tracing::warn!(context, error = ?e, "background task cancelled");
            }
        }
    })
}
```

## Semver discipline

v0.5 departs from the v0.3 / v0.4 capstone-only pattern for one deliberate reason — DAY-108's feature value is unlocked at its merge, not at the end of the phase (see header). The rule for this phase:

- **DAY-107 (plan draft) carries `semver:none`** per the v0.4 review §6 clarification — plan-draft PRs are part of the phase arc and roll up into the following release's notes.
- **DAY-108 (updater) carries `semver:minor`.** Merging it WILL trigger the release workflow, which runs `scripts/release/bump-version.sh minor` and publishes **`v0.6.0`**. DAY-108 is the only intermediate PR in v0.5 carrying a release label.
- **DAY-109..DAY-114 (hardening) each carry `semver:none`.** Merging any of them MUST NOT trigger the release workflow. Their user-visible value lands at the capstone's aggregate release.
- **DAY-115 (capstone) carries `semver:patch`.** Merging it WILL trigger the release workflow, which runs `scripts/release/bump-version.sh patch` and publishes **`v0.6.1`**.
- **Why `semver:patch` at the capstone, not `semver:minor`:** DAY-109..DAY-114 land no user-visible feature. The capstone release is a CHANGELOG "internal improvements + test coverage" bucket, which semver maps to patch. If an unexpected user-visible feature slips into any of DAY-109..DAY-114, that PR's label is promoted to `semver:minor` and the capstone's label correspondingly re-evaluates (the in-phase author makes the call; the capstone PR documents the reasoning).
- **Future-phase default:** the release-per-feature pattern here is a one-off exception. v0.6 (whatever it turns out to be) defaults back to capstone-only-releases unless it has a similar feature-shape asymmetry argument.

## Graphify re-score — timing

The v0.4 review §4 recorded that none of graphify's five re-evaluation triggers had fired, and deferred to v0.5's capstone. v0.5's capstone (DAY-115 step 2) runs the re-score explicitly. If v0.5 execution surfaces any of the five triggers firing (e.g. T7's supervised-spawn sweep turns out to need a cross-crate helper that makes the ~20-crate threshold concrete, or T5's injectable `HttpClient` spans more than 5 crates, or DAY-108's updater work pulls in enough new surface area that an AI agent without a graph starts struggling), the trigger gets recorded in the capstone review §4.X and the adoption task lands inline in DAY-115.

If no trigger fires, the re-score records the negative outcome (per the [`docs/decisions/2026-04-20-graphify-deferred.md`](../decisions/2026-04-20-graphify-deferred.md) audit-trail convention) and defers to v0.7+ (v0.6 is already spent on v0.5's two releases). The tracking issue [#61](https://github.com/vedanthvdev/dayseam/issues/61) is updated with a pointer to v0.5's review §4.X subsection as the latest negative re-evaluation.
