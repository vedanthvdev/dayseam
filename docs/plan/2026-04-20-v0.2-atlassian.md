# v0.2 Atlassian: Jira + Confluence — Implementation Plan

> **For agentic workers:** this plan is executed PR-by-PR. Each task below is one PR on its own branch. Follow the one-commit-per-branch rule: the first change on a branch is a real `git commit`; every subsequent change is `git commit --amend --no-edit` followed by `git push --force-with-lease`. When a step below says "Commit," obey that rule.

**Goal:** at the end of v0.2, a Dayseam user can connect their Atlassian Cloud workspace with a single email + API-token credential, see both Jira activity (issue transitions, comments, assignments) and Confluence activity (page authored/edited, comments) fused into the same daily EOD report as their GitLab and local-git events — with cross-source enrichment that links a GitLab MR titled `CAR-5117: Fix …` to the Jira issue `CAR-5117`, and annotates a Jira status transition with the MR that triggered it. One credential, two sibling connector crates, three event domains (issues, pages, comments), shared identity and auth plumbing.

**Why v0.2 and not v0.1.x:** this is an **additive minor release** — new connectors, new event kinds, new UI affordances, new public IPC commands. It carries `semver:minor` for the first PR that lands under the new SDK surface (Task 3 below) and the release cuts as `v0.2.0` on the task-12 capstone. No breaking IPC, no schema destructive migration, no v0.1.x user has to do anything when upgrading.

**Architecture:** v0.1.0 landed three connectors (`connector-local-git`, `connector-gitlab`, and by extension the orchestrator/report pipeline). v0.2 adds two more (`connector-jira`, `connector-confluence`) that share so much plumbing — auth header, identity seed, ADF body normalisation, cloudId discovery, cursor-pagination helper, 401/403 → `DayseamError::Auth` mapping — that they get a shared common crate (`connector-atlassian-common`) below them. The orchestrator registers them as two distinct connector kinds so a user can enable Jira without Confluence (and vice versa), and the `sources` table supports two source rows sharing one `secret_id` (FK-guard on delete — no schema migration, no destructive change).

v0.2 also delivers the first **cross-source enrichment pipeline** the design has anticipated since day one. A GitLab MR titled `CAR-5117: Fix review findings` gains a `target` `EntityRef` pointing at Jira issue `CAR-5117` with zero Jira API calls; a Jira status transition on `CAR-5117` looks at its `RemoteWorkItemLink` changelog items and annotates itself with the MR that triggered the transition. Both enrichments are additive on top of the existing `rolled_into_mr` rollup — no existing event shape changes.

**Tech stack additions on top of v0.1.0:** no new top-level crate families. `reqwest` 0.12 + `wiremock` 0.6 + `serde` + `tokio` all carry over. One new internal dependency: a ~80-LoC ADF-to-plain-text walker that lives in `connector-atlassian-common::adf`, built on `serde_json::Value` (no crates.io crate needed — Atlassian publishes the ADF schema but there is no maintained Rust binding at the time of writing). Base64 encoding for the Basic-auth header uses the `base64` crate (already in the dep tree via `dayseam-secrets`).

**Atlassian transport specifics:**

- **Cloud host:** `https://<workspace>.atlassian.net`
- **Jira base path:** `/rest/api/3/`
- **Confluence base paths:** `/wiki/api/v2/` (v2 — preferred) and `/wiki/rest/api/` (v1 — CQL search + older content APIs not yet in v2)
- **Auth header:** `Authorization: Basic <base64(email:api_token)>`
- **Body format flag:** Jira comments are always ADF; Confluence page/comment bodies default to storage format (XHTML-like), so every Confluence request that needs a body passes `body-format=atlas_doc_format` to force ADF. Single code path for body normalisation falls out of that decision.
- **Pagination:** Jira's `/search/jql` returns `{isLast, nextPageToken}`; Confluence v2 returns `{_links: {next: "/wiki/api/v2/...?cursor=eyJ..."}}`. The atlassian-common `pagination` helper exposes both behind one `Cursor<Page>` trait so walkers consume a single loop shape.
- **Rate limits:** per-user, published as `X-RateLimit-*` headers + 429 with `Retry-After`. The existing `connectors-sdk::HttpClient` retry logic from Phase 1 handles 429 verbatim — no new SDK work.

## Plan / design alignment

v0.2 consumes the contracts v0.1.0 shipped verbatim. The following pieces are referenced by tasks below and **must not drift**:

- `connectors-sdk::{SourceConnector, SyncRequest, SyncResult, SyncStats, Checkpoint, ConnCtx, HttpClient, AuthStrategy, RawStore, Clock}` — v0.2 adds two trait implementors (`JiraConnector`, `ConfluenceConnector`) and one new `AuthStrategy` variant (`BasicAuth`); neither is a trait-shape change. Both connectors handle `SyncRequest::Day` and return `Err(DayseamError::Unsupported)` with `CONNECTOR_UNSUPPORTED_SYNC_REQUEST` for `Range`/`Since`, exactly the way `LocalGitConnector` and `GitlabConnector` do today.
- `dayseam-core::types::{source, identity, activity, artifact, run}` — v0.2 adds to each enum **additively**: `SourceKind::Jira`, `SourceKind::Confluence`, seven new `ActivityKind` variants (four Jira + three Confluence), `SourceIdentityKind::AtlassianAccountId`, `ArtifactKind::JiraIssue`, `ArtifactKind::ConfluencePage`. The `ts-rs` drift guard in `crates/dayseam-core/tests/ts_types_generated.rs` surfaces every new variant in the generated TypeScript union types — no manual TS work required, but Task 10 consumes the surfaced types.
- `dayseam-secrets` — v0.1's `SecretString` + Keychain wrapper. v0.2's API token lives at `dayseam.atlassian.<source_id>` keyed by **the source row's id**, not by `jira.<...>` or `confluence.<...>` — because two sources (one Jira, one Confluence) for the same workspace share **one** secret row. The two sources reference the same `secret_id` in the `sources` table; a FK guard on delete (Task 9) refuses to drop the secret while any source still references it.
- `dayseam-events::{RunStreams, AppBus}` — v0.2 runs use `RunStreams` exactly as shipped. No ownership changes.
- `dayseam-orchestrator` — v0.1.0's `generate_report` lifecycle, fan-out, and `SyncRun` state machine stay frozen at the trait/IPC level. v0.2 adds two behaviours internal to the report layer: `group_key_from_event` (Task 6) so Jira/Confluence events roll up by `project`/`space` the way GitLab/local-git events roll up by `repo`; and the ticket-key enrichment pipeline (Task 6) that extracts `[A-Z]+-\d+:` prefixes from event titles and attaches a `target` entity. Both are pure additions; neither changes a public type or existing rollup's output for GitLab/local-git-only inputs.
- `dayseam-db` — v0.1.0's migrations `0001…0004` stay frozen. v0.2 adds exactly one migration (`0005_atlassian_sources.sql`) that registers the two new `SourceKind` values in the `source_kinds` lookup table and adds an index on `source_identities(kind, external_actor_id)` that Jira/Confluence walkers need for the self-filter hot path. No destructive changes; any v0.1.0 laptop upgrading to v0.2 keeps its `state.db`.
- The IPC surface — every v0.2 command goes through `apps/desktop/src-tauri/src/ipc/commands.rs`, is added to `apps/desktop/src-tauri/capabilities/default.json`, and has a typed entry in `@dayseam/ipc-types`. The Phase 1 capability / `Commands` parity guard and the dev-commands feature gate continue to hold.

Four pieces flagged in earlier reviews / spikes that v0.2 picks up directly:

## Task 2: `connectors-sdk` — `BasicAuth` strategy

**Why second:** Task 3 (atlassian-common) needs a `BasicAuth` to hang its cloudId-discovery HTTP call on. Every Task 4/7 `SourceConnector` impl consumes it to hydrate its auth from the `sources.secret_id` column. It has no dependencies beyond the existing `AuthStrategy` trait.

**Correction applied during DAY-74 implementation:** the original plan proposed routing 401/403 → `DayseamError::Auth` inside `HttpClient::classify_error`. This was overridden by the Phase-3 CORR-01 invariant, which lands in `v0.1.0`: **`HttpClient` must not auto-classify non-retriable statuses** — it returns the raw `reqwest::Response` and each connector's local `errors::map_status` owns the mapping (see `connector-gitlab::errors`). The `atlassian.auth.*` classifier therefore lives in DAY-75 (`connector-atlassian-common::errors`) rather than in `connectors-sdk`. DAY-74 still proves that 401/403 responses survive `HttpClient::send` as raw responses (the CORR-01 invariant for Atlassian), so a regression in either direction is caught.

**Shared-PAT vs separate-PAT design:** the DAY-73 spike assumed all users would bind Jira + Confluence to a single API token (one keychain row shared between two source rows, ref-counted on delete in DAY-81). Real tenants don't always split that way — some companies run separate service accounts per product, some run Jira Cloud + Confluence Cloud under different tenants. `BasicAuth::atlassian` was therefore designed to be **agnostic to the sharing decision**: each instance carries exactly one `(email, keychain_service, keychain_account)` triple. Two sources that pass the same triple collapse into a shared-PAT row; two sources that pass different triples stay independent. The DAY-81 refcount is still the correct guard for the shared case and degenerates cleanly (refcount = 1 per row) in the separate case. See `src/auth.rs::AuthDescriptor::Basic` docs and the `basic_auth_{same,different}_keychain_handle_*` + `basic_auth_{shared,separate}_handle*_live` tests.

**Files to modify:**

- `crates/connectors/connectors-sdk/src/auth.rs` — add `struct BasicAuth` + `AuthDescriptor::Basic { email, keychain_service, keychain_account }` next to the existing `PatAuth` / `AuthDescriptor::Pat`. `BasicAuth::atlassian(email, api_token, keychain_service, keychain_account)` base64-encodes `<email>:<api_token>` at construction time and wraps the full `Basic <…>` header value in a `SecretString`. `email` is plain `String` (not secret — it's visible in JQL results and the Jira UI); the `api_token` argument is consumed into the `SecretString` and never held in a named binding after the constructor returns.
- `crates/connectors/connectors-sdk/Cargo.toml` — add `base64 = "0.22"` (first consumer; DAY-75 will also need it for ADF `image.url` decoding).
- `crates/connectors/connectors-sdk/tests/basic_auth.rs` — new wiremock-driven integration test file.
- `CHANGELOG.md` — one line under `[Unreleased]` `### Added`.

### Invariants proven by tests

1. **Correct header.** `BasicAuth::atlassian("foo@example.com", "bar", …)` produces header `Authorization: Basic Zm9vQGV4YW1wbGUuY29tOmJhcg==` (RFC 7617 verbatim). Unit test: `basic_auth_header_is_base64_email_colon_token`. Wiremock test: `basic_auth_attaches_authorization_header_on_live_request`.
2. **Secret never leaks.** `Debug` output shows neither the plain `api_token` nor the encoded `Basic <base64>` header; `SecretString::Drop` zeroes the backing bytes. Test: `basic_auth_debug_does_not_leak_token_or_encoded_header`.
3. **Descriptor round trip.** `BasicAuth.descriptor()` round-trips `(email, keychain_service, keychain_account)` — the three fields `Source.secret_ref` persists. Test: `basic_auth_descriptor_round_trips_all_three_fields`.
4. **Shared-PAT invariant.** Two `BasicAuth` instances built with the same `(service, account)` pair produce byte-equal `Authorization` headers (for the same live token) and descriptor-equal serialisations. Unit test: `basic_auth_same_keychain_handle_produces_equal_descriptors`. Wiremock test: `basic_auth_shared_handle_produces_equal_headers_live`.
5. **Separate-PAT invariant.** Two `BasicAuth` instances built with different `(service, account)` pairs — the separate-tenant / per-product-service-account case — produce distinct descriptors and distinct on-the-wire headers; one product's PAT cannot leak into the other product's request. Unit test: `basic_auth_different_keychain_handles_stay_independent`. Wiremock test: `basic_auth_separate_handles_produce_distinct_headers_live`.
6. **CORR-01 compliance (401 / 403 not auto-classified).** A wiremocked endpoint returning 401 or 403 flows through `HttpClient::send` as a raw `reqwest::Response`; the status classification + `DayseamError::Auth` construction is DAY-75's job. Tests: `basic_auth_{401,403}_surfaces_as_raw_response_not_pre_classified_error`.
7. **UTF-8 preservation.** Unicode in email / token (some SSO-federated tenants allow non-ASCII emails) base64-encodes losslessly. Test: `basic_auth_preserves_utf8_in_email_and_token`.

### Steps

- [x] **2.1 Implement `BasicAuth` + `AuthDescriptor::Basic`.** TDD from `basic_auth_header_is_base64_email_colon_token` — the unit test is written first, the impl follows.
- [x] **2.2 Prove the CORR-01 invariant.** Add wiremock tests that 401 / 403 survive `HttpClient::send` as raw responses. Defers the `atlassian.*` mapping to DAY-75 per CORR-01.
- [x] **2.3 Prove the redaction + descriptor + sharing invariants.** `basic_auth_debug_does_not_leak_*`, `basic_auth_{same,different}_keychain_handle_*`, `basic_auth_{shared,separate}_handle_*_live`.

## Task 3: `connector-atlassian-common` — shared crate

**Why third:** Tasks 4-8 each import at least three of the five helpers that this crate exposes (auth builder, cloudId discovery, identity seed, ADF walker, pagination). Writing those helpers once, with one fixture set, prevents the CONS-addendum-04 class of cross-connector drift from happening between Jira and Confluence. This task carries `semver:minor` because once it lands on `master` the shared crate is a public surface a third-party consumer could build against.

**Files to create:**

- `crates/connectors/connector-atlassian-common/Cargo.toml` — declares `connectors-sdk`, `dayseam-core`, `dayseam-secrets`, `reqwest`, `serde`, `serde_json`, `base64`, `url`, `tracing`; dev-deps `wiremock`, `tokio`, `insta`.
- `crates/connectors/connector-atlassian-common/src/lib.rs` — re-exports `AtlassianCloud`, `discover_cloud`, `seed_atlassian_identity`, `adf_to_plain`, `CursorPaginator`, `AtlassianError`.
- `crates/connectors/connector-atlassian-common/src/cloud.rs` — `AtlassianCloud { workspace_url: Url, cloud_id: Uuid }` + `async fn discover_cloud(http: &HttpClient, auth: &BasicAuth, workspace_url: &Url) -> Result<AtlassianCloud, DayseamError>`. Hits `GET /oauth/token/accessible-resources` (wait, that's OAuth) — **actually for Basic auth** the equivalent is `GET /_edge/tenant_info` which returns `{ cloudId: "<uuid>" }`. Or: we skip the discovery call for Basic auth because the workspace URL **is** the tenant identity for a cloud instance (`company.atlassian.net` → one cloudId, stable), and instead validate the token with `GET /rest/api/3/myself` in Task 4/7's `validate_auth`.

  Decision recorded here: **cloudId is not needed for Basic auth request routing** — both Jira and Confluence endpoints accept the workspace hostname directly. The `AtlassianCloud` struct still carries a `cloud_id` field for future OAuth (v0.3+) compatibility; under Basic auth it's optional (`Option<Uuid>`) and remains `None`. `discover_cloud` becomes a thin wrapper around `GET /rest/api/3/myself` that returns `(account_id, display_name, email_opt, cloud_id: None)`.

- `crates/connectors/connector-atlassian-common/src/identity.rs` — `async fn seed_atlassian_identity(...) -> Result<SourceIdentity, DayseamError>`. Builds the `SourceIdentity { kind: AtlassianAccountId, external_actor_id: account_id, source_id: Some(...) }`. Same rollback-on-failure shape as `ensure_gitlab_self_identity` (v0.1 DAY-71).
- `crates/connectors/connector-atlassian-common/src/adf.rs` — ~80 LoC `fn adf_to_plain(body: &serde_json::Value) -> String` + `AdfError` for unknown node types. ## Task 6: `dayseam-report` — `group_key_from_event` + cross-source enrichment

## Non-goals of v0.2

- **No OAuth 2.0 (3LO) for either Atlassian product.** Basic-auth + API token is the v0.2 contract. OAuth introduces a code-verifier flow, an app-registration requirement, and a token-refresh lifecycle none of which are load-bearing for dogfood users. Tracked for v0.3.
- **No Jira Server / Data Center support.** Cloud-only. The spike §3.2 noted the shape is drop-in under `BasicAuth` once we want to add it; defer to v0.3 alongside OAuth.
- **No Jira worklog support.** The spike's empirical `worklogAuthor = currentUser()` returned zero issues; the feature can be added in v0.2.1 if a user asks. Shape is known (one extra CQL + one extra `ActivityKind::JiraWorklog`).
- **No Confluence storage-format body.** Every Confluence request forces `body-format=atlas_doc_format`. Storage-format support is a v0.3 follow-up if encountered in the wild.
- **No scheduler / autonomous runs.** Still v0.3.
- **No LLM rewrite of bullets.** Still v0.5.
- **No codesign / notarytool.** Still tracked in [#59](https://github.com/vedanthvdev/dayseam/issues/59); v0.2.0 ships with the same unsigned-DMG + Gatekeeper-bypass README as v0.1.0.
- **No graphify.** Re-deferred to v0.3 per the decision recorded at the top of this doc.

## Rollback plan

v0.2 adds two connectors, one atlassian-common crate, one `AuthStrategy` variant, seven `ActivityKind` variants, one migration, and one IPC command pair. The rollback shape if we need to pull v0.2.0 is:

1. **Release-level:** unpublish the `v0.2.0` tag + GitHub Release. Users on `v0.2.0` who have already added Atlassian sources continue to work; the connectors are self-contained and don't regress any v0.1 surface.
2. **Schema:** migration `0005` is pure-additive (two INSERTs + one index). No down-migration is needed; v0.1.x binaries skip unknown `SourceKind` rows in the registry load (the Phase 1 lenience already covers this — existing test at `crates/dayseam-orchestrator/tests/registry.rs::unknown_source_kind_is_skipped_with_warn`).
3. **Secrets:** the shared-secret refcount change in Task 9 is backward-compatible — a v0.1.x binary still correctly drops the secret on last-source removal because Task 9's change is in the delete path, not the create path.

No destructive migration, no breaking IPC change. A v0.2.0 laptop can roll back to v0.1.x in place.
