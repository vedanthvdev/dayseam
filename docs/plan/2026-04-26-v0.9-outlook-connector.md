# Dayseam v0.9 — Outlook calendar connector (Shape A)

**Status:** Draft plan, pre-DAY-200. Awaiting human sign-off before any code is written.
**Cuts:** `v0.9.0` (new connector = minor bump per [`docs/plan/README.md`](./README.md) semver rules).
**Shape:** "Shape A" — a standalone `## Meetings` section in the daily report. No cross-event enrichment (that is deferred to Shape B, out of scope for v0.9).
**Tickets:** DAY-200 through DAY-206 (six feature PRs + one capstone review PR).

## 1. Why this phase exists

Today Dayseam describes *authored work* — commits, pull requests, Jira transitions, Confluence edits. A real workday also contains *time consumed in meetings*, which today's report is blind to. Users asked for calendar context because a report that lists every commit but silently drops three hours of 1:1s, design reviews, and standups does not accurately narrate their day.

Outlook / Microsoft 365 is the dominant calendar surface for the user base we care about (work/school accounts). Integrating calendar events as a first-class source lets the daily report answer "where did your time actually go today" alongside "what did you ship today".

## 2. Non-goals

Named explicitly because these have come up in brainstorming and we are deferring them on purpose, not by omission:

- **No cross-event enrichment.** Commits and Jira transitions do not get annotated with "during *Pairing on DAY-174*" in v0.9. That is Shape B, designed-in but deferred so v0.9 has a reviewable scope.
- **No Outlook mail.** We pull calendar events only. Mail is a much larger PII surface and an entirely different sync model.
- **No personal outlook.com / hotmail.com accounts.** Azure app registration uses `https://login.microsoftonline.com/organizations/` (work/school tenants only). Personal-account users are cleanly rejected at sign-in rather than getting a half-broken experience. Added to the multi-tenant app later if demand emerges.
- **No delta sync or webhooks.** First ship is pull-on-Generate. Microsoft Graph `/me/calendar/events/delta` is attractive for efficiency but YAGNI for a per-day report that runs once a day.
- **No Teams meetings beyond calendar-event metadata.** The Teams connector ("On deck" on the marketing site) is a separate future phase.
- **No Scheduler-preferences UI for per-filter overrides.** Defaults ship; the preferences pane lands in Shape B or v0.9.1 if users ask.

## 3. Design decisions locked during brainstorming

| Decision | Choice | Why |
|---|---|---|
| Report shape | Standalone `## Meetings` section | Ship-able in a bounded phase; enrichment (Shape B) queued as a follow-on. |
| Account types | Work/school only (M365) | Keeps the Azure app registration simple (`/organizations/` tenant). Consumer accounts (outlook.com, hotmail.com) rejected at auth step with clear copy. |
| OAuth flow | PKCE + loopback redirect | Nicer UX than device code; Tauri can spin up a one-shot localhost listener. Sets the pattern for Slack / Teams / Linear later. |
| Filter posture | Accepted + attended (hybrid), honestly documented | See §5.4 — strict-verified where Graph supports it, RSVP-fallback everywhere else. |
| Privacy posture | Full titles rendered; attendee emails never in the report | Trust user to redact. Emails are PII we store but don't paste into the output. |
| OOF / PTO / focus blocks | Skipped | They are not meetings; different signal class. |
| Short meetings (<15 min) | Included | Standups are real signal. |
| Recurring | Included, tagged `(recurring, …)` | Lets the reader speed-read familiar entries. |
| Attendee count | Only when relevant (recurring, large meetings); small meetings show attendee names | Saves column noise, signals large/recurring at a glance. |
| Timezone | User's local TZ at render time | Consistent with the rest of the report. |
| Meeting body/agenda | Not stored | YAGNI for Shape A; agenda text is the most sensitive field. |

### 4.2 Auth strategy: new `OAuthAuth` in `connectors-sdk`

Today `AuthDescriptor` is an enum of `None | Pat | Basic`. We extend it:

```rust
pub enum AuthDescriptor {
    None,
    Pat { keychain_service: String, keychain_account: String },
    Basic { email: String, keychain_service: String, keychain_account: String },
    /// OAuth 2.0 with refresh tokens. Both tokens live in the Keychain under
    /// separate accounts so a compromised refresh token doesn't unblock the
    /// access-token row and vice versa.
    OAuth {
        issuer: String,              // e.g. "https://login.microsoftonline.com/organizations/v2.0"
        client_id: String,           // Azure app registration client id
        scopes: Vec<String>,         // delegated scopes granted at consent time
        keychain_service: String,    // stable Keychain service name, shared across the pair
        access_keychain_account: String,   // "{source_id}.oauth.access"
        refresh_keychain_account: String,  // "{source_id}.oauth.refresh"
    },
}

pub struct OAuthAuth { /* access_token, refresh_token, expiry, descriptor */ }

#[async_trait]
impl AuthStrategy for OAuthAuth {
    fn name(&self) -> &'static str { "oauth2" }
    async fn authenticate(&self, req: RequestBuilder) -> Result<RequestBuilder, DayseamError> {
        if self.is_expired() { self.refresh().await?; }   // mutates in-place
        Ok(req.bearer_auth(self.access_token.expose()))
    }
    fn descriptor(&self) -> AuthDescriptor { /* clone descriptor */ }
}
```

Key properties:

- **`SecretString` wrapping**: both tokens wrapped so they never appear in `{:?}` output, identical to the existing `PatAuth` pattern.
- **No `Clone` impl**: duplicating a refresh token should be a deliberate act.
- **Refresh is internal**: connectors call `authenticate()` per request; the strategy handles expiry and calls the token endpoint when needed. One refresh-in-flight guard per `OAuthAuth` instance (tokio `Mutex` around the refresh future) so a burst of concurrent requests doesn't trigger N parallel refreshes.
- **Refresh-failure path**: a failed refresh returns `DayseamError::Auth { code: OAUTH_REFRESH_REJECTED, .. }` which the UI maps to a Reconnect flow (same shape as Atlassian's email-missing flow from DAY-84).

### 4.3 PKCE + loopback redirect in the Tauri shell

New IPC command `oauth_begin_login(source_kind)` -> returns a short-lived `LoginSession` the UI watches for completion.

```
1. Tauri side generates:
   - code_verifier (43-byte random, base64url-no-padding)
   - code_challenge = base64url(sha256(code_verifier))
   - state (128-bit random, base64url)
   - PKCE-bound authorization URL with redirect_uri = http://127.0.0.1:<port>/oauth/callback
2. Tauri spins up a tokio HTTP listener on port 0 (OS-assigned), routed to /oauth/callback.
3. Tauri opens the authorization URL in the user's default browser.
4. User signs in + consents in browser.
5. Browser redirects to http://127.0.0.1:<port>/oauth/callback?code=...&state=...
6. Listener validates `state`, exchanges `code + verifier` at the token endpoint,
   stores access + refresh tokens in Keychain, returns a minimal HTML page
   ("You can close this tab. Dayseam has your consent."), shuts down.
7. Tauri emits `oauth:login_completed` with the source-id; UI proceeds.
```

Edge cases documented and tested:

- **User closes the browser without consenting** — listener times out after 5 minutes with `OAUTH_LOGIN_TIMEOUT`, UI shows "sign-in cancelled".
- **Azure returns `error=consent_required`** — UI routes to "your IT admin needs to approve this app" copy with a deep link to the Azure admin consent URL.
- **Loopback port collision** — port 0 is OS-assigned, collision is effectively impossible, but if the listener fails to bind we emit `OAUTH_LOOPBACK_BIND_FAILED` rather than hanging.
- **`state` mismatch** — the callback returns HTTP 400 and the token exchange is skipped; `OAUTH_STATE_MISMATCH` surfaced.
- **User has firewall software that blocks `127.0.0.1`** — genuinely rare on a developer Mac but the error should be clear; we log it and suggest device-code fallback (deferred to v0.9.1 if it becomes a real problem).

### 4.4 `connector-outlook` — walker and normalise

Microsoft Graph endpoint: `GET /me/calendar/calendarView?startDateTime=<iso>&endDateTime=<iso>&$top=250&$orderby=start/dateTime`

`calendarView` is used rather than `events` because it expands recurring events into their concrete instances — critical for a daily report. Without this, a weekly standup would appear once (from the master series) rather than as an instance dated to today.

Attendance lookup for Teams meetings where the user is the organizer:
- `GET /me/onlineMeetings?$filter=joinWebUrl eq '<onlineMeetingUrl>'` to resolve the meeting id.
- `GET /me/onlineMeetings/{id}/attendanceReports/{reportId}/attendanceRecords` to pull join records.
- Only attempted when `isOnlineMeeting == true AND onlineMeetingProvider == 'teamsForBusiness' AND organizer.emailAddress.address == me`.
- Any error is swallowed silently and the event falls back to RSVP semantics — attendance lookup is best-effort; the connector never fails a sync because attendance was unknowable.

Normalise output per event:

```rust
ActivityEvent {
    kind: ActivityKind::OutlookMeetingAttended,
    external_id: graph_event_id,  // Graph's opaque event id
    occurred_at: event.start_utc,
    title: event.subject,  // rendered as "Private meeting" in the final markdown if event.sensitivity == Private
    body: None,  // we do not store the body/agenda
    actor: Actor { display_name: organizer.display_name, .. },
    entities: vec![],  // no cross-source links in Shape A
    metadata: json!({
        "end_utc": event.end_utc,
        "is_recurring": event.series_master_id.is_some(),
        "is_online_meeting": event.is_online_meeting,
        "online_meeting_provider": event.online_meeting_provider,
        "attendee_count": event.attendees.len(),
        "user_response": event.response_status.response,       // "accepted" | "tentative" | "organizer"
        "attendance_verified": verified_flag_from_attendance_lookup,
        "is_private": event.sensitivity == Private,
    }),
    raw_ref: RawRef { source_id, raw_store_uri },
    privacy: Privacy::Evidence,  // stored locally, never transmitted
}
```

### 4.5 `## Meetings` section in the report renderer

Lives in `crates/dayseam-report/src/sections.rs` with a new `sections::meetings` module.

Rendering rules (matching the example from brainstorming):

- Events of kind `OutlookMeetingAttended` only.
- Sorted by `occurred_at` ASC.
- Each bullet: `- **HH:MM–HH:MM** · <title> *(annotations)*` where annotations are:
  - `(recurring, N attendees)` if `metadata.is_recurring == true`
  - `(N attendees)` if large (≥ 6) and not recurring
  - `(organized by <name>)` if the user is not the organizer AND it's a ≤ 2-person meeting
  - `(tentative)` if `metadata.user_response == "tentative"`
  - `(attendance unverified)` if `metadata.attendance_verified == false` AND Graph *could* have verified it AND the user has opted into the attendance-verification tag in Scheduler preferences (off by default — see §5.2).
- Title rendering:
  - If `metadata.is_private == true` → `*Private meeting*`
  - Else: raw `subject`
- Trailing summary line: `Total time in meetings: Xh Ym / <workday>h (Z%)` where workday defaults to 8h, configurable in a future Scheduler preferences row (deferred — v0.9 uses 8h flat).

### 4.6 Data-model additions (core types)

- `SourceKind::Outlook`
- `SourceConfig::Outlook { tenant_id: Option<String>, user_principal_name: String }`
  - `tenant_id` is `None` during consent and populated post-consent from the ID token's `tid` claim. This lets the UI show "ACME Corp (acme.onmicrosoft.com)" in the source card and flag cross-tenant accidents.
  - `user_principal_name` is the Graph-canonical email (e.g. `vedanth@acme.com`) for display and cross-source identity matching.
- `ActivityKind::OutlookMeetingAttended`
  - The variant name is `Attended` not `Event` because the kind encodes "user participated", which matches the filter semantics. Meetings the user declined never become events in Dayseam's model — they're upstream noise, filtered at the walker.

## 5. Key design subtleties (the non-obvious stuff)

### 5.1 Keychain layout for OAuth

Each Outlook source owns two Keychain rows, keyed to the source's UUID so orphaning is obvious:

- Service: `com.dayseam.oauth.outlook`
- Account: `{source_id}.access` — payload: access token + expiry (JSON, signed with XOR-scrambled app key only as a field-integrity check; the Keychain itself is the trust boundary).
- Account: `{source_id}.refresh` — payload: refresh token (plain).

Rationale for two rows:
- macOS Keychain has a per-row update semantic; access tokens rotate every hour, refresh tokens rotate every 90 days. Separating them means the hot-path (refresh-access) doesn't touch the refresh-token row, reducing accidental-overwrite surface.
- A future "reconnect" flow can revoke the access-token row while keeping the refresh-token row around for audit — not planned for v0.9 but costless to keep the option open.

### 5.2 Attendance verification: honest hybrid semantics

Documented in §3 table and surfaced in the report output as described in §4.5. The key property: **the attendance-verified flag is tri-state, not binary**:

| state | meaning |
|---|---|
| `verified_attended` | Graph returned a join record for this user on this event. |
| `verified_not_attended` | Graph returned attendance data that *excludes* this user. Event is silently dropped from the report. |
| `unverifiable` | Not a Teams meeting, or not organized by the user, or the attendance endpoint returned an error. Event is rendered based on RSVP only and tagged `(attendance unverified)`. |

The `(attendance unverified)` tag is **off by default** in v0.9 and exposed as a Scheduler preference (`Preferences → Report → Annotate meetings with attendance verification`). Users who know their workflow is Teams-heavy and want the honesty toggle it on; everyone else gets a cleaner report out of the box. The preference ships with DAY-204 (same PR as the Meetings section) so there's no v0.9 window where the toggle exists but doesn't do anything.

### 5.3 The "admin hasn't consented" path

Realistic for users at mid-to-large companies. Graph returns `error=consent_required` from the authorize endpoint when the tenant has blocked third-party app consent. The dialog needs copy that:

1. Doesn't blame Dayseam ("your tenant administrator has not approved Dayseam for Microsoft Graph access").
2. Tells them what to forward to IT (a link to the admin-consent URL with `client_id={CLIENT_ID}&scope={SCOPES}` that a global admin can click).
3. Mentions that Dayseam reads the calendar only; no write access requested.

Draft copy in `apps/desktop/src/features/sources/outlookAdminConsentCopy.ts` (paralleling `githubErrorCopy.ts`).

### 5.4 Rate limits

Graph's limits are generous (`10,000 requests / 10 minutes / app`) but per-user subject to throttling at `6 concurrent requests`. For a per-day calendarView on a single user this is irrelevant; for the attendance-lookup path we fan out one extra call per Teams meeting, which is still well under the throttle.

The `http.rs` module in `connectors-sdk` already has retry-with-jitter on 429; we pick up its behavior without changes. First-ship does not add dedicated Graph-throttling awareness — only if dogfooding flags it.

### 5.5 Timezones, again

- Graph returns `start.dateTime + start.timeZone` as strings; we parse both and convert to UTC at the walker boundary.
- The report renderer converts back to local-TZ for display.
- Meetings that *span midnight in the user's local TZ* (rare) are bucketed to the day they *start* on, matching how commits are bucketed by author-date.
- Meetings that span a DST transition are rendered with the start-side offset; the end time is computed from `UTC + duration` so DST shifts don't break arithmetic.

### 7.1 DAY-200's two-PR shape

DAY-200 is the only ticket in this phase that ships as two PRs against one GitHub issue:

- **Plan-PR-first** is a deliberate departure from the bundled convention that v0.2 / v0.3 / v0.4 plan docs followed. It lets a human reviewer read the ~500-line plan in isolation without being asked to review it *alongside* ~800 lines of brand-new OAuth SDK code in one diff.
- The repo's "one commit per branch" rule (per AGENTS.md) is satisfied in each PR independently: PR #1 is a single commit (the plan), PR #2 is a single commit (the OAuth SDK) on a fresh branch cut from the master that already contains the merged plan.
- Both PRs reference GitHub issue #200 in their bodies; the issue is not closed until PR #2 merges.
- No other ticket in this phase uses this pattern. DAY-201..206 all follow the standard one-issue-one-PR shape.

## 8. Decisions resolved during brainstorm

These were the three open questions before brainstorming. Recording the resolutions here so a fresh reviewer of this plan doesn't have to reconstruct the conversation:

- **Azure app registration owner.** Deferred for DAY-200 (pure SDK work, no `client_id` needed). At the start of DAY-201, register a new dedicated Dayseam Entra ID tenant (free tier) and register a multi-tenant OAuth 2.0 app there. Publisher name in the consent dialog will be "Dayseam", not any contributor's personal tenant. Documented in the DAY-201 prerequisite block above.
- **Client-id distribution.** Covered by the DAY-201 prerequisite: `client_id` lives in build-time config, not secret but not in source either. Rotating breaks every existing consent, so the registration is a one-way door — we pick a distinctive tenant name on first registration and commit to it.
- **"Attendance unverified" annotation.** Off by default in v0.9. Exposed as a Scheduler preference (`Preferences → Report → Annotate meetings with attendance verification`), so Teams-heavy users who want the honesty can opt in. See §4.5 rendering rules and §5.2 semantics; implementation scoped into DAY-204.

## 9. What this plan deliberately does not commit to

- **Not locked to specific line-level code.** This plan shows the shape and the invariants; the executing engineer at each DAY-N PR fills in imports, error-mapping boilerplate, and plumbing without renegotiating the plan.
- **Not a replacement for `ARCHITECTURE.md`.** `ARCHITECTURE.md` gets a section added in DAY-204 that describes the *shipped* shape of the Outlook connector, and that section supersedes anything in this plan that drifted during execution.
- **Not exhaustive of dogfood surprises.** v0.4 and v0.5's dogfood notes both surfaced real issues post-capstone; v0.9 will too, and that's the capstone's job to catch.
