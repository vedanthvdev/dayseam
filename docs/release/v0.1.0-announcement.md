## Long form (release page / blog post)

### Dayseam v0.1.0 — a work journal that drafts itself from the tools you already use

Dayseam turns a day's worth of your activity across git and GitLab
into a markdown report you can drop into your existing work journal.
No scheduler, no cloud round-trip, no "let us index your repos on our
servers." You run it on your Mac, point it at the sources you already
have, and it produces a draft you'd have written yourself — just
faster, and with clickable evidence for every claim.

v0.1.0 is the first release anyone outside the project can download
and run. The goal of this release is modest on purpose: a shippable
binary that earns enough trust to survive the second user.

#### What's in the box

- **Two connectors.** Local git (via libgit2 against any on-disk repo
  you grant access to) and GitLab (self-hosted or .com, via a
  Personal Access Token with the `read_api` scope). Either one works
  on its own; using them together is where the design earns its keep.
- **Cross-source deduplication.** If you pushed a commit to GitLab
  earlier in the day and it also shows up in your local revwalk,
  Dayseam collapses them into a single bullet — with both sources
  cited as evidence — and annotates the bullet with the MR it was
  rolled into. No more "I committed and mentioned the PR; both lines
  are now in my journal."
- **A Dev EOD template.** The first template to ship. Groups your
  day's activity into a structure that reads like how a developer
  would narrate it: what landed, what's in review, what got opened,
  what got reviewed. Verbose mode adds one clickable evidence link
  per bullet.
- **A markdown sink with a marker block.** Writes the report between
  machine-readable markers so you can edit *around* the report in the
  same file (notes, context, next-day TODOs) and the next run
  preserves those edits, only rewriting the managed region. Your
  file stays yours.
- **Per-source error cards.** If your PAT expires mid-day, the source
  chip turns red with the actual reason (`gitlab.auth.invalid_token`,
  `gitlab.auth.missing_scope`, `gitlab.host_unreachable`, etc.), and
  a Reconnect button takes you back to the validate-PAT dialog with
  your base URL pre-filled. Errors are recoverable, not just
  reportable.
- **Your data on your disk.** The app stores state in a local SQLite
  database under `~/Library/Application Support/app.dayseam.desktop/`.
  Secrets live in the macOS Keychain, keyed by source id, and never
  cross the IPC boundary in cleartext. Redaction lints enforce this;
  the Phase 3 review includes a repo-wide PAT grep that came up
  empty.

#### What's not in v0.1.0 (on purpose)

- **No codesign or notarisation.** v0.1.0 ships *unsigned*. The first
  launch walks you through Gatekeeper's right-click-Open dance; see
  [`UNSIGNED-FIRST-RUN.md`](UNSIGNED-FIRST-RUN.md). This is a
  one-release compromise: Developer ID codesign + notarisation lands
  in v0.1.1 (tracked in
  [issue #59](https://github.com/vedanthvdev/dayseam/issues/59) and
  scoped in [`PHASE-3-5-CODESIGN.md`](PHASE-3-5-CODESIGN.md)).
- **No scheduler.** You click Generate. v0.3 adds a background daily
  run if v0.1 earns enough trust to need one.
- **No LLM.** The report is a faithful summary of what the APIs
  actually returned, not a rewrite. v0.5 adds an *optional* LLM
  rewrite pass with the local-first / never-upload posture the design
  doc spells out.
- **No GitLab OAuth.** v0.1 is PAT only. v0.2 adds OAuth; the
  connector's `AuthStrategy` trait is already shaped for it.
- **No Windows, no Linux.** macOS 13+ only. Cross-platform is on the
  roadmap; the design doesn't preclude it, but v0.1 ships what's been
  built and tested.

#### Getting started

1. Download `Dayseam-v0.1.0.dmg` from the
   [release page](https://github.com/vedanthvdev/dayseam/releases/tag/v0.1.0).
2. Verify the checksum:
   `shasum -a 256 -c Dayseam-v0.1.0.dmg.sha256`.
3. Open the DMG, drag `Dayseam.app` to your Applications folder
   (either `/Applications/` or `~/Applications/` — both work; the
   latter doesn't require admin).
4. Follow [`UNSIGNED-FIRST-RUN.md`](UNSIGNED-FIRST-RUN.md) the first
   time you launch (right-click → Open, accept Gatekeeper's warning).
5. From the sidebar: Add a scan root for local git, Add a GitLab
   source (paste the base URL + a PAT with `read_api` scope), pick a
   date, click Generate, click Save.

If step 5 produces something you'd actually want to keep, Dayseam is
working. If it doesn't — that's the bug we want to hear about most.

#### What changed in this release

The [CHANGELOG](../../CHANGELOG.md) and the
[Phase 3 review](../review/phase-3-review.md) are the canonical
record. A few highlights worth calling out:

- Two correctness fixes inlined in the Task 8 hardening PR because
  they degraded the first-run experience: the GitLab
  authentication-error surface (`HttpClient::send` used to mask 401
  and 403 as generic network errors) and the GitLab evidence-link
  URLs (which 404-ed because the composed path mixed GitLab's UI
  prefix with its API prefix).
- Five consecutive release-workflow hotfixes (DAY-63 → DAY-67) that
  took the CI pipeline from "green dry-run, red real run" to a
  published DMG. Every one shipped with regression tests in
  `scripts/release/` so the same failure modes don't recur.
- A Playwright-BDD E2E gate on the generate-and-save happy path that
  runs on every PR and catches IPC-contract regressions the unit
  tests can't see.

#### Thank you

If you try v0.1.0 and it doesn't work for you, please
[file an issue](https://github.com/vedanthvdev/dayseam/issues/new).
The ones that come in during the first week are going to shape
v0.1.1 more than any plan doc we could write in advance. "The DMG
won't open" counts as a bug. "The evidence link takes me to JSON
instead of the MR" is already tracked. "The report is wrong" is the
most useful thing you could tell us.

## Intentionally out of scope for this draft

- Hacker News / Product Hunt submissions. v0.1.0 is deliberately not
  targeting a launch moment — the distribution surface above is the
  set that reaches the existing audience without opening the support
  surface area to strangers before codesign lands. v0.2 or v0.3 is a
  more honest launch-post target.
- Press outreach. Same reason.
- A tutorial video. If v0.1.1 ships within a month with a friendlier
  first-run (post-codesign), that's the right moment for one. v0.1.0
  still asks the user to right-click-Open a download.
