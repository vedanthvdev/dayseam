## 3. First-run / upgrade path observations

Free-form notes on anything the boot sequence, the GitHub add flow, or the per-kind renderer did that surprised the dogfood owner on Day 1. Focus areas:

- **No-migration upgrade from v0.3.0** — does the app boot cleanly against an existing `state.db`? `sources.kind` is a TEXT column, so adding `"github"` as a value required no DDL; the `SourceIdentityKind` column accepts the two new variants (`GitHubUserId`, `GitHubLogin`) for the same reason. Expected answer: boots silently.
- **AddGithubSourceDialog UX** — validate-edit flow correctly drops the cached ribbon when URL or PAT change? Submit disabled between ok → edit → ok transitions? Default label auto-fills from the normalised host? Reconnect mode pre-fills URL + label but blanks the PAT input? Numeric user-id mismatch on Reconnect surfaces as `github.auth.invalid_credentials` with the `action_hint: reconnect` rather than a silent rebind?
- **CORR-v0.4-01 evidence** — on a fresh add, the identity manager dialog (or the `sqlite3 state.db "SELECT kind, external_actor_id FROM source_identities WHERE source_id = '<uuid>'"` command) must show **both** `GitHubUserId` and `GitHubLogin` rows. Missing either → the DAY-101 fix did not reach this install; filing is a P0 on this sweep.
- **`ReportSection::MergeRequests` rendering** — on a day with both a GitHub PR and a GitLab MR, both render under `## Merge requests` with the renderer's per-forge `[#123]` vs `[!42]` badge distinguishing them (not a forge-name prefix on the bullet copy).
- **One-pass grouper (PERF-v0.3-01 → DAY-98)** — on a day with ≥20 events mixed across all four sections, does the report render in visibly <1s? The previous two-pass grouper was fine at 20 events; the rework was for a future >500-event day, so what we're verifying is that the rework *didn't* regress the small-day case.
- **`Unlinked activity` section (CORR-v0.3-01 → DAY-98)** — on a day with an orphan Confluence comment whose parent page couldn't be resolved, does the bullet render under `## Unlinked activity` (not under a misleading `## Other`)?
- **Cross-source enrichment** — a GitHub PR titled `"DAY-100: ..."` where `DAY-100` is a Jira key: does the Jira transition bullet carry `(triggered by #42)` with the PR number? A GitLab MR and GitHub PR both referencing `DAY-100`: does exactly one win the `(triggered by …)` slot (the earliest-opened)? The `extract_github_pr_urls` scan on GitLab MR bodies and the Jira key extraction from PR titles both active?
- **Error-code symmetry** — revoked PAT → `github.auth.invalid_credentials`, 410 on a deleted repo → `github.upstream.resource_gone`, rate-limited → `github.rate_limited` (retryable). These map symmetrically to the GitLab/Atlassian counterparts per the v0.4 §2.3 lens finding.
- **Migration `0005_secret_ref_index.sql`** — carries over from v0.3; a fresh GitHub source adds a new `sources` row with `secret_ref` set, so the partial index continues to cover the GitHub case without change.
