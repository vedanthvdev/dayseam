## 3. First-run / upgrade path observations

Free-form notes on anything the boot sequence, the GitHub add flow, or the per-kind renderer did that surprised the dogfood owner on Day 1. Focus areas:

- **No-migration upgrade from v0.3.0** — does the app boot cleanly against an existing `state.db`? `sources.kind` is a TEXT column, so adding `"github"` as a value required no DDL; the `SourceIdentityKind` column accepts the two new variants (`GitHubUserId`, `GitHubLogin`) for the same reason. Expected answer: boots silently.
- **AddGithubSourceDialog UX** — validate-edit flow correctly drops the cached ribbon when URL or PAT change? Submit disabled between ok → edit → ok transitions? Default label auto-fills from the normalised host? Reconnect mode pre-fills URL + label but blanks the PAT input? Numeric user-id mismatch on Reconnect surfaces as `github.auth.invalid_credentials` with the `action_hint: reconnect` rather than a silent rebind?
- **CORR-v0.4-01 evidence** — on a fresh add, the identity manager dialog (or the `sqlite3 state.db "SELECT kind, external_actor_id FROM source_identities WHERE source_id = '<uuid>'"` command) must show **both** `GitHubUserId` and `GitHubLogin` rows. Missing either → the DAY-101 fix did not reach this install; filing is a P0 on this sweep.
- **`ReportSection::MergeRequests` rendering** — on a day with both a GitHub PR and a GitLab MR, both render under `## Merge requests` with the renderer's per-forge `[#123]` vs `[!42]` badge distinguishing them (not a forge-name prefix on the bullet copy).
- **One-pass grouper (PERF-v0.3-01 → DAY-98)** — on a day with ≥20 events mixed across all four sections, does the report render in visibly <1s? The previous two-pass grouper was fine at 20 events; the rework was for a future >500-event day, so what we're verifying is that the rework *didn't* regress the small-day case.
- **`Unlinked activity` section (CORR-v0.3-01 → DAY-98)** — on a day with an orphan Confluence comment whose parent page couldn't be resolved, does the bullet render under `## Unlinked activity` (not under a misleading `## Other`)?
- **Cross-source enrichment** — a GitHub PR titled `"DAY-100: ..."` where `DAY-100` is a Jira key: does the Jira transition bullet carry `(triggered by #42)` with the PR number? A GitLab MR and GitHub PR both referencing `DAY-100`: does exactly one win the `(triggered by …)` slot (the earliest-opened)? The `extract_github_pr_urls` scan on GitLab MR bodies and the Jira key extraction from PR titles both active?
- **Error-code symmetry** — revoked PAT → `github.auth.invalid_credentials`, 410 on a deleted repo → `github.upstream.resource_gone`, rate-limited → `github.rate_limited` (retryable). These map symmetrically to the GitLab/Atlassian counterparts per the v0.4 §2.3 lens finding.
- **Migration `0005_secret_ref_index.sql`** — carries over from v0.3; a fresh GitHub source adds a new `sources` row with `secret_ref` set, so the partial index continues to cover the GitHub case without change.

| **DOGFOOD-v0.4-01** | 1 (release-day smoke test) | **P0 (release-workflow correctness, zero-user-facing-behaviour impact on the *product*; 100 % impact on "what version does macOS think this app is")** | **Published `Dayseam-v0.4.0.dmg` bundle's `Info.plist` reports `CFBundleShortVersionString = 0.2.1`** despite the DMG filename, git tag `v0.4.0`, and GitHub Release body all saying `v0.4.0`. Root cause traced to [`scripts/release/bump-version.sh`](../../scripts/release/bump-version.sh): the script-level `if [[ "$TARGET" != "$CURRENT" ]]` idempotency gate wrapped *all three* file writers, so whenever a capstone PR manually pre-bumped `VERSION` (the authoring pattern both v0.3 and v0.4 followed for reviewer visibility), the post-merge release workflow's `bump-version.sh minor` call saw `TARGET == $(cat VERSION)` and silently skipped rewriting `Cargo.toml` and `apps/desktop/src-tauri/tauri.conf.json` — which stayed pinned at `0.2.1` from the v0.2.1 release because no intermediate PR ever touched them. `tauri build` then embedded the stale `tauri.conf.json` version into `Contents/Info.plist`. The same bug affected `v0.3.0.dmg` (also stamped `0.2.1`) but went undetected because the v0.3 dogfood sweep never inspected the plist. The tripwire on v0.4 was that the `0.2.1` binary installed on this laptop at `/Applications/Dayseam.app` *already existed*, so macOS LaunchServices matched the newly-mounted DMG against it by CFBundleIdentifier and quietly replaced — first launch on Monday surfaced as a Rust panic (`SIGABRT` / "Reopen or Report") because the `0.2.1` binary could not open a `0.3.0`-schema `state.db`. Re-runnable evidence: `/usr/libexec/PlistBuddy -c "Print CFBundleShortVersionString" /Volumes/Dayseam/Dayseam.app/Contents/Info.plist` on the published DMG. | **Inlined — this PR (`DAY-102` → `v0.4.1`).** Three-part fix: (1) align the tree so `Cargo.toml [workspace.package].version` and `tauri.conf.json "version"` both say `0.4.1`; (2) harden `bump-version.sh` to run all three writers unconditionally with internal no-op-if-unchanged via `cmp -s`-before-`mv` (preserves the byte-for-byte idempotency contract the double-invoke test asserts); (3) add a PlistBuddy version-parity gate to [`.github/workflows/release.yml`](../../.github/workflows/release.yml) as defence-in-depth — if this gate had existed, `v0.3.0` and `v0.4.0` would both have red-failed in CI before publishing a mis-stamped DMG. Two new regression tests added to [`scripts/release/test-bump-version.sh`](../../scripts/release/test-bump-version.sh): one reproduces the exact capstone-PR shape (stale `Cargo.toml` + `tauri.conf.json`, pre-bumped `VERSION`) and asserts all three files end up on target; one standing three-file-parity cross-check on every invocation. |

### 4.1 `DAY-103` adversarial-review pass (post-dogfood hardening)

A tier-1/tier-2 adversarial-review pass over the `DAY-103` diff surfaced eight follow-up issues, seven of which ship in the same `v0.4.2` patch so the next dogfooder doesn't hit them. Landing these inside the same PR — rather than deferring to v0.5 — matches the compound-engineering discipline of "if the review found it, the fix ships with the thing being reviewed".

| Review ID | Severity | Finding | Fix (all inlined in `DAY-103` → `v0.4.2`) |

| **F-2** | **P1** | Truncation + reconcile silently deleted the tail of `local_repos` for any source with `>512` repos, including user-set `is_private` flags. The original DOGFOOD-v0.4-03 reconcile path didn't distinguish "walker returned its full result set" from "walker capped at `max_roots`", so the reconcile transaction treated a truncated outcome as authoritative. | `upsert_discovered_repos` now detects `outcome.truncated` and falls back to additive per-row `upsert` with a warn log. The cap is tuning, not a safety boundary — excess rows stay put. |

Two new tests pin both branches. |
